# 施策見積もり・投資判断フレームワーク

> **これは何？**
>
> 要件が未確定な段階で「概算でいいからコスト感を教えてほしい」と言われたときの見積もり方法と、施策の優先順位づけを支援するチェックリストツールの説明資料。
>
> **2つの見積もりシーンをカバーしている：**
>
> ① **施策比較・優先順位づけ（Part 1）** — チェックリストで不確実性を構造化し、コスト参考値を横並びで比較する
>
> ② **超概算見積もり（Part 2）** — チェックリストなしで最速でコストの桁感を出す

---

## 共通：見積もりの考え方

### 前提：スクラムチーム＝固定費

スクラムチームの人件費は**やってもやらなくても発生する固定費**であり、変動費ではない。

- チームを維持している以上、コストは常に発生する
- チーム人数は原則変動しないとみなすと、年間の工数総量は固定（例：4人 × 12ヶ月 = 48人月）
- 問われるのは「いくらかけるか」ではなく「**限られた枠を何に使うか**」
- 「この機能にいくらかかる？」よりも「このテーマに全体の何％投資する？」が正しい問い

### この見積もりで答える問い

基本計画を立てる上で、**「この施策に投資する価値があるか」のROI試算**を出すことが目的。

```
ROI = 期待効果 ÷ コスト試算
```

精密な見積もりではない。**投資判断ができる桁感**がわかれば十分。

### なぜ機能の積み上げで見積もらないのか

| 積み上げ見積もり | この見積もり |
|---|---|
| 機能を洗い出し → 工数を積む → 合計 | チーム × 期間 → ベースコスト |
| 要件が確定していないと積めない | 要件が不確定でも出せる |
| 「いくらかかる？」に答える | 「投資する価値があるか？」に答える |
| エンジニアが工数にコミットさせられる | エンジニアは不確実性だけを伝える |

スクラム体制では機能を増やしても減らしてもチームの人件費は変わらない。「何を作るか」ではなく「**チームを何ヶ月張るか**」がコストの実態。

---

### ベースコストの算出

```
ベースコスト = チーム人数 × 108万円/人月 × 想定期間（月）
```

| パラメータ | 説明 | 備考 |
|---|---|---|
| **チーム人数** | 施策に張るチームの人数 | 既存チームならその人数。新規なら確保可能な人数 |
| **108万円** | 1人月あたりの単価 | 人件費＋間接費込み |
| **想定期間** | MVPを構築する期間 | 後述の方法で設定する |

#### 体制（チーム人数）の決め方

体制は**見積もりの結果ではなく、組織制約で決まる**。

- 既存チームを充てる → そのチームの人数がそのまま入る
- 新規チームを立てる → 「何人の枠を張れるか」は採用・予算の制約で決まる

エンジニアに「何人必要？」と聞くのではなく、**「このチームで取り組む」が先にある**。

#### 期間の決め方 ─ 推定ではなく、前提条件として置く

不確実性が高い案件では、期間を正確に推定することはできない。したがって期間は**推定するのではなく、前提条件として置く**。

**設定プロセス：**

```
Step 1: PdMがMVPの像を出す
        「この施策のMVPは、○○ができる状態」
        （最小閉路 = ユーザーが最低限の行為を完結できる最小の流れ）
                ↓
Step 2: エンジニアに「期間感」を聞く（3択）
        「このMVP像、半年感？1年感？それ以上？」
        ※「何ヶ月かかる？」とは聞かない。コミットメントではなく感覚として聞く
                ↓
Step 3: 前提条件として期間を設定する
        例：「前提：4人チーム × 12ヶ月」
```

| 期間 | 目安 | ベースコスト（4人の場合） |
|---|---|---|
| **6ヶ月** | 小〜中規模のMVP | 2,592万円 |
| **12ヶ月** | 中〜大規模のMVP | 5,184万円 |
| **18ヶ月** | 大規模・複数フェーズ | 7,776万円 |

> **「1年」の根拠は？** と聞かれたら → PdMが定義したMVP像に対し、エンジニアが「1年感」と回答したことが根拠。ただし試算の前提条件であり、コミットメントではない。

#### MVPの定義 ─ 誰が、どう決めるか

| ステップ | 担当 | やること |
|---|---|---|
| 最小閉路の初案 | **PdM**（オーナー） | 「ユーザーがこの行為を完結できる最小の流れ」を定義する |
| 技術フィードバック | **エンジニア** | 「この部分は重い/軽い/先にやるべき」をインプットする |
| スコープ調整判断 | **PdM**（意思決定） | Must / Later / Never を仕分ける |
| 事業Go判断 | **ビジネス/事業** | 「このMVPで投資判断に足るか」を判断する |

**最小閉路とは：** ユーザーが「これができないと業務が止まる」行為が、最初から最後まで完了する最小の流れ。UIが未完成でも、裏側が手作業でもよい（ただし運用コスト上限は決める）。

**Must / Later / Never の仕分け：**

| 分類 | 基準 | 例 |
|---|---|---|
| **Must** | 最小閉路に含まれる。止まる。法務・会計・監査で必須 | ログイン、主要業務フロー |
| **Later** | 価値はあるが、代替運用ができる | レポート、一括処理 |
| **Never** | 使われていない。効果不明。新の目的と合わない | 旧UIの完全再現 |

> **ポイント：** 意図的にNever/Laterを作ることが重要。「全部同等に移植する」を避け、**価値が出る最初の到達点を先に合意する**。

### 不確実性とコスト幅

#### 不確実性とは何か

システム案件には「やってみないとわからないこと」が多く存在する。不確実性が高いと、当初の想定より時間がかかり、結果としてコストが増える。

**具体的に何が起きるのか：**

| # | 想定外の例 | 影響 |
|---|---|---|
| 1 | **複数部署との協業で意思決定者が曖昧、仕様が二転三転する** — 「この仕様でOK」と言われて作り始めたが、別の部長から「聞いていない、やり直し」 | 作り直しで**2週間追加** |
| 2 | **連携先システムの仕様が未確定** — 基幹システムとデータ連携する想定だったが、基幹側の改修が遅れ仕様が決まらない | 待ちと手戻りが発生 |
| 3 | **例外ケースが後から大量に出てくる** — 運用部門に見せたら「このケースは？」と想定外のパターンが20個出た | 例外対応で**1ヶ月追加** |
| 4 | **制度変更でやり直し** — 法改正や社内制度変更で前提の業務ルールが変わる | 作った機能の作り直し |
| 5 | **データが想定通りではない** — 「使える」と言われたデータが、NULLだらけ・表記揺れだらけ・重複だらけ | クレンジングで**3週間追加** |

#### 2種類の不確実性

不確実性は **「要件」と「技術」** の2軸で評価する。それぞれコストに影響する経路が異なる。

```
                    ┌──────────────────┐
                    │  技術不確実性     │
                    │  → ベースコスト   │──→ 体制・期間が膨らむ
                    │    を押し上げる   │
                    └────────┬─────────┘
                             │
                             ▼
┌──────────────┐    ┌──────────────────┐
│  チェック     │    │   ベースコスト    │
│  リスト入力   │──→│ × 振れ幅係数     │──→ コスト参考値（レンジ）
│              │    │ = コスト参考値    │
└──────────────┘    └──────────────────┘
                             ▲
                    ┌────────┴─────────┐
                    │  要件不確実性     │
                    │  → 振れ幅（レンジ）│──→ 手戻り・変更でブレる
                    │    を広げる       │
                    └──────────────────┘
```

| 不確実性の種類 | コストへの影響経路 | 意味 |
|---|---|---|
| **技術不確実性** | **ベースコスト**を押し上げる | 実装の複雑性が高い → 必要な体制・期間が増える |
| **要件不確実性** | **振れ幅（レンジ）**を広げる | 要件変更・手戻りのリスクが高い → コストのブレが大きくなる |

#### 要件不確実性Lvと振れ幅係数

| 要件不確実性Lv | 状態 | 振れ幅係数 | 意味 |
|---|---|---|---|
| **Lv1** | 要件明確、関係者少ない | 1.0〜1.2倍 | ほぼ想定通り。最大20%増 |
| **Lv2** | 一部曖昧だが大筋は見えている | 1.0〜1.5倍 | まあ読める。最大50%増 |
| **Lv3** | 曖昧な点が多く、調整が必要 | 1.0〜2.0倍 | 読みにくい。最大2倍 |
| **Lv4** | 不明点だらけ、前提が変わりうる | **見積もり不能** | → ディスカバリー投資を推奨 |

#### 技術不確実性Lvとベースコストへの影響

| 技術不確実性Lv | 状態 | ベースコストへの影響 |
|---|---|---|
| **Lv1** | 既存技術、実績あり | 影響なし（最小体制で実施可能） |
| **Lv2** | 一部新規技術、外部連携あり | 体制+1名 or 期間+1〜2ヶ月 |
| **Lv3** | 新規技術多い、大規模データ移行あり | 体制+2名 or 期間+3ヶ月以上 |
| **Lv4** | 前例なし、技術検証が必要 | → PoC（技術検証）を先行 |

#### コスト参考値の算出

```
コスト参考値 = ベースコスト（技術Lvで調整済み） × 振れ幅係数（要件Lvで決定）
```

**計算例：**
- 前提：4人チーム × 12ヶ月（PdMのMVP像に対しエンジニアが「1年感」と回答）
- ベースコスト：4人 × 108万 × 12ヶ月 = 5,184万円
- 技術不確実性Lv2 → 体制+1名 → 5人 × 108万 × 12ヶ月 = **6,480万円**（調整後）
- 要件不確実性Lv3 → 振れ幅 1.0〜2.0倍
- **コスト参考値：6,480万〜12,960万円**

#### 基本計画での表記

試算であることと前提条件を明示する。

| 表記 | 問題 |
|---|---|
| ~~「コスト：5,184万円」~~ | 確定に見える |
| **「コスト試算：5,184万〜10,368万円」** | レンジで表記 |

前提条件を必ず併記する：

```
【前提条件】
- 体制：4名チーム（Dev）
- 期間：12ヶ月（MVP構築）
- MVP定義：○○のフローが新システムで完結する状態
- 要件不確実性：Lv3（振れ幅 1.0〜2.0倍）
- 技術不確実性：Lv2（体制+1名で調整）
- 本試算はコミットメントではなく、投資判断のための参考情報
```

#### Lv4の場合：ディスカバリー投資を推奨

要件 or 技術のいずれかがLv4の場合、コストレンジを出しても判断材料にならない。代わりに**ディスカバリーフェーズへの投資**を提案する。

| 項目 | 内容 |
|---|---|
| **投資規模** | 2名 × 3ヶ月 = 648万円（固定） |
| **ゴール** | MVP定義、技術検証、不確実性をLv2以下に低減 |
| **判断ポイント** | ディスカバリー後にGo/NoGoを再判断 |

> **ビジネスへの伝え方：**「現段階でコストを出すと4倍以上のブレ幅になり、判断材料になりません。まず648万円・3ヶ月のディスカバリーに投資し、①やるべきか ②やるなら何から ③精度±50%の見積もりを出します」

### ROI判断

コスト試算が出たら、期待効果と比較してROIを判断する。

**計算例：**
- 期待効果：年間1億円のコスト削減
- コスト試算：6,480万〜12,960万円
- → **最悪ケースでも12,960万円。年間1億円の効果が2年続くなら投資価値あり**

| 判断 | 基準 |
|---|---|
| **Go** | 最悪ケースのコストでもROIが成立する |
| **条件付きGo** | 標準ケースならROI成立、最悪ケースは微妙 → リスク低減策を併用 |
| **ディスカバリーへ** | 不確実性Lv4、またはROIの判断がつかない → 先に不確実性を下げる |
| **NoGo** | 最良ケースでもROIが成立しない |

> どの数値で経営に出すかは、リスクを見ながらPdMが判断する。リスクの根拠（不確実性の要因）もセットで説明する。

## Part 1：施策投資判断チェックリスト

*複数の施策候補を比較して優先順位をつけるためのツール*

### なぜ必要か

施策の検討が始まると「どのくらいかかりそう？」という問いが来る。

- **ビジネス側：** 投資判断には定量的な根拠が必要。それが見えないとGo/NoGoが決められない
- **エンジニア側：** 不確実な段階で数字を出すと、試算ではなくコミットメントとして扱われることを危惧している。意味のない数字を出すために時間をかけたくない

双方とも正しい。噛み合っていないのは前提。

- ビジネスが求めているのは → **投資判断のための材料**
- エンジニアが避けたいのは → **意味のない数字を出す作業と、それがコミットメントになること**

### 解決する問い

**「エンジニアに工数をコミットさせずに、ビジネスが投資判断できる情報を出す」**

#### 入力の責任分離

| 役割 | 入力するもの | 入力しないもの |
|---|---|---|
| **エンジニア** | チェックリストへの回答（Yes/No）、期間感（3択） | 工数（人月）、期日、コミットメント |
| **PdM/SM** | MVP像（最小閉路）、体制・期間の前提設定 | — |
| **PdM** | 期待効果（万円/年） | — |
| **ツール** | 不確実性Lv、コスト参考値（レンジ）を自動算出 | — |

---

### 利用フロー

```
Step 1          Step 2                Step 3          Step 4
施策候補が    →  PdMがMVP像を定義   →  PdMが         →  ツールが
出る             エンジニアが           効果を試算        自動算出
                 チェックリストに
                 回答＋期間感          　                ・不確実性Lv
                 PdM/SMが体制・                         ・コスト参考値
                 期間を前提設定                          ・ROI参考値
```

### Step 1：施策候補が出る

ビジネス側から施策候補が上がる。

### Step 2：MVP像の定義 ＋ チェックリスト回答 ＋ 前提設定

**2-a. PdMがMVP像を出す**

「この施策のMVPは、○○ができる状態」を最小閉路として定義する。

**2-b. エンジニアがチェックリストに回答する**

エンジニアに**2種類のチェックリスト**を渡し、各項目について「該当する／しない」を回答してもらう。

**2-c. エンジニアに期間感を聞く**

MVP像に対して「半年感？1年感？それ以上？」を3択で聞く。

**2-d. PdM/SMが体制・期間を前提として設定する**

チーム人数（組織制約）とエンジニアの期間感をもとに、前提条件を設定する。

---

#### ① 要件不確実性チェックリスト（8項目）

「この施策は、要件の変動・手戻りがどれだけ起きそうか」を測る。

| # | チェック項目 | 該当=1点 |
|---|---|---|
| 1 | **決裁者が不明確、または複数いる** — 最終的に仕様をOKする人が1人に定まっていない | □ |
| 2 | **ユーザー（利用者）が特定できていない** — 誰のどんな課題を解くのかが曖昧 | □ |
| 3 | **連携先の仕様が未確定** — 外部システムや他部署のAPIが決まっていない/変わりうる | □ |
| 4 | **法規制・制度変更の影響を受ける可能性がある** — 前提となるルールが変わりうる | □ |
| 5 | **社内に類似案件の実績がない** — 比較対象となる過去プロジェクトがない | □ |
| 6 | **要件を決める人と使う人が異なる** — 決定者の想定と現場の実態がズレるリスク | □ |
| 7 | **業務フローが文書化されていない** — 例外パターンが後から出てくるリスク | □ |
| 8 | **3部署以上をまたぐ調整が必要** — 合意形成のコストが高い | □ |

**スコアリング：**

| 合計点 | 要件不確実性Lv | 振れ幅係数 |
|---|---|---|
| 0〜1点 | **Lv1** — ほぼ読める | 1.0〜1.2倍 |
| 2〜3点 | **Lv2** — まあ読める | 1.0〜1.5倍 |
| 4〜5点 | **Lv3** — 読みにくい | 1.0〜2.0倍 |
| 6点以上 | **Lv4** — 見積もり不能 | → ディスカバリー推奨 |

---

#### ② 技術不確実性チェックリスト（12項目 ＋ 規模）

「この施策は、技術的にどれだけ複雑・未知か」を測る。結果はベースコスト（体制・期間）に影響する。

**A. 技術複雑性（12項目）**

| # | チェック項目 | 該当=1点 |
|---|---|---|
| 1 | **外部システム連携が2つ以上必要** — API設計・認証・エラーハンドリングが複雑化 | □ |
| 2 | **データ移行が必要** — 既存データの変換・クレンジング・整合性検証が発生 | □ |
| 3 | **性能要件が厳しい** — 大量トランザクション、リアルタイム処理、高可用性が求められる | □ |
| 4 | **セキュリティ・コンプライアンス要件がある** — 個人情報保護、認証基盤、監査対応等 | □ |
| 5 | **チームに経験のない技術を使う** — 新規フレームワーク、新規クラウドサービス等 | □ |
| 6 | **レガシーシステムへの依存がある** — ドキュメントが乏しい既存システムとの結合 | □ |
| 7 | **複数のフロントエンド/チャネル対応** — Web・アプリ・管理画面など複数UI | □ |
| 8 | **インフラ構成の変更が必要** — 新規環境構築、ネットワーク変更、CI/CDパイプライン変更 | □ |
| 9 | **非同期処理・バッチ処理が中心** — ジョブ設計、リトライ、監視の複雑性 | □ |
| 10 | **マルチテナント対応が必要** — データ分離、権限制御の設計 | □ |
| 11 | **E2Eテスト・負荷テストの設計が必要** — テスト環境構築、シナリオ設計 | □ |
| 12 | **既存の共通基盤への変更が必要** — 認証、課金、通知等の共通サービスへの影響 | □ |

**B. 規模感（択一）**

| 選択肢 | 基準 | 加算点 |
|---|---|---|
| **S** | 画面数 1〜3、API 1〜5本程度 | +0点 |
| **M** | 画面数 4〜10、API 6〜15本程度 | +2点 |
| **L** | 画面数 11以上、API 16本以上 | +4点 |

**スコアリング（A + B の合計）：**

| 合計点 | 技術不確実性Lv | ベースコストへの影響 |
|---|---|---|
| 0〜3点 | **Lv1** — 既存技術で対応可能 | 影響なし（最小体制で実施可能） |
| 4〜7点 | **Lv2** — 一部新規要素あり | 体制+1名 or 期間+1〜2ヶ月 |
| 8〜11点 | **Lv3** — 複雑性が高い | 体制+2名 or 期間+3ヶ月以上 |
| 12点以上 | **Lv4** — 前例なし | → PoC（技術検証）を先行 |

---

### Step 3：PdMがビジネス効果を入力する

| 入力項目 | 内容 | 例 |
|---|---|---|
| **期待効果（万円/年）** | 売上増・コスト削減など定量化できる効果 | 年間1億円のコスト削減 |

### Step 4：ツールが自動算出する

入力が完了すると、ツールが以下を自動で算出する。

| 算出される値 | ロジック | 例 |
|---|---|---|
| **要件不確実性Lv** | チェックリスト①の合計点 → Lv変換 | Lv3 |
| **技術不確実性Lv** | チェックリスト②の合計点 → Lv変換 | Lv2 |
| **調整後ベースコスト** | 前提体制 × 108万 × 期間 + 技術Lvによる調整 | 6,480万円 |
| **コスト参考値（レンジ）** | 調整後ベースコスト × 振れ幅係数 | 6,480万〜12,960万円 |
| **ROI参考値** | 期待効果 ÷ コスト参考値（上限） | 0.8倍 |

## Part 2：超概算見積もり

*チェックリストなしで最速でコストの桁感を出す方法*

### いつ使うか

- 施策候補が多すぎてチェックリストを全部回す時間がないとき
- 「そもそもやるか」の初期スクリーニングをしたいとき
- 経営層に桁感を即答する必要があるとき

### アプローチ：3つの質問で不確実性を判定

チェックリスト（20項目）の代わりに、**3問**で要件不確実性Lvを判定する。

#### Step 1：PdMがMVP像を出し、体制・期間を前提として置く

Part 1と同じ。PdMが最小閉路を定義し、エンジニアに「半年感？1年感？」を聞く。

#### Step 2：3問で不確実性をざっくり判定

| # | 質問 | Yes → |
|---|---|---|
| 1 | 「要件を決める人が明確に1人いるか？」 | No → +1 |
| 2 | 「類似案件を過去にやったことがあるか？」 | No → +1 |
| 3 | 「外部との調整が2件以上あるか？」 | Yes → +1 |

| 合計 | Lv | 振れ幅 |
|---|---|---|
| 0 | Lv1 | 1.0〜1.2倍 |
| 1 | Lv2 | 1.0〜1.5倍 |
| 2 | Lv3 | 1.0〜2.0倍 |
| 3 | Lv4 | → ディスカバリーへ |

#### Step 3：レンジを算出し、ROIで判断する

共通セクションの振れ幅係数を適用する。

**計算例：**
- 前提：3人チーム × 12ヶ月（エンジニアが「1年感」と回答）
- ベースコスト：3人 × 108万 × 12ヶ月 = **3,888万円**
- 不確実性Lv3 → 振れ幅 1.0〜2.0倍
- **コスト参考値：3,888万〜7,776万円**

**ROI判断：**
- 期待効果：年間1億円のコスト削減
- → **最悪ケースでも7,776万円。年間1億円の効果があるなら投資価値あり**

## 使いどき・使わないとき

|  | 場面 | 理由 |
|---|---|---|
| **✅ Part 1** | 複数の施策候補を比較して優先順位を決めるとき | リスク構造とコスト規模が横並びで見えることで、根拠のある議論ができる |
| **✅ Part 2** | 要件がほぼ白紙で「そもそもやるか」の投資判断が必要なとき | チェックリストなしで桁感を出し、ROIで初期スクリーニングできる |
| **✅ 共通** | 要件がふわっとしている段階でコストの目安がほしいとき | 精密な見積もりではなく「桁感」がわかれば十分な場面 |
| **❌ 使わない** | 要件が確定していて、詳細な見積もり・期間が必要なとき | エンジニアが直接見積もりを出す方が正確 |
| **❌ 使わない** | 対外的な契約金額・予算確定の根拠が必要なとき | コスト参考値は社内の意思決定支援用であり、契約根拠にはならない |
| **❌ 使わない** | すでに着手が決まっていて、優先順位の議論が不要なとき | ツールの目的は「着手前の意思決定支援」 |

## FAQ

### Q. なぜ機能積み上げでは見積もりできないの？

スクラム体制では「チームを一定期間確保する費用」がコストの実態。機能を増やしても減らしても、チームの人件費は変わらない。そのため機能を積み上げて見積もるのではなく、**「チーム人数 × 月額コスト × 期間」**でベースコストを算出する。また、初期段階ではたくさんの機能を作っても効果が出るかわからない。期間を固定し、その中で**最も価値が高いものから順に作る**。

### Q. 今と同じ機能を移行しないとUX毀損してしまうのでは？

「ユーザージャーニーの最小閉路」でMVPを定義し、Must/Later/Neverで仕分ける。「全部同等に移植する」のではなく、**価値が出る最初の到達点を先に合意する**（→ 詳細は「MVPの定義」セクション参照）。

### Q. コスト参考値はどのくらいの精度？

±50%〜2倍程度の精度。これは「精度が低い」のではなく、この段階では「桁感が合っていれば投資判断できる」のが前提。精度を上げたい場合はディスカバリーを実施する。

### Q. ビジネスが「この金額でコミットしてほしい」と言ってきたら？

コスト参考値は**コミットメントではなく、投資判断のための参考情報**。コミットメントが必要な場合は、要件が確定した段階（四半期計画時）に改めて相対見積もりを行う。基本計画に「前提条件」セクションを設け、試算であることを明記し合意しておく。

### Q. 「期間：1年」の根拠は？

PdMが定義したMVP像に対し、エンジニアが「1年感」と回答したことが根拠。精密な推定ではなく、「半年 / 1年 / それ以上」の3択で得た感覚値を前提条件として採用している。

### Q. チェックリストの項目は変えてもいい？

チーム・組織の特性に合わせてカスタマイズしてよい。ただし、**項目数を大幅に増やすとエンジニアの回答負荷が上がる**ため、各リストは10〜15項目を上限とする。追加・削除したら、スコアリングの閾値も合わせて再調整する。

## Appendix：移行戦略パターン

*大規模リプレース案件のMVPリリース戦略*

### 段階リリースの3パターン

| パターン | 概要 | 向いている場面 |
|---|---|---|
| **Strangler（つる植物）方式** | 既存の周りに新を巻き付けて、入口から新へ切替し、既存を徐々に枯らす | 入口（導線・API・画面）で切れる場合 |
| **並走＋スイッチ方式** | 新旧を並走させて、条件を満たしたら一気に切替 | 「いつ切れるか」の条件が明確な場合 |
| **Shadow（影武者）方式** | ユーザーには既存を使わせつつ、新が裏で同じ処理をして結果を突合 | 品質の確証を取りながら段階的に移行したい場合 |

### 段階リリースを成立させる2つの「逃げ道」

| 仕掛け | 内容 |
|---|---|
| **データ移行の段階化** | 全量移行を最初にやらず、「読むだけ」「新規だけ新」「一部ユーザーだけ」など段階を作る |
| **ロールバック/フェイルセーフ** | 入口スイッチ、旧へのフォールバック、二重書き込みなど。「戻せる」ことが段階リリースの心理的安全性 |

### 投資判断向けのフェーズ分割テンプレート

| フェーズ | 内容 | 投資判断のポイント |
|---|---|---|
| **MVP（◯ヶ月）** | 最小閉路を新で完結させ、対象ユーザーの一部で稼働させる | 「価値が出る最初の到達点」 |
| **Phase 2（◯ヶ月）** | 対象ユーザー拡大、主要周辺機能追加 | MVP実績を見てGo/NoGo |
| **Phase 3（◯ヶ月）** | 残存機能の取捨選択、旧の停止 | Phase 2実績を見て判断 |
