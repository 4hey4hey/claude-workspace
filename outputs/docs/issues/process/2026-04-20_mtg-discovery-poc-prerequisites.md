# mTG Discovery PoC — 開発前提事項の整理

> **これは何:** mTG Discovery PoCで個別要望がシューティング化している状態を解消するため、αβ開発を進めるうえで事前に決めておくべき前提事項を構造化した整理メモ。
>
> **3行サマリー:**
> - 個別要望が噴出する根本原因は、**「検証速度優先」と「mTG統合品質優先」が同じテーブルで未整理のまま議論されている**こと。
> - 解決策は **α=学習装置／β=移行装置** の2段階アーキテクチャに分離し、それぞれで判断基準（品質・セキュリティ・IDCS・スコープ）を別物として定義する。
> - G0（4月末〜5月上旬）までに最低限決めるべき前提は **5項目**。それ以外はG1（6月末）／G2（7月末）までに順次確定させる。

## L1：対立構造

PoCが個別要望のシューティング化に陥っている背景には、以下の3軸の構造的対立がある。**どちらも正しい**ため、どちらかを選ぶのではなく「局面で使い分ける」設計が要る。

| # | 対立軸 | A側の主張 | B側の主張 | 両者の正しさ |
|---|--------|-----------|-----------|-------------|
| 1 | 進め方 | **(A) 顧客に早く当てて手応えを掴む** — Discoveryの存在意義は学習速度。mTG統合や品質を先に固めたら、検証機会が潰れる | **(B) mTG統合を急ぐ** — PoCで作ったものがそのまま本番に移せないと、全部捨て作業になる。移行コストが膨らむ | 学習速度を上げないとG1・G2で根拠が揃わない／統合設計を後回しにすると本番移行でスコープ膨張・再実装リスク |
| 2 | 品質基準 | **軽く作る** — 紙芝居・ローカル・使い捨てで十分。品質・セキュリティを先に積むと検証が死ぬ | **mTG基準準拠** — お客様に当てる時点で東京ガス品質・セキュリティ・個人情報保護が必要 | 検証段階でフル品質を要求すると何も出せない／本番でmTG基準を外すとリリース不可 |
| 3 | ID・基盤 | **IDCS連携は後回し** — αでは独自認証・ゲスト・メールで十分 | **最初からIDCS/Auth0前提で設計** — 後から移植するほうがコストが高い | 早期に基盤に乗ると検証速度が落ちる／後乗せすると会員体系・ランクS情報アクセスの設計やり直し |

### なぜ個別要望が噴出するか

上記の対立軸が**明示されていない**まま、各ステークホルダーが自分の立場から要望を出している：
- ビジネス側：「早く当てたい」→ A側
- セキュリティ・インフラ側：「ちゃんと作って」→ B側
- 開発チーム：「どっちだよ」→ シューティング

→ 「どの判断軸をαで優先し、どの軸をβで優先するか」を**事前に合意していない**ことが、シューティングの根源。

## L2：リフレーミング — α＝学習装置／β＝移行装置 の2段階分離

**αとβは同じ延長線上のマイルストーンではない。別の目的を持つ別の装置である。**

### αは「学習装置」

| 観点 | 方針 |
|------|------|
| **目的** | お客様の課題・解決策に対する受容性を学習する（Problem/Solution Discovery） |
| **成果物の寿命** | 使い捨て前提。学習が済んだら捨ててよい |
| **品質基準** | 「学習が歪まない最低限」。本番品質は求めない |
| **セキュリティ** | PoC用途限定ルール（同意ベース、限定ユーザー、期間限定） |
| **ID** | 独立認証（ゲスト/メール/限定招待）を許容 |
| **ゴール** | **「仮説Xが検証された／反証された」状態** であって「機能Yが実装された」状態ではない |

### βは「移行装置」

| 観点 | 方針 |
|------|------|
| **目的** | αで検証済みの仮説を、mTG本番に移行可能な形で実装する |
| **成果物の寿命** | 本番移行前提。資産化される |
| **品質基準** | mTG本体と同等。セキュリティ審査通過必須 |
| **セキュリティ** | mTG基準準拠（表示ランクS/A、二段階認証、監査ログ等） |
| **ID** | IDCS（→Auth0）連携必須。mTG会員IDと紐付け |
| **ゴール** | **「本番リリース可否を判断できる状態」**（G2: 開発着手＝βのスコープ確定） |

### このリフレーミングの意味

- 「A vs B vs C（バランス型）」の議論は**不要**になる。Cではなく**時系列でA→Bに切り替える**が正解。
- αで「mTG統合どうする？」という質問が来たら「それはβの議題」と返せる。
- βで「早く当てたい」という要望が来たら「αで検証済みか？」と返せる。
- セキュリティ部門への説明も「αはPoC用途限定ルール、βで正規審査」と切り分けられる。

## 構造マップ（因果ループ分析）

### 変数

| # | 変数名 | 説明 |
|---|--------|------|
| 1 | 個別要望発生数 | ステークホルダーから差し込まれる要望の量 |
| 2 | 判断軸の明確度 | 「αで何を優先、βで何を優先」の合意レベル |
| 3 | 開発リードタイム | 仮説→検証完了の所要時間 |
| 4 | G0〜G3ゲート通過確度 | 承認に必要な根拠の揃い具合 |
| 5 | セキュリティ調整コスト | セキュリティ部門との都度調整の負荷 |

### ループ

**R1：シューティング悪循環（現状）**  
判断軸の明確度↓ → 個別要望発生数↑ → セキュリティ調整コスト↑ → 開発リードタイム↑ → ゲート通過確度↓ → （GMから「どうなってる？」と問われる）→ 判断軸の明確度↓

**B1：2段階分離による安定化ループ（目指す姿）**  
α/β分離合意↑ → 判断軸の明確度↑ → 個別要望の仕分け可能 → 調整コスト↓ → 開発リードタイム↓ → ゲート通過確度↑

### 盲点

- E1（個別要望の受付窓口）を決めても、**入口での仕分けルール（これはα？β？）が運用できないと意味がない**。長島のPMO機能が、仕分け判断まで担えるかは要確認。
- 富田がPdM代行として権限を持っても、**セキュリティ部門や認証基盤責任者との横断調整権限はないはず**。E3の権限範囲に「横断調整権限」を明記しないと、結局高松にエスカレが来る。
