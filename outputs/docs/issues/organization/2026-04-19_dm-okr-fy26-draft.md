## 2. Mission・Objective

### Mission（A版から継承）

> **作る力を、届く価値に変える。**
>
> 作る力が価値まで流れ切るように、流れを整える。ムダなく・ムリなく・ムラなく。

3つの流れ（ムダ／ムリ／ムラ）を整えるために、4つの行動原則（守る・速くする・強くする・見極める）で動く。3ロール（SM＝プロセスの番人／PO＝投資の番人／DM＝人の番人）で分担する。

### Objective（今回確定）

> **顧客価値を、正しく・速く・持続的に届けるデリバリーラインを確立する。**

* 「**正しく**」＝入口品質（KR1）＋学習ループ（KR3）。何を作るかの判断と検証の質

* 「**速く**」＝フロー効率（KR2）。着手からリリースまでの速度

* 「**持続的に**」＝健全性（基盤KR1）＋強化・育成（基盤KR2）。走り続けられる土台

### Objectiveの位置づけ

デリバリーマネジメントチームは\*\*"デリバリー機能"の責任組織\*\*である。個別プロダクトの事業成果（会員数・送客・熱狂顧客獲得）は各プロダクト責任者（片山TL／髙松TL／柏木TL／湯浅PdM等）が持ち、**当チームは「事業成果が生まれる条件」を整備する側**に回る。基盤KR3（⑦事業貢献PDCA）だけが外向きで、事業側から見た信頼度を外部検証する。

## 2.5 成果の方程式と組織能力指標の位置づけ

### 成果の方程式

> **内製開発のリソースを事業成果に転換して、投資対効果を最大化する**

```
試行回数 × 施策の精度 ＝ 事業成果
```

DPG全体としては「内製開発のリソースを事業成果に転換する」ことが至上命題。本チームはその**"How"側**として、事業側（PdM・ビジネス推進）が事業成果を出すための「試行回数 × 施策精度」のループを支える側に立つ（§2「Objectiveの位置づけ」参照）。

事業成果をスピーディーに出し続けるため、試行回数と施策精度の向上をねらう。そのためには「**仮説・検証・学習が止まらず高速に巡るプロダクト開発のカタ**」の定着が必要。

### DPG KR②（組織能力指標）の位置づけ — 先行指標である理由

> 本節以降の「KR②」は、**本チーム挑戦KR2（フロー効率）ではなく、上位のDPG KR②（内製化組織の組織能力指標）**を指す。混同に注意。

DPG KR②は**組織能力指標**。会員数・GP契約数・プロジェクト達成度などのアウトカム指標（遅行指標）は結果が出るまで時間がかかるため、組織能力指標を先行指標として置き、継続的改善で事業成果の達成確度を高める位置づけ。

| 指標の性質                       | 例               | 用途              |
| --------------------------- | --------------- | --------------- |
| アウトカム指標／遅行指標                | 会員数・GP契約数・PJ達成度 | 結果が出ないと計測できない   |
| **ケイパビリティ指標／先行指標（DPG KR②）** | 内製化組織の組織能力指標    | 早期予測・原因分析・継続的改善 |

> **早さを追うことが目的ではない。** あくまでチーム改善を促進するための指標として取り扱う。サブKPI（変更失敗率15%以下／開発者満足度SPACE軸4.0以上）で健全性を担保する。

### 4アプローチ構造とチームKRの対応

**KRを達成するためになにをやるのか？** —「**プロダクト開発の型を回す**」を構造的課題に分解して取り組む。DPG KR②は4アプローチに分解され、本チームKRとは以下の対応関係を持つ:

| DPG KR②アプローチ         | 中身                                       | 対応するチームKR                      |
| -------------------- | ---------------------------------------- | ------------------------------ |
| **①計測基盤を整備する**       | 各開発工程を計測する基盤を整備し、改善サイクルをデータで回せる状態を作る     | 基盤KR2 サブ3（計測基盤整備）／挑戦KR1〜3の共通前提 |
| **②開発に入る前の入口品質をあげる** | 実装に入る前の前提整理や要求定義の曖昧さをなくし、手戻りゼロにする        | **挑戦KR1 入口品質**                 |
| **③開発のフロー効率をあげる**    | 開発工程のムリ・ムダ・ムラをなくし、案件を停滞させずにスムーズに整える      | **挑戦KR2 フロー効率**                |
| **④学習ループを回す**        | リリースして終わりにせず、お客様の反応を見て「次に何をやるか／やめるか」を決める | **挑戦KR3 学習ループ**                |

### 目標根拠 — DORA High レベル

**挑戦KR1（入口品質）・KR2（フロー効率）の定量目標**「平均1ヶ月以内」は、DORA（Google傘下の10年研究、世界最大の開発組織ベンチマーク）の**High レベル（1日〜1ヶ月）**に相当する。メガベンチャー・デジタル先進企業の水準であり、Elite（Amazon/Netflix/Google）/ High / Medium / Low の4段階のうちHighを目標として置く。

詳細は [2026-04-21\_dm-kr2-plan-for-gm.md Appendix A](../../stakeholder/2026-04-21_dm-kr2-plan-for-gm.md) を参照。

### チーム6KR × DPG KR 接続マトリクス（全体俯瞰）

上記の「4アプローチ構造とチームKRの対応」表はDPG KR②に閉じた表だが、本チーム6KRはDPG KR②以外（KR⑥・⑦）にも接続している。全体俯瞰は以下:

| チームKR                     | DPG KR接続                   | 接続強度   | 接続の性質                 |
| ------------------------- | ------------------------- | ------ | --------------------- |
| 挑戦KR1 入口品質                | DPG KR②アプローチ②             | ◎ 直接受け | 要求定義の曖昧さ除去をデリバリー側で実現  |
| 挑戦KR2 フロー効率               | DPG KR②アプローチ③             | ◎ 直接受け | ムリ・ムダ・ムラ削減で停滞解消       |
| 挑戦KR3 学習ループ               | DPG KR②アプローチ④             | ◎ 直接受け | リリース後検証→次施策反映         |
| 基盤KR1 健全性＋開発環境            | DPG KR②サブKPI（変更失敗率／SPACE） | ○ 安全装置 | 速さ追求の暴走を防ぐ／環境の前提整備    |
| 基盤KR2 能力強化（サブ3 計測基盤）      | DPG KR②アプローチ①             | ◎ 直接受け | 計測基盤の整備で①〜④を支える共通前提   |
| 基盤KR2 能力強化（サブ3 inet発注最適化） | DPG KR⑥（コスト構造改善）          | ◎ 直接貢献 | 外部リソース効率化で原価改善        |
| 基盤KR3 事業貢献PDCA            | DPG KR⑦                   | ◎ 直接受け | 事業側との四半期相互検証で外部妥当性を担保 |

**接続強度の凡例**: ◎ 直接受け＝DPG KR側の定義文言をチームKRで直接受ける／◎ 直接貢献＝DPG KRの目標達成に数値で直接寄与／○ 安全装置＝直接の達成ではなく、暴走防止や前提条件としての防御的接続

→ 6KR中5本がDPG KRと**直接受け／直接貢献**で接続。基盤KR1のみ「安全装置」として防御的接続だが、これは挑戦KRの暴走を防ぐ意味で意図的な配置。

## 3. KR構造

### 3.0 ロジックツリー（問いの形）— Mission/Oからの分解過程

OKRをロジックツリー形式で分解すると、6つのKRは **Missionを問いに変換した6つの分解問い** から導出される（PLAID方式）:

```
Q: 作る力を価値に変えるために、デリバリーチームは何を実現すべきか？
   （= 顧客価値を、正しく・速く・持続的に届けるデリバリーラインを、どう確立するか？）
│
├─ 開発ラインに流す仕事の質と判断スピードをどう高め、
│   仕様起因の手戻りと無駄な開発をなくせるか？
│   （→ 挑戦KR1: 入口品質 / 見極める / ムダ）
│
├─ 顧客に届くスピードをどう上げるか？
│   （→ 挑戦KR2: フロー効率 / 速くする / ムダ）
│
├─ 届けた価値を検証し、学びを次のサイクルにどう反映し続けるか？
│   （→ 挑戦KR3: 学習ループ / 見極める / ムダ）
│
├─ チームが走り続けられる土台と開発環境をどう担保するか？
│   （→ 基盤KR1: 健全性＋開発環境の安定 / 守る / ムリ）
│
├─ デリバリー組織そのものを中長期にどう強くしていくか？
│   （→ 基盤KR2: 能力の多面的強化 / 強くする / ムラ）
│
└─ 事業側から見て、このデリバリー機能は本当に価値を生んでいるか？
    （→ 基盤KR3: 事業貢献PDCA / 見極める×強くする / 外部妥当性 ※DPG KR⑦直接受け）
```

### 挑戦KR1 入口品質 ─ 開発に入る前の前提整理・要求定義の曖昧さをなくし、仕様起因の手戻りをゼロにする

**Mission/Oからの分解パス**: 第1問「開発ラインに流す仕事の質と判断スピードをどう高め、仕様起因の手戻りと無駄な開発をなくせるか？」\
**対応する行動原則／効く流れ／主な担い手**: 見極める／ムダ／PO主導

#### 大目標（複数年）

「**開発に入る前の入口品質をあげる**」— 実装に入る前の前提整理や要求定義の曖昧さをなくし、手戻りゼロにする。

#### 問題認識

Naikist（Webチーム）のデプロイ頻度は1週間で、開発工程そのものは決して遅くない。「遅い」と体感される本質は**開発の前後**にある — 大きい要件のまま着手して仕様確定に時間がかかる／PO判断待ちで開発が止まる／並行PJとの依存で前提が変わり手戻りが発生する／「ついでにこれも」でスコープが膨張する。さらに仕様が曖昧なまま着手すれば、本来不要な機能や方向違いの実装を作ってしまうムダな開発のリスクも生む。**エンジニアの技術力や人数の問題ではなく、開発に入る前の仕様・判断・スコープの制御が効いていないことが「遅い」の大きな要因**。人を増やしても、仕様が固まらないまま着手する人が増えるだけで同じことが起きる。

### 3.2 KR間連動図 — 6KRがどう絡み合うか

挑戦3KRは因果ループで連動し、基盤3KRはそれぞれ異なる役割で挑戦KRを支える:

```
                ┌────────────────────────────────────────────┐
                │                                            │
                ▼                                            │
   ┌────────────────┐    ┌────────────────┐    ┌──────────┴─────┐
   │ 挑戦KR1         │ →  │ 挑戦KR2         │ →  │ 挑戦KR3         │
   │ 入口品質        │    │ フロー効率      │    │ 学習ループ      │
   │（見極める/ムダ）│    │（速くする/ムダ）│    │（見極める/ムダ）│
   └────────────────┘    └────────────────┘    └────────────────┘
                ▲                ▲                       │
                │                │                       │
                └─── 学びが仮説精度を高める（KR3→KR1） ──┘

   ─────────── 挑戦3KRすべての前提・支え（基盤3KR） ───────────

   ┌──────────────────────────────────────────────────────────┐
   │ 基盤KR1: 健全性＋開発環境                                  │
   │ → 3KRすべての「前提条件」＋「安全装置」（守る/ムリ）       │
   │   人が走り続けられる／環境が安定して動く／早さの暴走を防ぐ │
   └──────────────────────────────────────────────────────────┘

   ┌──────────────────────────────────────────────────────────┐
   │ 基盤KR2: 能力の多面的強化                                  │
   │ → 3KRすべての「中長期持続性」（強くする/ムラ）             │
   │   プロセス標準化／次世代リーダー育成／デリバリー基盤刷新   │
   └──────────────────────────────────────────────────────────┘

   ┌──────────────────────────────────────────────────────────┐
   │ 基盤KR3: 事業貢献PDCA（DPG KR⑦直接受け）                  │
   │ → 6KR全体の「外部妥当性チェック」（見極める×強くする）    │

   └──────────────────────────────────────────────────────────┘
```

**読み解きのポイント**:

* **挑戦KR1〜3の循環**: KR1が改善 → 手戻り減 → KR2フローが速くなる → 検証頻度↑ → KR3学習が回る → 学びが仮説精度を高め KR1入口品質がさらに上がる。**3つが連動して初めて成果の方程式（試行回数×施策精度）が回る**

* **基盤KR1（健全性＋環境）の二面性**: 持続可能性の前提条件であると同時に、DORAサブKPI（変更失敗率15%以下／SPACE軸4.0以上）で「速さの暴走」を防ぐ安全装置でもある

* **基盤KR2（能力強化）の中長期性**: FY26単年では成果が見えにくいが、FY27以降にDORA Highレベルへ到達するための組織能力を蓄積するKR

* **基盤KR3（事業貢献）の外向き性**: 他5KRがすべて内向き（デリバリー機能の内部最適）であるのに対し、唯一外向き（事業側から見た信頼度）の軸。デリバリーチームが内向きに自己完結しないための外部妥当性チェック
