# プロダクトオーナー（PO）採用要件

## この文書の位置づけ

myTOKYOGASネイティブアプリチームのPO採用にあたり、求める人物像・スキル要件・評価基準を整理したもの。能力マップ（6領域×Lv1-3）とPO機能文書をもとに、採用選考で使える粒度に落としている。

---

## ポジション概要

| 項目 | 内容 |
|------|------|
| ポジション | プロダクトオーナー（PO） |
| 所属 | デジタルプラットフォームG デリバリーマネジメントチーム |
| 対象プロダクト | myTOKYOGAS ネイティブアプリ |
| チーム構成 | 開発チーム（エンジニア数名）+ SM + PO |
| 上位連携先 | PdM（プロダクト戦略・ロードマップ）、サービスデザイナー、UIUXデザイナー |
| レポートライン | デリバリーマネジメントチーム マネージャー（杉崎） |

### ミッション

開発チームのリソースを活用し、myTOKYOGASネイティブアプリの**プロダクト価値の最大化**を実現する。PdMが定めた戦略・ロードマップを開発チームが実行可能なバックログに翻訳し、スプリント単位でROIを最大化する意思決定を行う。

### PdMとの役割分担

- **PdM**: 何を作るか・なぜ作るか（What/Why）の最終判断。仮説立案・顧客検証・ロードマップ策定
- **PO**: どの順番で・どこまで作るか（How much/When）の最終判断。バックログ管理・受入条件定義・日々の仕様判断

## 必須要件（MUST）

### 経験

- **スクラム開発でのPOまたはそれに準ずる役割の実務経験（2年以上）**
  - バックログの優先順位決定、受入条件定義、スプリントゴール設定を主体的に行った経験
  - 「何を作るか」だけでなく「何を作らないか」の判断を事業観点で行った経験
- **B2Cプロダクトの開発に関与した経験**
  - エンドユーザーの行動データやフィードバックをもとに意思決定した経験
- **ステークホルダー（事業部門・経営層）との調整・合意形成の経験**
  - 優先順位の根拠を非エンジニアに説明し、合意を取った実績

### スキル・能力

| 能力領域 | 必須レベル | 具体的な要件 |
|---------|----------|----------|
| 価値駆動の意思決定力 | Lv.2 | ビジネス価値・ユーザー価値・リスク・コストを総合的に評価し、優先順位の根拠を説明できる。「やらない」判断とその説明ができる |
| PBI翻訳力 | Lv.1 | ロードマップの意図を理解し、開発チームが実行可能なPBIに分解できる。受入条件を検証可能な形で記述できる |
| スクラム運営力 | Lv.1 | プランニング・リファインメント・レビューでPOとしての役割を果たせる。スプリントゴールを設定しDevsに価値を伝えられる |
| ステークホルダー調整力 | Lv.1 | 自分で判断すべきこと・エスカレーションすべきことを使い分けられる。期待値の調整ができる |
| チームとの協働力 | Lv.1 | バックログの透明性を保ち、「なぜこれを作るのか」をDevsに伝え、対話を通じてHowを引き出せる |

### マインドセット

- **タスク駆動ではなく価値駆動で考えられる**: 「チケットを消化する」ではなく「ユーザーに価値が届いたか」で成果を測る
- **マイクロマネジメントをしない**: 開発チームの自己組織化を尊重し、WhatとWhyの提示に集中できる
- **「スケジュール」ではなく「マーケット」を見る**: PMとPOの違いを理解し、マーケットのニーズ（顕在・潜在の双方）に目を向けられる

## 求める人物像

### こんな方にフィットする

- 「プロダクトの価値を上げる」ことに強い当事者意識を持てる方
- 事業側と開発側の間に立ち、双方の言語で会話できる方
- 不確実な状況でも仮説を立てて前に進められる方
- 「全部やる」ではなく「最も価値のあることに集中する」判断ができる方
- 開発チームを信頼し、自律的に動ける環境をつくることに価値を感じる方

### こんな方にはフィットしない

- スケジュール管理・進捗追跡がPOの主業務だと考える方（それはPMの仕事）
- 開発チームに「何をどう作るか」まで指示したい方（POはWhatまで、HowはDevs）
- ステークホルダーの要望を全て受け入れることが顧客志向だと考える方（ROI判断が必要）

## 選考で確認するポイント

### 面接での評価観点

| 評価観点 | 確認する質問例 | 見るポイント |
|---------|------------|-----------|
| **価値判断力** | 「バックログの優先順位をどう決めていましたか？」「やらないと決めた判断の具体例を教えてください」 | ROI・事業インパクトの視点があるか。声の大きいステークホルダーに流されていないか |
| **技術的ROIの見極め** | 下記「深掘り質問集」参照 | 技術コストの嗅覚があるか。Devsに正しい問いを立てられるか。ステークホルダーに技術的根拠で「やらない」を説明できるか |
| **PBI翻訳力** | 「事業戦略をどうやってバックログに落としていましたか？」「受入条件はどう書いていましたか？」 | Why→Whatの変換ができるか。検証可能な受入条件を書けるか |
| **チーム協働** | 「開発チームとの関わり方で工夫していたことは？」「仕様について開発チームと意見が割れたとき、どう進めましたか？」 | マイクロマネジメントせず対話でHowを引き出せるか。透明性を重視しているか |
| **ステークホルダー調整** | 「事業側から無理な要望が来たとき、どう対応しましたか？」 | 期待値を調整しつつ関係を維持できるか。エスカレーション判断ができるか |
| **PO/PM/PdMの区別** | 「POとPMの違いをどう捉えていますか？」「PdMとの役割分担はどうしていましたか？」 | スケジュール駆動ではなく価値駆動の思考ができるか |

### 深掘り質問集：技術的ROIの見極め力（「やること／やらないこと」の判断）

POはエンジニアではないが、ステークホルダーの要望に対して「技術的に投資対効果が悪い」ことを見抜き、根拠をもって断る力が求められる。以下の質問で、その嗅覚と実践力を確認する。

> 推奨度: ★★★ 必ず聞く / ★★ 時間が許せば聞く / ★ 候補者の傾向に応じて選択

#### A. 経験ベースの質問（過去の行動を聞く）

**1. 技術コストの嗅覚 ★**

> 「ステークホルダーから強く求められた機能を、技術的な理由で見送った（または後回しにした）経験はありますか？ そのとき、どうやって技術的コストに気づきましたか？」

- 見るポイント: 自分で気づいたのか、Devsから情報を引き出したのか。どちらでもよいが「気づく仕組み」を持っているかが重要

**2. Devsへの問いの立て方 ★★**

> 「開発チームに技術的なリスクやコストを確認するとき、どんな質問をしていましたか？ 具体的に聞いていた問いを教えてください」

- 見るポイント: 「工数どれくらい？」だけで終わっていないか。以下のような問いが出るかを見る
  - 「この機能を入れると、既存の○○にどういう影響がありますか？」
  - 「今の設計のまま実現できますか？ それとも構造的な変更が必要ですか？」
  - 「後から変更しやすい作り方と、しにくい作り方がありますか？」
  - 「テスト・運用の負荷はどう変わりますか？」

**3. 「やらない」の説明責任 ★**

> 「技術的な理由で要望を断ったとき、ステークホルダーにどう説明しましたか？ 相手はどう反応しましたか？」

- 見るポイント: 「技術的に難しいので」で終わらせず、事業言語に翻訳して説明できているか。「今やると○○のリスクがあり、△△を先にやった方が結果的に早い」のような説明ができるか

#### B. シナリオ質問（状況判断力を見る）

**5. 隠れたコストの発見 ★★★**

> 「事業部長から『競合がやっているから、うちも○○機能をつけてほしい。3ヶ月以内に出したい』と言われました。開発チームに確認する前に、あなたならまず何を確認しますか？」

- 期待する回答の方向性:
  - その機能が本当にユーザーの課題を解決するのか（価値の検証）
  - 今のアーキテクチャで実現可能か、構造変更が必要か
  - 実現した場合の保守・運用コスト
  - 他のバックログアイテムへの影響（何を押し出すか）
  - 3ヶ月という期限の根拠と交渉余地
- 見るポイント: 「わかりました、やります」でも「無理です」でもなく、**判断に必要な情報を先に集めにいく姿勢**があるか

**6. 品質とスピードのトレードオフ ★**

> 「リリースまであと1スプリント。開発チームから『この実装だと動くけど、テストカバレッジが低く、後で問題が出る可能性がある。ちゃんとやるなら+1スプリント必要』と言われました。ステークホルダーはリリース日を動かしたくない。あなたならどう判断しますか？」

- 見るポイント:
  - 「品質」と「スケジュール」のどちらかに思考停止で倒れないか
  - 影響範囲を確認しようとするか（どの程度のリスクか、障害が出たときの影響は、部分リリースは可能か）
  - ステークホルダーに対して選択肢とリスクを提示する形で説明できるか

**7. 性質の異なる要望の同時対応と価値分割 ★★★**

> 「チームに対して、ステークホルダーAから『基幹システムの刷新対応を進めてほしい（期限あり）』、ステークホルダーBから『ユーザー向けの新機能を早く出してほしい』と同時に求められています。どちらも重要で、チームのキャパシティは両方を同時に全力で進められるほどありません。あなたならどう進めますか？」

- 期待する回答の方向性:
  - 両方を「全部やる or どちらか諦める」の二択にしない
  - それぞれの要望を**価値単位で分割**し、小さく届けられる単位に分解する思考が出るか
    - 基幹対応: 全機能一括ではなく、影響範囲の大きい部分から段階的に対応
    - 新機能: MVP的に最小の価値単位で先にリリースし、フィードバックを得る
  - 不確実性の高いもの（新機能）こそ小さく出して検証し、確実性の高いもの（基幹対応）は期限から逆算して計画する、という使い分け
  - 「1スプリントの中で両方を少しずつ進める」のか「スプリント単位で交互に集中する」のか、チームの特性を踏まえた判断ができるか
- 見るポイント:
  - **「全部やります」（キャパ無視）でも「優先順位をつけて片方だけ」（二択思考）でもなく、価値を分割して両方から少しずつ届ける発想があるか**
  - 分割の粒度を考えるとき、「ユーザーに届く価値の単位」で切れるか（技術タスクの分割ではなく）
  - 不確実性の大小によって進め方を変える意識があるか（不確実なものほど小さく・早く検証）
  - スプリントレビューで「基幹も新機能も毎回何かしら見せられる状態」を目指す姿勢があるか

**7-2. 「全部揃ってからリリース」vs「小さく出して学ぶ」 ★**

> 「ステークホルダーから『この機能は、A・B・Cの3つが全部揃った状態でリリースしてほしい。中途半端に出すとユーザーが混乱する』と言われました。一方、開発チームの見積もりでは3つ揃えるのに4スプリントかかります。あなたならどう進めますか？」

- 期待する回答の方向性:
  - まず「なぜ全部揃わないとダメなのか」をステークホルダーに掘り下げる（本当にユーザーが混乱するのか、それとも見栄えの問題か）
  - A・B・Cそれぞれが単独でユーザーに価値を届けられるか検討する
  - 「Aだけ先に出して反応を見て、B・Cの優先度を再判断する」という選択肢を提示できるか
  - 全部揃えるリスク（4スプリント後に出して初めてフィードバックが得られる＝軌道修正が遅れる）を説明できるか
- 見るポイント:
  - ステークホルダーの「全部揃えてから」を鵜呑みにせず、その前提を検証しにいくか
  - **「早く出す＝雑に出す」ではなく「早く出す＝早く学ぶ」**という思考があるか
  - フィーチャーフラグや限定公開など、リスクを抑えつつ学ぶ手段を知っているか

**7-3. 途中で「やめる」判断 ★**

> 「2スプリントかけて開発した機能が、リリース直前のユーザーテストで『あまり使われなさそう』という結果が出ました。ステークホルダーは『もう作ったんだから出してほしい』と言っています。どうしますか？」

- 見るポイント:
  - サンクコスト（すでに投じた工数）に引きずられず、「今から出すことの価値」で判断できるか
  - 「出さない」判断をするために何を確認するか（運用コスト、ユーザー体験への悪影響、保守負荷）
  - ステークホルダーに「作ったけど出さない」を事業合理性で説明できるか（「出すと保守コストが発生し続ける」「混乱したUXが他の指標を下げる」）
  - 「学びとして次に活かす」というフレームで前向きに整理できるか


#### C. 構造的な理解を見る質問

**8. 投資対効果の判断軸 ★**

> 「ステークホルダーの要望を技術面で評価するとき、あなたはどんな軸で投資対効果を判断していますか？」

- 期待する回答に含まれるとよい観点:
  - 開発コスト（初期の工数だけでなく保守・運用コストも含む）
  - 影響範囲（既存機能への副作用、他チームへの依存）
  - 可逆性（後から変更・撤回できるか）
  - 代替手段（もっと安く同じ価値を届ける方法はないか）
  - 機会コスト（これをやることで、やれなくなることは何か）
