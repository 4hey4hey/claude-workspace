# PO体制の設計 — 中長期の体制づくりにおけるPO育成と配置の方針

> **これは何：** mTGのプラットフォーム進化（3〜5年）を支える開発体制のうち、最も手薄なPO機能の整備計画。既存POを基準に候補者を育成し、全4チームのPO機能を安定化させる方針をまとめたもの。GM承認事項②（PO配置）の根拠資料。

**3行サマリー：**

* **体制課題：** mTGプラットフォーム進化に必要な3役割（エンジニア・SM・PO）のうち、POだけ供給の仕組みがない。4チーム中、POが安定しているのはNaikistだけ

* **方針：** Naikist（福田さん）を基準チームとし、Fluppuccino（伊藤さん）・Guardians（室山さん）のPO候補を段階育成。FY26はPO育成の型を確立する年

* **承認依頼：** 森さんDiscovery PO兼務の継続 / 室山さんGuardians PO候補化（Kraken段階移行）

***

## 1. 中長期的にやっていくこと

> **キーメッセージ：** mTGは3〜5年かけてプラットフォームに進化させる必要がある。14会員制度統合・Kraken移行・EA基盤接続・マイクロサービス化が並行で走る中長期の取り組み。内製化体制のゴールは、施策の打席数を増やして成果を継続的に出し続けられる状態を作ること。

myTOKYOGASは「ガス・電気の会員サイト」から「総合エネルギー・暮らしサービスのプラットフォーム」へ進化する必要がある。具体的には：

* **14ある会員制度の段階的統合**

* **Kraken移行に伴う顧客接点の再設計**

* **EA/CIAM/SSO基盤との接続による認証・ID統合**

* **サービス拡張に対応できるマイクロサービス化**

これは1〜2年の単発施策ではなく、**3〜5年かけて継続的にプロダクトを進化させる取り組み**。「プロジェクトを終わらせる」仕事ではなく「プロダクトの価値を高め続ける」仕事。

内製化体制のゴールは、**施策の打席数を増やして、成果を継続的に出し続けられる状態を作ること**。体制もそれに合わせて設計する必要がある。

***

## 2. そのために必要な体制

> **キーメッセージ：** 打席数を増やすにはチーム数・スピード・ムダ打ちしないの3要素が必要で、3つ全部にPOが関わる。しかし4チームあるのに施策を打ち切れるのはNaikistだけで実質1ライン。エンジニアとSMは供給設計があるが、POだけ仕組みがない。POが最大のボトルネック。

### 打席数を増やすために必要な3つの要素 — チーム数・スピード・ムダ打ちしないの3要素すべてにPOが関わる

施策の打席数を増やすには、3つの要素が必要になる：

| 要素          | 意味                  | 現在の阻害要因                                        |
| ----------- | ------------------- | ---------------------------------------------- |
| **チーム数**    | 同時に施策を回せるラインの数      | 4チームあるが、施策を打ち切れるのはNaikistだけ＝**実質1ライン**         |
| **スピード**    | 1つの施策をどれだけ速く打ち切れるか  | 大きいまま着手→手戻り、判断待ち、スコープ膨張で遅くなる（第二のKrakenをつくらない ） |
| **ムダ打ちしない** | 前提が揃っていない施策に打席を使わない | 前提未整理のまま着手→方向違い→やり直し＝打席を浪費（二重投資などを防ぐ）          |

チームが1つの施策を打ち切るには、3つの役割がセットで必要になる：

* **エンジニア** — コードを書いてリリースする

* **SM** — 開発プロセスを回し、チームが前に進めるようにする

* **PO** — 何を作るか・作らないかを判断し、小さく分割して検証サイクルを回す

**3つの要素すべてにPOが関わる。** チーム数を実質的に増やすにはPO機能が安定する必要がある。スピードを上げるにはPOが小さく分割しスコープを制御する必要がある。ムダ打ちを減らすにはPOが「今ではない」と判断する必要がある。

### 4チーム体制の現在地 — 4チームあるが施策を打ち切れるのはNaikistだけ。エンジニア・SMは供給設計があるがPOだけない

| チーム                   | SM        | PO                | 状態                       |
| --------------------- | --------- | ----------------- | ------------------------ |
| **Naikist（Web）**      | 堤さん（SM補佐） | 福田さん              | CT安定 ✅ 基準チーム             |
| **Fluppuccino（モバイル）** | 中道さん      | 伊藤さん（PO）          | CT未安定                    |
| **Discovery**         | 長島さん      | 森さん（兼務）           | CT未安定                    |
| **Guardians（MS）**     | 不在        | 迫田さん（暫定）→ 室山さん検討中 | 回ってはいるが迫田さんの開発リソースが削れている |

エンジニアは暖簾分け方式で供給設計がある。SMも中道さん（経験者）+ 長島さん（育成中）+ 採用で整備が進んでいる。**POだけ供給の仕組みがない。** 4チームあるのに施策を打ち切れるのが1チームだけという状態は、POが最大のボトルネックであることを意味する。

## 3. なぜPOが必要か

> **キーメッセージ：** mTGが扱う開発テーマは不確実性が高く、最初から正しい要件を定義することは不可能。だからこそ「小さく分割し、検証しながら進め、やらないことを決める」判断者が必要であり、それがPO。Krakenで起きた5ヶ月遅延・カスタマイズ肥大・計画引き直し・影響額14.9億円は、この判断が構造的に弱かったことが原因。人を増やしても構造は変わらない。

### 不確実性の高い開発で投資をムダにしないために — 小さく分割し検証しながら進める判断者が必要。人を増やしても構造は変わらない

エンジニアは「コードを書く人」。しかし、エンジニアへの投資が事業成果に変わるかどうかは、**何を・どの大きさで・どの順番で作るか**の判断で決まる。

mTGが扱うのは、14会員制度の統合やKraken移行のような不確実性の高いテーマ。こうしたテーマでは、最初から正しい要件を定義することは不可能。だからこそ**小さく分割し、検証しながら不確実性を段階的に減らして進める**必要がある。

この「小さく分割する」「検証してから次に進む」「やらないことを決める」判断をする人がPO。POがいない場合、実際には以下のことが起きる：

| 起きていること | 結果 |
| --- | --- |
| 大きい要件を分割せずにそのまま着手 | 途中で「やっぱりこう」が発生し、手戻りで1〜3週間のロス |
| 不確実なまま全部作ってからリリース | リリース直前に想定外が見つかり、大幅な手戻り |
| 「ついでにこれも」でスコープが膨張 | 予定通りに終わらない＝遅く見える |
| 前提が揃っていない案件にも打席を使う | 方向違いの実装を作ってしまうムダな開発 |
| PO判断・ステークホルダー確認待ちで開発が止まる | エンジニアは手が空いているのに進まない |

これはエンジニアの技術力や人数の問題ではない。**人を増やしても、大きいまま・不確実なまま着手する人が増えるだけで同じことが起きる。**

### Krakenで実際に起きたこと — 大きいまま着手して5ヶ月遅延、やらない判断ができずカスタマイズ肥大、学習ループなく計画引き直し。影響額14.9億円

Krakenプロジェクトで構造的に同じことが起きている。

**大きい要件のまま着手 → リリース直前で仕様変更が噴出**

新規受付機能の開発で、リリース日間近になってからオペレーション確認を行ったところ、仕様変更・追加が発生した。Kraken側も「初期要件定義の決定の仕方に課題」があったと振り返っている（FY25 Q2ステコミ）。要件を最初から完璧に定義できなかったことが問題ではない。**大きい要件を分割せず、途中で検証するサイクルを回さなかった**ことが問題。小さく切って早い段階でオペレーション確認を組み込んでいれば、リリース直前の噴出は防げた。

**「やらない」を決められない → カスタマイズが止まらない：**
TG固有のカスタマイズが多発し、これが開発遅延の主因になった。Krakenには標準機能があるが、TG側の業務をKrakenに合わせるのではなく、Krakenをカスタマイズする方向に判断が倒れ続けた。「まず標準機能で小さくリリースし、本当に必要なカスタマイズだけを後から追加する」という段階的な判断——つまり**やらないことを決めて、小さく始める判断**——ができる人がいなかった。

**不確実性を検証せず着手 → 5ヶ月遅延**

新規受付機能は7月リリース予定だったが、実際に完了したのは12月（5ヶ月遅れ）。主因はKrakenの標準機能とTGの新規受付機能の間に**想定以上に大きなギャップ**があったこと。さらに、電気ガスセット契約では、ハンド前提の運用とシステム自動前提のKrakenが混在し、調整が困難になった。ギャップの大きさは事前にはわからない。だからこそ**最もリスクの高い部分を先に小さく試し、ギャップを早期に検知する**進め方が必要だった。

**学習ループがない → 同じパターンの繰り返し：**
8月の電力CIS脱却を断念（Q1）、その後のスケジュールも遅延（Q2）。四半期ステコミごとに計画引き直しが続いた。新規受付遅延の影響額は設備6.1億円＋諸経費8.8億円＝**約14.9億円**。四半期ごとに「なぜ遅れたか」を検証し、次の四半期の計画に学びを反映するループがあれば、同じパターンの繰り返しは避けられた。

***

これらはエンジニアの技術力の問題ではない。**「小さく分割する」「不確実性を早く検証する」「やらないことを決める」「学びを次に活かす」——この判断が構造的に弱かった**ことが原因。この判断を担うのがPO。

Naikistが安定しているのは、PO機能とチームの成熟度の両方が揃っているから。他の3チームとの差はPO機能だけではないが、**POが機能しなければチームの成熟も進まない**。PO機能の整備がチーム成熟の前提条件になる。

> POの役割の詳細な分析は [PO不在問題の構造分析](2026-03-28_po-function.md) を参照。

## 4. POの増やし方

> **キーメッセージ：** PO候補にはドメイン知識・SH調整・判断経験・エンジニアとの信頼・段取り力の5条件が必要。調達は業務知識と信頼が既にある内部登用を最優先にしつつ、スピードを上げたい場合は外部採用も併用する。3人目以降は施策を1本任せる業務アサインで2年かけて育てる。育成と打席数の拡大が両立する。

### PO候補に必要な5つの条件 — ドメイン知識・SH調整・判断経験・エンジニアとの信頼・段取り力。Krakenで欠けていたものから逆算

セクション3のPOの役割（分割・優先順位・スコープ制御・学習ループ）から逆算すると、PO候補には5つの条件が必要になる：

| # | 条件                   | なぜ必要か                                                                         | Krakenで欠けていたもの                            |
| - | -------------------- | ----------------------------------------------------------------------------- | ----------------------------------------- |
| 1 | **業務ドメインの知識**        | 要件を小さく分割するには、業務の全体像と依存関係がわかっている必要がある。知らなければ「どこで切れるか」が判断できない                   | Kraken標準機能とTG業務のギャップを事前に見積もれなかった          |
| 2 | **ステークホルダーとの調整経験**   | 「やらない」判断はPO単独では通せない。事業側を説得し、落としどころを見つける経験が必要                                  | TGカスタマイズが止まらなかった。「標準機能で妥協する」交渉ができる人がいなかった |
| 3 | **判断の経験**            | 優先順位・トレードオフ・「今ではない」判断。正解のない状況で意思決定した経験                                        | 四半期ごとに計画引き直し。「何にいつ打席を使うか」の判断を下せる人がいなかった   |
| 4 | **エンジニアと対話でき、信頼される** | 技術の深い知識は不要だが、チームと対話でリスクを検証する力が要る。加えて、POの判断でチームが動くには信頼が必要。信頼は権限ではなく、姿勢と実績で築くもの | 大きい要件のまま着手。技術的なリスクを開発チームと事前に検証するプロセスがなかった |
| 5 | **段取り力**             | 施策を前に進めるには、関係者・タスク・依存関係を整理し、適切な順序で物事を動かす力が要る。POは「判断する人」であると同時に「施策を回す人」        | 複数コフォートの同時対応で体制混乱。依存関係の整理が追いつかないまま開発に入った  |

条件1〜3はスキル・経験であり、後天的に獲得可能。条件4の信頼はチームに入りながら築くもの。条件5は仕事の基本動作であり、施策を1本回す経験で身につく。

### 調達方針 — 業務知識と信頼がある内部登用を最優先。スピードを上げたい場合は外部採用も併用。inetは異動リスクで軸にしづらい

5条件を踏まえると、PO調達には4つの方針があり、状況に応じて使い分ける：

| 方針         | いつ選ぶか                                | 強み・制約                                                       |
| ---------- | ------------------------------------ | ----------------------------------------------------------- |
| **内部登用**   | **最優先**                              | 業務知識（条件1）・社内システムの知識が既にある。チームとの信頼（条件4）も一定程度築かれている。最も立ち上がりが速い |
| **外部採用**   | 内部に適切な人がいない場合、またはチームを増やしてスピードを上げたい場合 | PO経験者を即戦力で獲得できる。ただし業務ドメイン知識（条件1）と信頼（条件4）の立ち上がりに時間がかかる       |
| **ジュニア育成** | 中長期の仕込み                              | 既存のPOにつけつつ、施策を回す経験をさせ、2年程度かけて育てる。打席数の拡大と育成が両立する             |
| inet活用     | 難しい枠                                 | PO候補になりうるが、異動リスクがありPO供給の軸にはしづらい                             |

内部登用を最優先にするのは、条件1（業務知識）と条件4（信頼）を既に持っているから。外部採用は、PO経験やスキルがあっても業務ドメインとチームの信頼をゼロから築く必要がある。ただし、内部に適切な候補がいない場合や、チーム数を増やしてスピードを上げたい場合には外部採用も積極的に検討する。

### GM承認事項②：現在のPO配置 — 森さんDiscovery PO兼務継続と室山さんGuardians PO候補化の承認を依頼

及川GMに承認を求める3項目のうちの1つとして、以下のPO配置方針を提示する。

| チーム             | 現状             | 方針                                            |
| --------------- | -------------- | --------------------------------------------- |
| **Naikist**     | 福田さん（PO）       | 安定運用を継続。基準チームとしてPO動作の型を他チームに展開                |
| **Fluppuccino** | 伊藤さん（PO候補）     | 福田さんの指導の下で段階的に権限を拡大。Q2で独力判断範囲を明確化             |
| **Discovery**   | 森さん（兼務PO）      | 兼務を継続。Ready定義導入・手戻り分類運用をDiscoveryで実践          |
| **Guardians**   | 迫田さん（暫定）→ 室山さん | 室山さんのGuardians PO候補化を検討。Kraken業務知識を活かし段階的にPO化 |

### 中長期のPO育成 — 特別なプログラムではなく施策を1本任せる業務アサインで5条件を自然に身につけさせる。2年でペアPOへ

室山さん・伊藤さんは既にPO候補としての条件を持っている人材。しかし3人目・4人目のPOを確保するには、ジュニアを中長期で育てる道筋が必要になる。

鍵は「PO育成プログラム」を特別に組むのではなく、**施策を1本任せる業務アサイン**を通じて5条件を自然に身につけさせること。

#### 1年目：施策を1本回す

「この施策のオーナーとして、企画から検証まで回してみて」という業務アサイン。POがメンターとして伴走し、判断の型をフィードバックする。

| やること                        | 身につく条件                 |
| --------------------------- | ---------------------- |
| 施策の背景・目的を理解し、関係者に説明する       | 条件1（ドメイン知識）・条件2（SH調整）  |
| PBIに分解し、チームと優先順位を決める        | 条件3（判断）・条件4（エンジニアとの対話） |
| スプリントごとに進捗を確認し、スコープを調整する    | 条件3（スコープ制御）            |
| レビューで「狙い通りか？」を検証し、次を決める     | 条件3（学習ループ）             |
| うまくいかなかったときに「やめる」「変える」を判断する | 条件3（やらない判断）            |
| チームと一緒に施策を回す過程で信頼が築かれる      | 条件4（エンジニアの信頼）          |

1年で「施策を1本、自分で回し切った」実績ができれば、5条件は実践を通じて身についている。

#### 2年目：ペアPOとして育成フレームに乗る

1年目の実績をもとに、セクション5の育成フレーム（Phase 1: 観察 → Phase 2: 実践 → Phase 3: 独り立ち）に入る。1年目で施策を回した経験があるため、Phase 1（観察）は短縮でき、早期にPhase 2（ペアPO実践）に進める。

```text
1年目: 施策を1本回す（業務アサインとして。POがメンター）
          ↓ 5条件が実践を通じて身につく
2年目: ペアPO → 独り立ちへ（育成フレームに乗る）
```

この方式の利点：

* 「PO育成」という特別なプログラムを組まなくていい

* 施策を任せること自体がチームの打席数を増やす（育成とアウトプットが両立する）

* 本人にとっても「PO候補になれ」より「この施策を回して」の方が自然に動ける

## 5. PO育成方針

> **キーメッセージ：** PO候補の育成は、既存POとのペアで判断の型を日常的に体験する実践と、Odd-eからの外部フィードバックで判断の質を上げる支援の2本柱で進める。候補者だけでなく既存PO自身も含めてPO機能全体を底上げする。SM育成で既にOdd-eが入っているのと同じ構造。

### 育成の2本柱 — 既存POとのペアで判断の型を日常的に体験し、Odd-eの外部フィードバックで判断の質を上げる。既存PO自身も底上げされる

PO候補の育成は、**日常の実践（既存POとのペア）**と**外部フィードバック（Odd-e支援）**の2本柱で進める。

```text
日常：既存POとペアで実践（判断の型を体験・実践）
  ↕
Odd-e：既存PO＋候補者をセットでフィードバック・指導・1on1
  → 候補者だけでなく既存PO自身のPO力も一緒に底上げされる
```

**既存POとのペア（日常の実践）：**

* 候補者が既存POと一緒にPO業務を回し、判断の型を体験する

* 段階的に候補者が判断を担う範囲を広げていく

* 候補者が一部を引き取ることで、既存POの負荷はむしろ下がる

**Odd-eの支援（判断の質の底上げ）：**

* 既存POと候補者をセットで見て、判断の質にフィードバックを入れる

* 候補者への1on1で個人の課題に対応する

* 「既存POの型をコピーして終わり」にならない。既存PO自身も含めてPO機能全体が底上げされる

* SM育成で中道さん＋長島さんをOdd-eが見ているのと同じ構造をPOにも作る

> Odd-eのPO育成支援の具体的なスコープ（頻度・内容）は、Odd-eとの打ち合わせで詰める。
