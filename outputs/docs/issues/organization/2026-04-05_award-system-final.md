## 1. 目的

「組織として何を価値ある貢献と見なすか」を明示し、見えにくい仕事が正当に評価される構造をつくる。

当グループは、プロダクト開発・基幹リプレース・基盤刷新・サービスデザイン・組織運営など多様な仕事が並走している。従来の評価では「目に見える成果を出した人」に偏りやすく、以下の貢献が構造的に過小評価されるリスクがある。

- 基盤・リプレース系（Kraken等）：成果が見えにくい、完了が年度をまたぐ、成功＝何も起きない
- 支援職（SM、PO等）：成果がチームに帰属し、うまくいくほど本人が透明になる
- 専門職（エンジニア、デザイナー等）：技術的判断・設計の価値が事業指標に直接現れにくい
- 運用・保守：「守り」が当たり前とみなされる

本制度は、Valueに基づく評価軸を設計することで、これらの構造的な不公平を是正し、多様な貢献を公平に評価する仕組みとする。

## 2. Valueの位置づけ

### グループValueへの昇格

今回の表彰を機に、mTG MVVのValueを**デジタルプラットフォームＧ全体の行動指針**として位置づける。キックオフで宣言する。

**背景：**
- Valueの3項目（お客さま起点・圧倒的当事者意識・感謝と尊敬）はmTG固有ではなく、TGK・EA含むグループ全体の仕事に当てはまる内容
- SDチーム（片山TL）もTGK担当メンバー含めMVVを意識して業務している
- 対象をmTGに限定すると、TGK・EAメンバーのモチベーションにマイナスに働くリスクがある
- Notion上もMVV説明の場も「myTOKYOGASの〜」で定義されているため、名称を「グループValue」へ変更し違和感を解消する

### 3つのValue

| Value | 核心 |
|---|---|
| **お客さま起点** | 常にお客さま起点で考え判断する。顕在化されていない課題を発見し、真の価値を提供する |
| **圧倒的当事者意識** | 自ら考え行動を起こす。果敢に挑戦し、やり抜く。困難は当たり前 |
| **感謝と尊敬** | 共に価値を創るすべての人に感謝し、尊敬する。自分自身が誇れる仕事をしよう |

## 3. 表彰の枠組み

### 2軸による推薦

**2つの軸の掛け合わせ**で推薦する。軸1は「貢献がどこに効いたか」、軸2は「どう効いたか」を表す。これにより、専門職・支援職を含むあらゆる貢献が自然に表現できる。

#### 軸1：貢献の種類（5カテゴリ × Value紐づけ）

| カテゴリ | Value | 意味 | 判断基準 | 例 |
|---|---|---|---|---|
| **今期の成果** | お客さま起点 | 今年度、顧客やビジネスに届いた価値がある | 今期中にリリース・達成した成果か | 機能リリース、KPI改善、顧客価値の実現 |
| **未来への布石** | 圧倒的当事者意識 | 将来の成果のために土台を作った | 終わりのあるPJで、完成すれば組織の能力が不可逆的に上がるか | 基幹リプレース（Kraken移行）の推進、認証基盤刷新（IDCS→Auth0）、EA/ID基盤の要件定義、デザインシステム構築、PO・SM等の後継育成、開発プロセスの型づくり |
| **守りと接続** | お客さま起点 | 事業継続・チームの健全性を支えた | 終わりのない仕事で、やめた瞬間に劣化が始まるか | 本番障害の対応・再発防止、セキュリティ対応、既存システムの維持管理、チーム間の依存関係の調整・ブロッカー除去、オンボーディング支援 |
| **挑戦・実験** | 圧倒的当事者意識 | 不確実な中で踏み出し、成否にかかわらず組織の学びを増やした | 結果ではなく「踏み出した行動」と「得られた学び」で評価できるか | DiscoveryチームのPoC、MVP仮説検証（OKR KR①）、新技術の導入検証、前例のないやり方への最初の一歩 |
| **越境・共創** | 感謝と尊敬 | 組織やチームの境界を超えて、人・情報・判断をつないだ | 自分の持ち場を超えて、他者・他チーム・他部の力を引き出したか | ビジネスサイド⇔開発の翻訳、KT社⇔mTG側の調整、他部（エネ事業革新部等）との合意形成、iネットとの協働設計 |

> **「未来への布石」と「守りと接続」の違い：**
> 迷ったら「その仕事が完了したらなくなるか」で判断する。**終わりがあるなら布石、終わりがないなら守り。**
> - **未来への布石**＝今は存在しないものを新たにつくる仕事。完成すれば組織の能力が不可逆的に上がる（例：Kraken移行が完了すればホスト脱却、Auth0が稼働すればSSO実現）
> - **守りと接続**＝今あるものを壊さず動かし続ける仕事。やめた瞬間に劣化が始まる（例：障害対応を止めればサービス停止、チーム間調整を止めればブロッカーが滞留）

> **「未来への布石」と「挑戦・実験」の違い：**
> 迷ったら「成果の確度」で判断する。**ゴールが見えているなら布石、見えていないなら挑戦。**
> - **未来への布石**＝完成形が定義されており、確実にやり遂げることに価値がある（例：Kraken移行、Auth0導入）
> - **挑戦・実験**＝やってみないと分からない領域に踏み出すこと自体に価値がある（例：PoC、MVP検証、新プラクティスの試行）

#### 軸2：貢献の性質（どう効いたか）

| 選択肢 | 意味 | 該当しやすい役割 |
|---|---|---|
| **意思決定・推進** | スコープ判断、優先順位づけ、合意形成 | PO、PM、SM |
| **技術・専門性** | 技術的判断、設計、品質、難問の突破 | エンジニア、デザイナー |
| **発見・定義** | 課題発見、ユーザー理解、要件定義 | サービスデザイナー、リサーチャー |
| **仕組み化・横展開** | プロセス改善、ナレッジ共有、標準化 | 全職種横断 |

#### 掛け合わせの例

| 対象者 | 軸1 | 軸2 |
|---|---|---|
| PO（Kraken連携のスコープ分割を主導） | 未来への布石 | 意思決定・推進 |
| エンジニア（難しいAPI設計を解いてリリースに直結） | 今期の成果 | 技術・専門性 |
| デザイナー（デザインシステムを構築） | 未来への布石 | 仕組み化・横展開 |
| サービスデザイナー（リサーチから施策の方向転換に貢献） | 挑戦・実験 | 発見・定義 |
| SM（チームの自律性を高め、ベロシティを改善） | 守りと接続 | 仕組み化・横展開 |
| PM（ビジネスサイドと開発の間で要件を翻訳・調整） | 越境・共創 | 意思決定・推進 |
| エンジニア（PoCで新技術の可能性を検証） | 挑戦・実験 | 技術・専門性 |
| SM（他部との依存関係を調整しブロッカーを解消） | 越境・共創 | 仕組み化・横展開 |

## 4. 推薦フォーマット

推薦文は以下の3問で構成する。この形式により、プロダクト開発も基盤もSMもエンジニアも、同じフォーマットで公平に語れる。

| 問い | 狙い |
|---|---|
| **何をしたか（事実）** | 行動・判断・成果を具体的に記述する。数字・固有名詞を使う |
| **この貢献がなかったら何が起きていたか（反実仮想）** | 見えにくい仕事の価値を可視化する |
| **チーム・組織にどんな変化をもたらしたか（影響）** | 短期成果だけでなく、持続可能性・再現性・組織学習への波及を語る |

補足として、**「この貢献を持続可能にするために組織として取り組むべきこと」**を一文添える。表彰を個人の称賛で終わらせず、組織課題への接続点とする。

## 5. 推薦・選考プロセス

### 対象範囲

**デジタルプラットフォームＧ全体**（組織表上の全チーム＋スクラムチーム横断）から、グループ一括で選出する。TGK担当・EA推進メンバーも含む。

**対象者の定義：** 「FY2025年度中に当グループで活動していた方」。組織表上の所属ではなく実態ベース。FY26に異動した方も対象に含める。

チームごとの枠割り振りは行わない。理由は以下の通り。

- 組織表チームとスクラムチームの二重構造があり、「どのチームから推薦するか」が曖昧になる
- チーム規模差（ソフトウェアエンジニアリングT 12名 vs サービスデザインT 4名等）で不公平が生じる
- 福田さんのような複数PJ横断の貢献が、単一チームの視点では全体像が見えない

### プロセス

| ステップ | 内容 |
|---|---|
| **1. 推薦** | 全TLが推薦文を提出する。自チーム以外の推薦も可 |
| **2. 推薦上限** | 1TLあたり最大2名（絞る力を求める） |
| **3. ピアコメント** | 推薦された人に対し、一緒に働いたメンバーから一言コメントを収集（任意） |
| **4. 選考** | 及川GM ＋ TL合議で決定 |
| **5. バランスチェック** | 軸1の5カテゴリに偏りがないか確認する（全カテゴリ必須ではないが、「今期の成果」だけに集中していないかは是正する） |
| **6. 発表** | 推薦文をそのまま共有する（選考理由の透明性を担保） |

### 受賞人数

全体で5名前後を想定。

## 6. 投票アンケート

TL推薦に加え、全メンバーからの投票を材料として収集する。**票数で自動決定しない。投票＋コメントを材料にGM＋TL合議で受賞者を決定する。**

### アンケート冒頭文

> **FY2025 Value表彰 — あなたが選ぶ「Valueを大切にしていた人」**
>
> デジタルプラットフォームＧの3つのValueを、FY2025に最も大切にしていたと思う人に投票してください。
> 日々一緒に働く中で「この人はこのValueそのものだな」と感じた人を選んでください。
>
> - Valueごとに1名を選び、選んだ理由を一言で教えてください
> - 自分以外のデジタルプラットフォームＧメンバーが対象です
> - FY2025年度中に当グループで活動していた方が対象です（年度途中の異動者を含む）
> - 同じ人を複数のValueで選んでも構いません
> - 回答は集計後、本人へのフィードバックとして共有されます

### 集計・選考の運営ルール

| 項目 | 内容 |
|---|---|
| 投票対象 | デジタルプラットフォームＧ 全メンバー（TGK担当・EA推進含む） |
| 投票者 | 同上（自分自身への投票は不可） |
| 匿名性 | 集計者（杉﨑）には回答者が見える。発表時には「寄せられたコメント」として匿名で共有 |
| 集計方法 | Valueごとに得票数とコメント内容を整理 |
| 選考方法 | **投票結果（票数＋コメント）を材料に、GM＋TL合議で受賞者を決定**。最多得票で自動決定しない |
| 選考の観点 | コメントの質・具体性を重視する。票が少なくてもコメントの説得力が高ければ候補になる |
| 発表方法 | 受賞者の名前 ＋ 寄せられた一言コメント（匿名）をまとめて共有 |
| 発表の場 | キックオフ（グループ全体の取り組みとして位置づけ、mTG限定の文脈に置かない） |

### 設計上のポイント

- **Value原文をそのまま掲載**する。抽象化せず、読んだ上で「この人だ」と思える状態にする
- **理由の一言コメントが最も重要**。票数だけだと人気投票になるが、コメントがあることで「なぜその人か」が言語化され、表彰の説得力が上がる
- **票数で自動決定しない**。人数比の大きいmTG系メンバーに票が集中しやすい構造のため、TL合議で公平性を担保する
- 発表時にコメントを本人に届けることで、**表彰が「周囲からのフィードバック」として機能する**

## 7. 選考時の判断軸

推薦文を読んだうえで、以下の4軸で該当性を判断する。点数化・序列化はしない。

| 判断軸 | 問い |
|---|---|
| **具体性** | 事実が特定できるか。「頑張った」「活躍した」ではなく、何をしたかが書かれているか |
| **影響の大きさ** | 反実仮想が説得的か。「なかったら止まっていた」の範囲はどこまでか |
| **再現困難性** | その人でなければできなかったか。属人性が高いほど貢献の代替不可能性が高い |
| **組織への波及** | 個人の成果で閉じず、チーム・組織の能力を引き上げたか |

## 8. アンチパターン（避けるべきこと）

| やってはいけないこと | 理由 |
|---|---|
| 稼働量・残業時間を評価基準にする | 「頑張った＝長時間働いた」のメッセージになる |
| 推薦文が感想文になっている | 事実と反実仮想を求める。差し戻して書き直しを依頼する |
| 属人性の高さを称えるだけで終わる | 「すごい」で終わらず、持続可能にするための組織課題まで接続する |
| 「今期の成果」ばかり表彰する | 軸1のバランスチェックで是正する |
| 毎年全チームから均等に出そうとする | 目的は公平な分配ではなく、最も価値のあった貢献を選ぶこと |

## 9. 参考：推薦文の例（福田 美桜 / 未来への布石 × 意思決定・推進）

### 何をしたか（事実）

FY25を通じて、福田さんはmyTOKYOGAS唯一の稼働POとして、エンハンス開発の意思決定・スコープ管理・ステークホルダー調整を一手に担った。

特筆すべきはKraken直連携プロジェクトへの主導的な関与だ。リリース時期の揺れ・料金体系の未確定・データ担保の不確実性が残る状況下で、「電気単独をファーストリリースとする」スコープ分割案を自ら提案。API仕様調査と認証方式の確認を先行させ、「最小限の表示・利用が成立する」ことを技術的に見極めたうえで、関係者が意思決定できる材料を揃えた。結果として「5月末に開発合意→8月に結合テスト」というマイルストーンを関係者間で合意形成し、2Q以降の推進基盤を確立した。

また、当番制が機能しない局面での障害対応、Naikistの維持管理・緊急対応、室山さんへの計画的なタスク委譲、内製化サミットでの外部発信も並行して担っている。

### この貢献がなかったら何が起きていたか（反実仮想）

Krakenは料金体系確定まで着手判断が棚上げになり続け、少なくとも2〜3四半期の開発着手遅延が発生していた可能性が高い。「決まってから動く」という判断になりやすい不確実性の高い局面で、「まず成立するスコープで動く」という提案がなければ、関係者の意思決定の前提となる技術的事実が揃わないまま会議が繰り返されていたはずだ。

また、唯一の稼働POが不在・機能不全だった場合、スプリントレビューのプロダクトゴール判断・バックログの優先付け・ビジネスサイドとの要件折衝はいずれも停止に近い状態になり、開発チームのアウトプットはベロシティ維持ではなく待機状態に転じていた。

### チーム・組織にどんな変化をもたらしたか（影響）

Krakenプロジェクトにおいて「小さく作って前に進める」という進め方の実績を組織内に示したことで、不確実性の高いプロジェクトの進め方の雛形が生まれた。高松さん・及川さんからの信頼獲得は、ビジネスサイドと開発の間に翻訳レイヤーが機能していることの証左であり、今後の施策合意形成コストの低減に直結する。

室山さんへの権限委譲の進捗は、属人依存のリスクを段階的に分散させる動きであり、チームとしての持続可能性を高めている。外部発信（内製化サミット登壇）は、組織のケイパビリティを対外的に可視化する行為として、採用・パートナー関係にも波及効果を持つ。

---

この貢献を持続可能にするためには、現在1名に集中しているPO機能を複数体制に分散させることが組織として取り組むべき構造課題である。
