# バリューチェーンの上流設計 — POが打席に立つ前に、弾を込める仕組みを作る

> **これは何：** PO体制資料（[po-supply-pipeline.md](2026-03-31_po-supply-pipeline.md)）の続編。POが打席に立てるのは「要求が見えてから」。その手前——ビジネス要件の整理・業務フロー調整・ステークホルダー間合意——が仕組み化されていない問題を構造的に示し、上流プロセスの整備方針を提案するもの。

**3行サマリー：**

* **構造問題：** 4チームのPO機能を整備しても、POに渡す「弾」の前提が揃っていなければ空振りが増えるだけ。別部署のサービスをmTGに組み込むような施策では、as-is/to-be整理・ビジネス試算・業務フロー調整・既存業者統廃合が開発着手前に必要だが、この工程の担い手と進め方が未定義

* **方針：** バリューチェーン全体を可視化し、POの守備範囲（How/When）の手前にある「施策の前提整理」を、成果物と完了基準を持つ活動として定義する。担い手は施策の性質に応じてPdM・事業部門・兼務企画担当で使い分ける

* **承認依頼：** バリューチェーン全体像の認識共有 / 上流プロセスの担い手方針 / 施策Ready定義のQ2導入方針

***

## 1. 打席数を増やしても、空振りが多ければ成果にならない

> **キーメッセージ：** PO供給は整備中（前作）。次の問題は「POに渡す弾の品質」。打席数 × 打率 = 成果。打席を増やすのがPO供給、打率を上げるのが上流整備。KR②実行計画の課題③「前提が揃っていないものに打席を使っている」の根本原因がここにある。

[PO体制資料](2026-03-31_po-supply-pipeline.md)では、4チームのPO機能を安定させて「施策の打席数を増やす」方針を示した。[KR②実行計画](../../stakeholder/2026-04-05_dm-kr2-plan-for-gm.md)では、入口品質・フロー改善・学習ループの3つで「打席数を増やし、精度を上げる」方針を承認いただいた。

しかし、打席に立つ前にもう1つの問題がある。**POに渡される施策の前提が揃っていない。**

```
成果 ＝ 打席数 × 打率

打席数を増やす ← PO供給の整備（前作で対応中）
打率を上げる   ← 上流プロセスの整備（本文書のテーマ）
```

POが「何を・どの順番で・どこまで作るか」を判断するには、前提となる情報——事業目的、現行業務の実態、目指すべき姿、ビジネス試算、関係者間の合意——が揃っている必要がある。前提が揃っていない施策にPOが打席を使うと、開発途中で「そもそもの前提が違った」が発覚し、打席そのものが無駄になる。

***

## 2. バリューチェーン全体像 — POの守備範囲と「その手前」

> **キーメッセージ：** ビジネスアイデアからリリースまでの全工程を可視化すると、POの守備範囲は中盤以降。前半の「施策の前提整理」は別の機能が担う必要があるが、現在そこが仕組み化されていない。PdMのWhat/Whyを更に分解すると「戦略判断」と「施策前提整理」は別の活動であることが見える。

### 全体像

```
   事業ニーズ        施策の前提整理         要求定義        開発〜リリース      検証
  ───────────    ─────────────────    ──────────    ───────────────    ──────
  「これをやりたい」    前提を揃える         PBIに分解する     コードを書いて届ける   効いたか確認する
  ニーズの発見       ↓                ↓            ↓              ↓
  ロードマップ判断    as-is / to-be整理    優先順位決定      スプリント実行       レビュー検証
                 ビジネス試算        受入条件定義     リリース           学習フィードバック
                 業務フロー調整       スコープ制御
                 SH間合意形成
                 既存業者・契約整理

  ─── PdM ───    ── 未定義（本文書） ──   ──── PO ────   ── PO + Dev ──    ── PO + PdM ──
  What / Why       前提を揃える        How / When    実行            学習ループ
```

現在の[PdM/PO分界](../../../../assets/context/projects/delivery-mgmt/_index.md)は「What/Why（PdM）vs How/When（PO）」として定義されている。この定義自体は正しい。

問題は、**PdMのWhat/Whyと、POのHow/Whenの間に「施策の前提整理」という工程が存在し、それが誰の守備範囲にも明示的に割り当てられていない**こと。

### PdMのWhat/Whyは2つの活動に分かれる

| レベル       | 活動       | 内容                                          | 頻度     |
| --------- | -------- | ------------------------------------------- | ------ |
| **戦略レベル** | ロードマップ判断 | 「何を作るべきか・なぜ作るか」の方向性を決める。プロダクトビジョン・顧客検証・優先方針 | 四半期〜年度 |
| **施策レベル** | 前提整理     | 個別施策の実現可能性を検証する。as-is/to-be・業務フロー・ビジネス試算・調整 | 施策ごと   |

PdMが戦略レベルで「ガス機器通報サービスをmTGに組み込む」とロードマップに載せても、**施策レベルの前提整理（現行業務の実態把握、既存業者との関係整理、ビジネス試算等）は別の活動として必要になる。** この施策レベルの前提整理が、PdMの仕事なのか、事業部門の仕事なのか、それとも別の誰かの仕事なのかが決まっていない。

結果として、前提が揃わないまま開発チームに渡るか、POが本来の守備範囲外の仕事を抱え込むことになる。

***

## 3. 別部署のサービスをmTGに組み込むとき、何が起きるか

> **キーメッセージ：** 別部署で開発しているサービスをmTGに組み込む場合、開発着手前に5つの前提整理が必要。これが揃わないまま進めると、Krakenで起きた構造と同じ問題が繰り返される。POがいても、前提が揃っていなければPOが判断する材料がない。

別部署で開発しているサービス（例：ガス機器の通報サービス）をmTGに組み込みたいという話が来たとする。この施策は「コードを書く」だけでは成立しない。開発の手前に、以下の前提整理が必要になる。

***

## 4. 上流プロセスの活動一覧 — 「POの手前」で必要なこと

> **キーメッセージ：** POの手前で必要な活動は漠然とした「企画」ではなく、具体的な6つの活動として定義できる。それぞれが揃っていない場合に何が起きるかを明確にすることで、「なぜ上流の整理が必要か」が伝わる。

### 施策の前提整理：6つの活動

| # | 活動 | 揃っていないとどうなるか |
|---|---|---|
| 1 | **as-is整理** | 現行との機能重複・仕様矛盾が開発中に発覚し、作り直しになる |
| 2 | **to-be定義** | ゴールが曖昧なまま進み、関係者ごとに期待がずれて手戻りが発生する |
| 3 | **ビジネス試算** | 投資判断根拠がないまま開発に入り、途中でROI疑問→凍結リスク |
| 4 | **業務フロー調整** | リリース直前にオペレーション不整合が発覚し、リリース延期になる |
| 5 | **既存契約・業者整理** | 二重運用コストが発生。契約の制約で移行できず計画が頓挫する |
| 6 | **SH間合意形成** | 開発途中で「聞いていない」→ やり直し。信頼毀損 |

この6つが揃った状態が「施策Ready」——POがPBIに分解して開発パイプラインに投入できる状態。

### なぜNaikistの施策はスムーズに回るのか

Naikistが安定しているのは、PO（福田さん）の力量だけが理由ではない。NaikistのメインワークであるWeb施策は、**上流の前提が比較的整理された状態でPOに届きやすい構造**にある。

|    | Naikist（Web施策） | クロスドメイン施策（例：通報サービス組込） |
| -------- | -------------- | --------------------- |
| 事業部門     | mTGチーム内で完結しやすい | 複数部署を横断               |
| 業務フロー    | 既存のWebフローの延長   | 新しいオペレーションの設計が必要      |
| 既存業者     | 関与少            | 契約・切替の調整が必要           |
| SH調整     | 少数             | 多数の関係者の合意が必要          |
| 上流の整理コスト | 低い             | 桁違いに高い                |

クロスドメインの施策が増えるほど、上流の整理コストは跳ね上がる。mTGが「総合エネルギー・暮らしサービスのプラットフォーム」に進化する過程では、こうしたクロスドメイン施策が確実に増える。上流プロセスの仕組み化は、今後の成長に不可欠な投資。

## 5. 誰が担うべきか — 施策の性質で使い分ける

> **キーメッセージ：** 上流プロセスの担い手には3つの選択肢があり、施策の性質で使い分ける。特別な新組織は不要。既存の役割の守備範囲を明確にし、施策ごとに「誰が上流を担うか」を決める仕組みを作る。

### 3つの選択肢

| 選択肢            | 担い手            | 強み                                    | 制約                                           | 適する施策                        |
| -------------- | -------------- | ------------------------------------- | -------------------------------------------- | ---------------------------- |
| **A. PdMが担う**  | PdMチーム         | プロダクト戦略との一貫性がある。ロードマップ判断とセットで前提も整理できる | PdMの稼働がボトルネックになりうる。戦略判断と施策前提整理の両方を抱えるのは負荷が高い | mTG内で完結する施策（Web改善、機能追加等）     |
| **B. 事業部門が担う** | 各事業部門          | ドメイン知識が最も深い。業務フロー・既存業者の実態を知っている       | 「開発チームに渡せる形」への翻訳力が不足しがち。整理の型がない              | ドメイン特化の業務改善（特定サービスの刷新等）      |
| **C. 兼務の企画担当** | 上流と下流を橋渡しできる人材 | 事業側の文脈と開発側の言語の両方がわかる。育成にもなる           | 専任ではないため稼働制約あり                               | クロスドメインの大型施策（通報サービス組込、EA連携等） |

### 推奨：組み合わせ + 施策ごとに明示的にアサインする

1つに統一する必要はない。施策の性質に応じてA・B・Cを使い分ける。重要なのは**施策ごとに「上流プロセスの担い手は誰か」を明示的に決める仕組み**を作ること。

現状の問題は「誰がやるか決まっていないまま、なんとなく進む」こと。PdMが気づけばやるし、誰も気づかなければ前提未整理のままPOに渡る。この曖昧さを排除するだけで、上流の品質は大きく変わる。

### 育成との両立

PO体制資料で「施策を1本任せる業務アサインがPO育成になる」と示したのと同じ構造がここにもある。上流プロセスの担い手として施策の前提整理を1本任せることは、そのまま**PdM・企画人材の育成**になる。上流の仕事を回すことで事業理解・SH調整力・判断力が身につく。追加の育成プログラムは不要で、業務そのものが育成機会になる。

## 6. 仕組み化 — 施策の「入口ゲート」を設定する

> **キーメッセージ：** KR②実行計画の「入口品質制御」をバリューチェーンレベルに拡張する。開発チーム向けのPBI Ready定義に加え、その手前に「施策Ready」を設定し、前提が揃った施策だけが開発パイプラインに入る仕組みを作る。

### 2段階のReady定義

```
施策Ready（本文書で定義）
  前提整理が完了し、POがPBIに分解できる状態
    ├─ as-is / to-be整理済み
    ├─ ビジネス試算あり
    ├─ 業務フロー確認済み
    ├─ 既存契約・業者整理済み
    ├─ SH間合意あり
    └─ 上流プロセスの担い手がアサイン済み
        ↓
PBI Ready（KR②実行計画で定義済み）
  仕様・受入条件が明確で、スプリントに投入できる状態
    ├─ 仕様・受入条件が明確
    ├─ PBIサイズが1スプリントで完了可能
    └─ 依存関係が解消済み
        ↓
開発・リリース
```

施策Readyを満たさない施策は「やらない」のではなく「上流プロセスに戻す」。KR②実行計画の方針④「依存関係が未整理の案件は、検討時期自体を調整する」と同じ考え方を、バリューチェーンレベルに拡張したもの。

### Q2の「共通基準整備」に組み込む

[FY26年度計画](../../../assets/context/projects/delivery-mgmt/dm-annual-plan-fy26.md)のQ2マイルストーン「共通基準の整備完了」に、以下を含める：

| 既存の計画（PBI Ready） | 追加する観点（施策Ready）        |
| ---------------- | ---------------------- |
| Ready定義の標準化      | 施策Readyチェックリストの整備      |
| PO判断の迅速化         | 上流プロセスの担い手アサインルール      |
| スコープ制御           | 施策Ready未達時のエスカレーションフロー |

施策Readyの導入は、PBI Readyと同じく段階的に進める。Q2でチェックリストを整備し、まずクロスドメイン施策から適用を始める。Web施策のように上流が軽い施策にまで一律適用する必要はない。
