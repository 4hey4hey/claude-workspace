# PO（プロダクトオーナー）機能の確保による開発スピードの最大化

## 成果への影響

現在の体制にはPO（プロダクトオーナー：開発の優先順位と仕様を決める役割）が不在。このまま開発を開始した場合、エンジニアが本来の開発業務に集中できず、期待する開発スピードに影響の可能性あり。

## 構造的な原因

PdMやサービスデザイナーは本業があり、PO業務を片手間で担うことは現実的ではない。開発チームとともに動く人が必要。現状、富田さんも仮説立案・検証の中核を担う存在であり、両立は難しいと考える。

PO不在でも、バックログ作成やスプリントの優先順位判断・調整といったPO業務そのものは消えない。担い手がいなければ、エンジニアが非公式にこれらを引き受けることになる。その結果、エンジニアの開発時間が削られる。さらに、権限・責任が曖昧なまま判断を行うため、意思決定が遅れ、手戻りも発生する。つまり、POを置かないことはエンジニアリソースの温存にはならず、開発もPO業務も中途半端になる状態を生む。

加えて、権限を正式に付与されていないエンジニアがPO業務を非公式に担うと、「誰が決めるのか」が曖昧なまま進むことになる。その結果、判断のお見合い（誰も決めず停滞）、開発とPO業務の往復によるコンテキストスイッチ、確認不足による二度手間・手戻りが常態化する。

## POがやるべき具体業務（＝POが不在時にエンジニアが対応する業務）

POがいない場合、以下の業務がエンジニアに非公式に流れ込む。いずれも「誰かがやらないと開発が進まない」業務であり、放置すれば開発が止まるため、エンジニアが善意で拾わざるを得なくなる。

**スプリント前（計画・準備）**
- バックログアイテムの洗い出し・起票（PdMや事業側の要望を聞き取り、チケット化する）
- 優先順位の判断（「次に何を作るかについて」を事業観点と技術観点の両方で作成する）
- ユーザーストーリー・受入条件の作成（「何ができたら完成か」の定義を自分で書く）
- リファインメントの準備・進行（見積もり対象のアイテムを揃え、仕様の曖昧な箇所を事前に潰す）
- やらないことを決める（ビジネス側・UIUX側からのオーダーの中でROIに見合わないものを提案し「なぜやらないほうがいいか」説明する）

**スプリント中（判断・調整）**
- 仕様の不明点が出るたびにPdM・事業側に確認に行く
  - 技術観点をステークホルダーにわかりやすく翻訳することが必要
- スコープの判断（「ここまで作るか、ここは切るか」を権限なく自己判断する）
  - 技術的・UIUX・ビジネスの判定が必要。
- 割り込み要望のトリアージ（「急ぎで対応して」に対して受ける・断るの判断）
- ステークホルダーからの問い合わせ対応（「進捗どうなってる？」「あの機能いつ出る？」）

**スプリント後（検査・調整）**
- スプリントレビューの準備・デモ（誰に何を見せるかの設計含む）
- フィードバックの取りまとめと次スプリントへの反映判断

**常時発生**
- バックログの整理・メンテナンス（古いチケットの棚卸し、重複の統合、粒度の調整）
- 関係者間の認識合わせ（PdM・デザイナー・事業側の期待値がズレていないか確認）
- ドキュメント整備（仕様書・PRDが不在のまま進む場合、エンジニアが実装しながら仕様を起こす）

これらの業務は1つ1つは小さく見えるが、合算するとエンジニア1人あたり日に1〜2時間、スプリントプランニング前後はさらに集中的に発生。しかも本来の開発業務と並行でコンテキストスイッチが頻発するため、実際の生産性低下は工数以上に大きくなる。

## PO不在コストの試算

仮にエンジニア3名が各20%の工数をバックログ整理・仕様確認・優先順位の相談に費やした場合、月あたり約0.6人月が非公式PO業務に消える。さらに、権限が曖昧なまま判断した仕様の手戻りが月1〜2件発生すると想定すると、手戻り対応でさらに工数が加算される。これらは開発スプリントの計画に現れない隠れコスト。

## このまま進めた場合の見通し

**放置コスト**: エンジニアがPO業務を非公式に吸収し、開発工数の一部がバックログ整理や仕様確認に消える。PO機能がないまま進めても、エンジニアの生産性はすでに下がる。

**機会損失**: 開発初期にPOを明確に置けば、最初からバックログ整備と「やらないこと」の判断が機能し、立ち上がりのスピードが大きく変わる。後から置くほど、この初速の差は取り戻しにくくなる。また、ユーザーにとって価値の高い機能を早期に届ける機会も逃す。

**不可逆性**: 役割が曖昧なまま開発が進むと、優先順位の根拠がない仕様や不要な機能が積み上がる。これらの負債は後から巻き戻すコストが大きく、早期に解消するほど効果的。

## ご相談したいこと

森さん（エンジニア）をPO兼務で配置することについて相談したい。エンジニアとしての稼働は一部減るが、PO不在のままではエンジニア全体の生産性が落ちるため、チーム全体の開発スループットとしてはプラスに働くと考えている。森さんはプロダクトへの理解が深く、現メンバーの中で最も適任。

配置にあたっては、以下を整備した上で進めたい。

**兼務の範囲と比率**
- PO業務50%・開発50%を目安とし、スプリント単位で比率を明示管理する
- PO業務のうち「事業優先度の判断」はPdMがインプットする接点を設け、森さんは「開発プロセスを回す」役割に集中する

**権限とエスカレーション**
- PO権限の範囲（バックログ優先順位決定、受入条件定義、スコープ判断）を文書化し、組織として正式に委任する
- 大きなスコープ変更・仕様凍結が必要な場合のエスカレーション先をPdM/上位責任者として明示する

**立ち上げの進め方**
- 開発スプリント開始前にスプリント0（1〜2スプリント）を設け、バックログ整備・受入条件の初期定義・依存関係の洗い出しに投資する
- PO業務の判断根拠・優先順位基準を初期からドキュメント化し、属人化を防ぐ

## スクラムマスター長島さんとの分担

長島さんはSMになったばかりであり、スクラムの実践を学びながら進めるフェーズにある。そのため、PO業務のうち「プロセス運営・ファシリテーション寄り」の業務をSMが引き取ることで、森さん（PO）は判断・意思決定に集中でき、長島さんはSMとしての実践経験を積める構造にする。

### 分担の考え方

PO業務には「何を・なぜ作るか（判断）」と「どう回すか（プロセス）」の2つの性質がある。判断はPO、プロセスはSMが持つのが原則。

### 具体的な分担

| 業務 | PO（森さん） | SM（長島さん） |
|---|---|---|
| **スプリント前** | | |
| バックログの優先順位決定 | ◎ 最終判断 | ー |
| ユーザーストーリー・受入条件の作成 | ◎ 内容を書く | △ フォーマットや粒度のフィードバック |
| リファインメントの準備 | ◎ 対象アイテムの選定・仕様説明 | ◎ 会議のファシリテーション・時間管理 |
| やらないことの判断・説明 | ◎ 判断と根拠の提示 | △ ステークホルダーとの場のセッティング |
| **スプリント中** | | |
| 仕様の不明点の確認・判断 | ◎ PdM・事業側との確認・意思決定 | △ 確認が滞っている場合の障害除去 |
| スコープ判断 | ◎ 何を切るかの判断 | △ 判断が必要なタイミングの検知・エスカレーション |
| 割り込み要望のトリアージ | ◎ 受ける・断るの判断 | ◎ チームへの割り込みを防ぐ盾役 |
| ステークホルダーからの問い合わせ | ○ 内容に関する回答 | ◎ 窓口の一本化・問い合わせの交通整理 |
| **スプリント後** | | |
| スプリントレビュー | ◎ 何を見せるかの設計 | ◎ レビュー会のファシリテーション |
| フィードバックの反映判断 | ◎ 次スプリントへの取り込み判断 | △ フィードバックの記録・整理 |
| レトロスペクティブ | △ 参加・インプット | ◎ 進行・改善アクションの追跡 |
| **常時** | | |
| バックログの整理・棚卸し | ◎ 優先度の見直し・不要アイテムの削除 | △ 棚卸しのリマインド・定例化 |
| 関係者間の認識合わせ | ○ プロダクトの方向性に関する合意 | ◎ 会議設計・ズレの早期検知 |
| ドキュメント整備 | ◎ 仕様・判断根拠の記述 | △ テンプレート整備・記録の習慣づくり |

◎ = 主担当　○ = 関与　△ = 補助・サポート

### 長島さんがSMとして優先的に身につけるとよいスキル

SMとしての立ち上がりにあたって、以下を実践の中で習得していく想定。

1. **ファシリテーション** — リファインメント・レビュー・レトロの進行。まずは「時間内に終わらせる」「全員が発言できる」レベルから
2. **障害除去** — チームの開発を止めているものを検知して取り除く。仕様確認の滞留、環境の問題、割り込みの防御など
3. **可視化** — スプリントの進捗、ベロシティ、ブロッカーの状態をチームとステークホルダーに見える形にする
4. **チームの自律支援** — 自分が答えを出すのではなく、チームが自分たちで判断できるよう問いかけで導く

最初から完璧である必要はない。森さん（PO）と役割を明確に分けた上で、スプリントを回しながら学んでいく形で進める。

## 想定FAQ

### Q. スクラムマスター（長島さん）がPO業務を兼務すればいいのでは？

**A. SMとPOは利益相反の関係にあり、兼務すると両方が機能不全になる。**

POの仕事は「もっと作れ、これを優先しろ」（プロダクト価値の最大化）。SMの仕事は「チームを守れ、無理させるな」（プロセスの健全性とチームの持続可能性）。同一人物が担うと、自分で積んだ仕事を自分で「無理です」と言う構造になり、どちらかが必ず犠牲になる。

| 場面 | PO判断 | SM判断 | 兼務すると |
|---|---|---|---|
| バックログが溢れている | もっと詰め込みたい | キャパ超過、削るべき | 自分で自分にブレーキをかけられない |
| 割り込み要望が来た | 事業的に受けたい | スプリントを守りたい | 受ける側の立場が勝ちがち |
| 見積もりが甘い | スコープを維持したい | リスクを可視化して調整したい | 都合の悪い情報を握りつぶせてしまう |
| レトロで「仕様が曖昧」と指摘 | （自分が原因） | 改善を促す | 自分の問題を自分で追及できない |

加えて、長島さんはSMになったばかりで、ファシリテーション・障害除去・チームの自律支援といった基本動作を実践の中で習得していく段階。ここにPO業務まで乗せると、どちらの役割も中途半端になる。SM単体でも立ち上げ負荷は十分に大きく、まずSMとして「チームを回せる」状態を作ることが先決。

### Q. POに専任を置くほどの規模か？外部から採用すべきでは？

**A. まず森さん兼務で立ち上げ、1〜2スプリント後に専任化の要否を判断する。**

現段階では開発チームの規模・プロダクトの複雑性がまだ見えきっていない。最初から専任POを外部採用するのはコストとオンボーディング期間の両面でリスクがある。森さんはプロダクトへの理解が深く、チームとの信頼関係もあるため、立ち上げフェーズでは最も速く機能する。兼務比率と成果指標を追跡し、PO業務が50%を超える状態が続くようであれば専任化を検討する。

### Q. PdMがPO業務を見ればいいのでは？

**A. PdMとPOは役割が異なる。PdMは「何を作るべきか」を考え、POは「どの順番で作るか」を決めて開発を回す。**

PdMは仮説立案・顧客検証・ロードマップ策定が本業であり、スプリント単位のバックログ管理・受入条件定義・日々の仕様判断を並行することは現実的でない。PdMが戦略とビジョンをインプットし、POがそれを開発チームの動きに翻訳する——この分業が機能してはじめて、PdMもPOもそれぞれの役割に集中できる。

### Q. エンジニアの開発時間が減るのでは？（森さんの稼働減少）

**A. 森さん1人の開発稼働は減るが、チーム全体の開発スループットは上がる。**

PO不在のまま進めた場合、エンジニア全員が非公式にPO業務を引き受け、1人あたり日1〜2時間が仕様確認・優先順位の相談・判断待ちに消える。森さんが明示的にPO業務を担うことで、他のエンジニアはその分を開発に集中できる。1人の稼働を半分POに割くことで、残り全員の生産性が上がる構造。

### Q. 兼務がうまくいかなかったらどうする？

**A. 撤退基準を事前に決めておく。**

以下のいずれかに該当した場合、体制を見直す：
- 森さんのPO業務比率がスプリント2回連続で70%を超える（開発に戻れない状態）
- バックログ消化率がスプリント3回連続で目標の70%を下回る
- チームから「PO判断がボトルネックになっている」というフィードバックがレトロで2回以上出る

これらが出た場合は、専任PO配置・外部採用・体制変更を具体的に検討する。
