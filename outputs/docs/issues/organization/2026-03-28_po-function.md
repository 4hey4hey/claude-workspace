# DiscoveryチームPO配置と中長期のPO確保方針

> **これは何？**
> DiscoveryチームのPO不在課題への短期対応（森さん兼務）と、中長期でPO人材をどう確保・育成していくかの方針をまとめたページ。西村GM・及川GMへの説明材料。

**3行サマリー**

* **短期（セクション1-2）：** PO不在で月0.3〜0.6人月が隠れコストとして流出中。森さんの兼務が立ち上がり最速の解

* **中長期（セクション3）：** 森さんのPO化はキャリアとしても合理的。ただし不可逆性が高いため、評価プロセスを設けて計画的に判断する

* **進め方（セクション4）：** 評価タイミングを3段階で設定し、6ヶ月以内に転向 or エンジニア復帰の結論を出す

***

## 1. ディスカバリーチームPO不在の課題

DiscoveryチームにはPOが不在。PO業務（詳細な要件定義、優先順位決定、仕様判断、スコープ管理）は消えないため、エンジニアが非公式に吸収することが想定される。

エンジニア3名が各10〜20%の工数をバックログ整理・仕様確認・優先順位の相談に費やしたと考えた場合、**月あたり約0.3〜0.6人月**が非公式PO業務に消える。さらに、権限が曖昧なまま判断した仕様の手戻りが月1〜2件発生すると想定すると、手戻り対応でさらに工数が加算されるリスクあり。

POを置かないことはエンジニアリソースの温存にはならない。開発もPO業務も中途半端になる状態を生む。

| PO不在で発生する問題       | 結果                     |
| ----------------- | ---------------------- |
| エンジニアがPO業務を非公式に吸収 | 開発時間が削られる。1人あたり日1〜2時間  |
| 権限・責任が曖昧なまま判断     | 意思決定の遅れ・手戻り            |
| 「誰が決めるのか」が不明確     | 判断のお見合い・コンテキストスイッチの常態化 |

### 放置した場合のリスク

**放置コスト：** エンジニアの生産性はすでに下がっている。PO機能がないまま進めても、隠れコストが積み上がるだけ。

**機会損失：** 開発初期にPOを置けば、最初からバックログ整備と「やらないこと」の判断が機能し、立ち上がりのスピードが大きく変わる。後から置くほど、初速の差は取り戻しにくい。

**不可逆性：** 役割が曖昧なまま開発が進むと、優先順位の根拠がない仕様や不要な機能が積み上がる。この負債は後から巻き戻すコストが大きい。

## 2. 森さんをPO兼務として解決したい

森さんよりPO兼務ができないか申し出あり、チームのケイパビリティと立ち上がり速度を踏まえると、**森さんのPO兼務が最も現実的かつ最速の解**。

| 調達方法      | 立ち上がり  | 強み                                       | リスク                                  |
| --------- | ------ | ---------------------------------------- | ------------------------------------ |
| **森さん兼務** | **即時** | プロダクト理解が深い。チームとの信頼関係がある。本人もPO/PdMへの関心がある | エンジニア稼働の一部減少                         |
| 外部採用      | 3〜6ヶ月  | PO経験者を即戦力で獲得できる可能性                       | オンボーディング期間。カルチャーフィット。採用コスト           |
| PdM兼務     | ー      | ー                                        | PdMは仮説立案・顧客検証が本業。スプリント単位の管理との両立は非現実的 |
| SM兼務      | ー      | ー                                        | SMとPOは利益相反。兼務すると両方が機能不全になる           |

森さん1人の開発稼働は減るが、**他のエンジニア全員がPO業務から解放される**ため、チーム全体の開発スループットはプラスに働く。

## 3. 森さんの今後について（中長期）：トライアルで見極める

森さんは、現状のエンジニアPOへの転向に関心を持っており、今回のPO兼務は本人も希望している。ただし、このタイミングでPOとしての素養があるのか、本当に転向するのかはすぐに判断できない。一度POに転向してしまうと、スキルセット・キャリアパスの両面で**エンジニアには戻りにくい**。森さん本人のキャリアにとっても、見極めなしの転向は本人とっても組織にとってもリスクになる。

中長期目線では、DiscoveryチームでのPO兼務を**トライアル**として位置づけ、一定期間でPO転向の見極めを実施したい。

### トライアルの評価体制

| 評価観点                         | 評価者   | 確認ポイント                                       |
| ---------------------------- | ----- | -------------------------------------------- |
| **プロダクトマネジメント・プロジェクトマネジメント** | 杉崎    | バックログ優先順位の判断力、ステークホルダーとの合意形成、仕様の定義精度         |
| **テクノロジーマネジメント**             | 中嶋・中島 | 技術的判断の妥当性、アーキテクチャ理解に基づくスコープ判断、エンジニアチームとの連携品質 |

## 4. PO転向見極めに向けてのステップ：４半期ごとに評価・3Q時点で最終結論を出す

評価タイミングを3段階で設定し、計画的に判断し及川さん・西村さんに報告します。

### 評価タイミング（仮案）

| 時期            | 判断内容                   | 見るもの                                   |
| ------------- | ---------------------- | -------------------------------------- |
| **Q1末**（約3ヶ月） | 兼務が回っているか。PO適性の初期判断    | PO業務の負荷比率、チームの体感、バックログ運営の自走度、森さん本人の手応え |
| **Q2末**（約6ヶ月） | PO適性の中間判断              | SHとの合意形成、エンジニアからの信頼、スコープ判断の質           |
| **Q3末**（約9ヶ月） | **転向 or エンジニア復帰の最終判断** | PO業務の比率推移、チームのスループット変化、本人の意思           |

* 評価は杉崎・中嶋・中島で行い**Q3末に結論を出す。**

* うまくいけば「こういう条件でエンジニア→PO移行が成立する」という実績になる。うまくいかなければ「ここが足りなかった」がわかる。いずれにせよ次に同じ判断をするときの精度が上がる

### 評価方針

ロールは変更しない。今期の森さんの個人目標にPO業務の評価項目（プロダクトマネジメント・プロジェクトマネジメント）を追加設定する。

評価の実務は杉崎（プロダクトマネジメント・プロジェクトマネジメント）・中嶋/中島（テクノロジーマネジメント）が担い、四半期ごとに西村GMへ報告する。Q3末の最終判断の時点で、正式なロール変更の要否を判断する。

## 想定FAQ

### Q. SM（長島さん）がPO業務を兼務すればいいのでは？

**SMとPOは利益相反の関係にあり、兼務すると両方が機能不全になる。**

POは「もっと作れ、これを優先しろ」（プロダクト価値の最大化）。SMは「チームを守れ、無理させるな」（プロセスの健全性）。同一人物が担うと、自分で積んだ仕事を自分で「無理です」と言う構造になる。

| 場面             | PO判断       | SM判断           | 兼務すると             |
| -------------- | ---------- | -------------- | ----------------- |
| バックログが溢れている    | もっと詰め込みたい  | キャパ超過、削るべき     | 自分で自分にブレーキをかけられない |
| 割り込み要望が来た      | 事業的に受けたい   | スプリントを守りたい     | 受ける側の立場が勝ちがち      |
| 見積もりが甘い        | スコープを維持したい | リスクを可視化して調整したい | 都合の悪い情報を握りつぶせてしまう |
| レトロで「仕様が曖昧」と指摘 | （自分が原因）    | 改善を促す          | 自分の問題を自分で追及できない   |

長島さんはSMの基本動作を実践の中で習得していく段階。ここにPO業務まで乗せると、どちらの役割も中途半端になる。

### Q. POに専任を置くほどの規模か？

**まず森さん兼務で立ち上げ、評価ステップに沿って判断する。**

開発チームの規模・プロダクトの複雑性がまだ見えきっていない段階で、最初から専任POを外部採用するのはコストとオンボーディング期間の両面でリスクがある。兼務比率と成果指標を追跡し、PO業務が50%を超える状態が続くようであれば専任化を検討する。

### Q. PdMがPO業務を見ればいいのでは？

**PdMとPOは役割が異なる。PdMは「何を作るべきか」を考え、POは「どの順番で作るか」を決めて開発を回す。**

PdMは仮説立案・顧客検証・ロードマップ策定が本業であり、スプリント単位のバックログ管理・受入条件定義・日々の仕様判断を並行することは現実的でない。PdMが戦略とビジョンをインプットし、POがそれを開発チームの動きに翻訳する分業が前提。

### Q. エンジニアの開発時間が減るのでは？

**森さん1人の開発稼働は減るが、チーム全体の開発スループットは上がる。**

PO不在のまま進めた場合、エンジニア全員が非公式にPO業務を引き受け、1人あたり日1〜2時間が仕様確認・優先順位の相談・判断待ちに消える。1人の稼働を半分POに割くことで、残り全員の生産性が上がる構造。

### Q. 転向が前提なのか？

**転向は前提ではない。評価ステップに沿って見極める。**

森さん本人の希望はあるが、Q2末（約6ヶ月後）の最終判断まではPO兼務の状態。プロダクトマネジメント（杉崎）・テクノロジーマネジメント（中嶋・中島）の両観点で評価し、本人の意思確認と合わせて判断する。
