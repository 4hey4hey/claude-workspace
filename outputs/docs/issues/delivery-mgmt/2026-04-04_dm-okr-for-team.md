# デリバリーマネジメントチーム — Mission・OKR概念書

## これは何

チームの存在意義・行動原則・OKRの核をチームメンバー向けに伝えるドキュメント。

* Mission：作る力を、届く価値に変える。ムダなく・ムリなく・ムラなく流れを整える

* 日々の判断基準は「守る・速くする・強くする・見極める」の4原則

* SM・PO・DMの3ロールが原則を分担する。SMはプロセス面、POは投資面、DMは人面

* 5つのKRは連動しており、入口品質→フロー効率→学習ループの順に回す

***

## このチームは何のために存在するか

### Mission

> **作る力を、届く価値に変える。**

作る力が価値まで流れ切るように、流れを整える。
ムダなく・ムリなく・ムラなく。

開発組織は、作る力を持っている。しかしその力は、価値に届いて初めて意味を持つ。リリースされても使われない機能、仕様の曖昧さで生まれる手戻り、人への過負荷、チーム間のばらつき——こうしたものが間に挟まると、作る力は価値に変換されないまま終わる。
デリバリーマネジメントチームは、作る力と価値の間をつなぐ。作ったものが顧客価値・事業価値・社会価値として届き切る状態を、流れを整えることでつくり出す。

***

### 何を整えるのか — 3つの流れ

作る力は、そのままでは価値に届かない。途中で流れが滞ると、せっかくの力が価値になる前に失われる。私たちが整えるのは、この3つの流れだ。

|      | 意味                           |
| ---------- | ---------------------------- |
| **ムダなく流す** | 手戻り・不要な工程・使われない機能で流れを止めない    |
| **ムリなく流す** | 人に過負荷をかけず、持続可能なペースで流す        |
| **ムラなく流す** | チーム間でプロセス・品質のばらつきをなくし、安定して流す |

***

### どう動くのか — 4つの行動原則

日々の仕事で迷ったら、この4つのどれに当たるかで判断する。
**どれにも当てはまらないなら、デリバリーチームの仕事ではない。** 逆に言えば、この4つに当たる行為はロールを問わず全員の仕事だ。

| 原則       | 意味           | 効く流れ  | 主に動く  | 具体的な場面 |
| -------- | ------------ | ----- | ----- | ------ |
| **守る**   | フロー・人の健全性を守る | ムダ／ムリ | SM（チーム内）・DM（横断＋人） | **SM:** チーム内のブロッカー検知・除去、フロー阻害の早期検知 / **DM:** チーム横断でムダ（手戻り・曖昧仕様）・ムリ（過負荷）を観測→SM/POにFB + 1on1・コンディション管理・負荷分散 |
| **速くする** | フロー効率の最大化    | ムダ    | SM＋PO＋DM | **SM+PO:** チーム内のボトルネック解消、PBI小分割、手戻り削減 / **DM:** 横断視点でフロー改善を推進、SMの打ち手をレビュー |
| **強くする** | プロセスと人を強くする  | ムラ    | SM（チーム内）・DM（横断＋人） | **SM:** チーム内のふりかえり・プロセス改善 / **DM:** 横断でプロセス標準化・型化・展開 + リーダー育成・SM育成・採用・オンボーディング |
| **見極める** | 投資判断・価値の選択   | ムダ    | PO    | 優先順位判断、やらないことの決定、受入条件定義、スコープ判断、レビューでの価値検証 |

※「主に動く」ロールはリード役。原則そのものは全員の共通言語。SMとDMの分担は「プロセス vs 人」ではなく「チーム内 vs チーム横断」。DMはSMのシニア版として、プロセスも人も両方を横断的に見る。

***

### 誰が担うのか — 3つのロール

4つの原則は全員の共通言語だが、日々の動きの中で誰がリードするかは分かれている。このチームは3つのロールで構成される。

| ロール | 一言で言うと | 守備範囲 | 主にリードする原則 |
| --- | --- | --- | --- |
| **SM**（スクラムマスター） | チーム内の流れの番人 | 担当チーム内のフロー障害除去・スクラム運営・プロセス改善 | 守る（チーム内）・速くする・強くする（チーム内） |
| **PO**（プロダクトオーナー） | 投資の番人 | バックログの優先順位・受入条件・スコープ判断。流す価値そのものを見極める | 見極める・速くする |
| **DM**（デリバリーマネージャー） | SMのシニア版＋ピープルマネジメント | 複数チームを横断的に俯瞰し、フロー・プロセス・人の全体を観測。SMがチーム内で回すことを横断で標準化・型化し、SM自身の成長も支援。同時にメンバーのコンディションと成長を見て、流し続けられる組織を保つ | 守る（横断＋人）・速くする（横断）・強くする（横断＋人） |

ロールは分かれているが、原則は共通。自分のロールでなくても、原則に当たる場面では全員が動く。

**SMとDMの境界：** 「プロセス vs 人」ではなく「チーム内 vs チーム横断」。DMはSMのシニア版として、プロセスも人も両方を横断的に見る。SMがチーム内で回すことを、DMが横断で標準化・型化し、SM自身の育成も担う。

> **現時点の体制（2026年4月）:** PO: 福田さん（Naikist/Web）、伊藤さん（Fluppuccino PO候補・育成中）、室山さん（Guardians PO候補）、森さん（Discovery 兼務）。SM: 中道さん（Fluppuccino）、長島さん（Discovery/Guardians・新任）、堤さん（Naikist SM補佐）。DM: **採用中**（入社までは杉崎が兼務）。

***

### 誰にとって、何を意味するか

同じMissionでも、読む人によって見え方は変わる。経営にはこう見え、チームにはこう感じられる。

**経営にとって：** 同じ予算・体制で、施策をもっと速く・多く回せるようにすること。

**チームにとって：** 作る力が価値まで流れ切るよう、プロセスを守り速くし強くし、人を守り育て、投資を見極めること。

***

## 杉崎のOKR

### Objective

> **正しいものを、速く、持続的に届けるデリバリーラインを確立する**

「速く作る」だけではない。入口（何を作るか）から出口（届けた結果から学ぶ）まで、一気通貫で回す状態を作る。

***

### 5つのKR

| KR        | 名前    | 何を目指すか                       | 対応する行動原則 | 主な担い手 | 効く流れ |
| --------- | ----- | ---------------------------- | -------- | ----- | ---- |
| **KR1**   | 入口品質  | 開発ラインに流す仕事の質を上げ、仕様起因の手戻りをなくす | 見極める     | PO | ムダ   |
| **KR2**   | フロー効率 | 着手からリリースまでの時間を短縮する           | 速くする     | SM + PO | ムダ   |
| **KR3**   | 学習ループ | 届けた結果から学び、次の投資判断に活かすサイクルを作る  | 見極める     | PO + SM | ムダ   |
| **基盤KR1** | 健全性   | チームが持続可能なペースで走れる状態を守る        | 守る       | DM（人面）+ SM（プロセス面） | ムリ   |
| **基盤KR2** | 強化    | プロセスを標準化し、次世代リーダーを育てる        | 強くする     | SM（プロセス面）+ DM（人面） | ムラ   |

***

### 3つの挑戦KRは連動している

```
KR1（入口品質）→ KR2（フロー効率）→ KR3（学習ループ）
      ↑                                      │
      └──────────── 学びが仮説精度を高める ────┘
```

* KR1が改善 → 手戻りが減り、フロー（KR2）が速くなる

* KR2が改善 → 速く届く → 検証頻度が上がり、学習（KR3）が回る

* KR3が回る → 学びが仮説精度を高め、入口品質（KR1）がさらによくなる

**どれか1つだけやっても効果は限定的。3つが連動して初めてMissionが達成される。** 基盤KR1・KR2は、この3つすべての前提条件。

挑戦KR3つはムダ対策に集中させ、ムリは基盤KR1、ムラは基盤KR2でカバーする。3つの流れ全体を取りに行く構造だ。

***

### PO・SM・DMを貫くひとつの原則

> **価値を小さく分割し、比較可能な単位で届け、仮説検証のスピードを上げる**

| 実践            | やること                         | 効くKR    |
| ------------- | ---------------------------- | ------- |
| 価値の垂直スライス     | 技術レイヤーではなく、ユーザー価値の単位で分割する    | KR2・KR3 |
| 比較可能な単位で出す    | A案/B案を同時に出して定量比較できるサイズにする    | KR3→KR1 |
| 仮説を明示してから着手する | PBIに「検証したい仮説」を1行書いてスプリントに入れる | KR3・KR1 |
| 「やらない」を先に決める  | 分割時に今回やらないことを明示的にリストアップする    | KR1・KR2 |

## Appendix: Mission → OKR の導出過程

Missionを「問いの形」でロジックツリーに分解し、KRを導出した。

```
作る力を価値に変えるために、デリバリーチームは何を実現すべきか？
│
├─ 開発ラインに流す仕事の質と判断スピードをどう高め、
│   仕様起因の手戻りと無駄な開発をなくせるか？
│   （→ KR1: 入口品質 / 見極める / ムダ）
│   ├─ バックログを常にReadyに保ち、曖昧な仕様のまま着手させないためには？
│   ├─ 仕様起因の手戻りをどう根絶するか？
│   ├─ PO判断がフローを止めないようにするには？
│   ├─ 関連プロジェクトの依存関係や影響度合いの透明性をどう上げるか？
│   └─ スコープ膨張をどう防ぎ、やらない判断をどう徹底するか？
│
├─ 顧客に届くスピードをどう上げるか？
│   （→ KR2: フロー効率 / 速くする / ムダ）
│   ├─ サイクルタイムをどう短縮するか？
│   ├─ ボトルネックを可視化し渋滞をどう解消するか？
│   └─ スプリントゴールを確実に達成するには？
│
├─ 届けた価値を検証し、学びを次のサイクルにどう反映し続けるか？
│   （→ KR3: 学習ループ / 見極める / ムダ）
│   ├─ スプリントレビューで価値仮説を確実に検証するには？
│   ├─ 検証結果をPdMにフィードバックし、次の投資判断に反映するには？
│   └─ 「作った→検証→学んだ→次の優先順位に反映」のサイクルをどう定着させるか？
│
└─ チームが走り続けられる力をどう担保するか？
    （→ 基盤KR1: 健全性 / 守る / ムリ、基盤KR2: 強化 / 強くする / ムラ）
    ├─ メンバーが健全なペースで働けているか？
    ├─ プロセスのばらつきをどう減らすか？
    └─ 次世代リーダーをどう育てるか？
```

**KR間の因果接続:** KR1（入口）→ KR2（フロー）→ KR3（出口・学習）で開発バリューチェーンの全区間をカバーする。KR3の学習結果がKR1の入口品質をさらに改善する循環構造。
