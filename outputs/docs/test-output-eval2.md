# PdMとエンジニアの衝突が納期予測精度に与える影響

## 経営課題：納期遅延の構造的原因

現在、プロダクト開発において「毎回同じ衝突が繰り返される」という問題が、プロジェクトの納期予測精度と実績を大きく損なっています。この問題は個別の人間関係ではなく、**組織レベルのプロセス欠陥**です。

## 目下の状況

### PdM側の行動パターン
- 詳細な要件定義や技術的可行性の検証なしに、機能要件を提示
- 実装期間に対する現実的な理解が不足
- 「これ作って」という指示形で進める

### エンジニア側の反応
- 技術的制約や実装の複雑性を理由に対案を提示できていない
- 詳細な要件定義がないまま正確な工数見積もりが困難
- 要件定義の質問に応じてもらえないと感じ、責任を一方的に負わされていると感受

### 結果としての影響
- **納期予測精度の低下**：開発途中での仕様変更が頻発し、スケジュール遅延が常態化
- **プロジェクト停止リスク**：意思決定が遅延し、開発スケジュール全体が圧迫
- **チームモチベーション低下**：同じ衝突が繰り返されることで、双方の信頼関係が悪化

## 根本原因分析

この問題には以下の構造的要因があります：

### 1. プロセスの不備
- 要件定義から開発着手までの段階で、PdMとエンジニアの協働が欠落している
- 技術的可行性検証が組み込まれていない
- 結果：開発着手後の追加検討が必須となり、スケジュール压迫の原因となる

### 2. 意思決定フレームワークの欠如
- 「できる/できない」を判断する基準が曖昧
- 代替案検討のプロセスが体系化されていない
- 結果：同じ議論が繰り返される

### 3. 相互理解の不足
- PdM：エンジニアが直面する技術的制約や工数の複雑性を十分に理解していない
- エンジニア：ビジネス要件や優先順位の背景にある経営判断を十分に理解していない

## 納期予測精度への具体的な影響

- **予測精度の喪失**：毎回の衝突と仕様変更により、納期予測が信頼性を失う
- **バッファの浪費**：設計されたスケジュール・バッファが、対立の解消に消費される
- **リリース遅延の頻発**：不確実性が高まり、納期遅延が組織の常態化リスク

## 改善施策（優先度順）

### 段階1：即時対応 - プロセスの構築（1-2週間）
- **合同要件検討会議の設定**：PdMとエンジニアが合同で実装可能性を検討する正式プロセスを確立
- **技術可行性調査フェーズの組み込み**：大規模機能については、事前に技術検証の時間を確保
- **詳細要件書テンプレートの導入**：機能仕様、制約条件、優先度を明確に文書化し、曖昧性を排除

### 段階2：短期改善 - コミュニケーション体制の強化（2-4週間）
- **定期的なPdM-エンジニア間の1on1ミーティング**：信頼関係構築と懸念事項の共有
- **代替案ベース提案の文化構築**：「できない」で終わらず、「この条件であればできる」という提案スタイルへの転換
- **共通の成功指標の設定**：「納期を守る」という共通目標を明確化

### 段階3：体系的な改善 - 意思決定フレームワークの導入（1ヶ月）
- **優先度マトリックス**：ビジネス価値と実装コスト（工数）を可視化し、優先順位判断を定量化
- **技術リスク評価プロセス**：新規技術や複雑な実装については、事前リスク評価を実施
- **進捗・精度レビュー**：毎月の進捗と見積もり実績をもとに、予測精度を継続改善

## 期待効果

### 直接的な改善
- **納期予測精度の向上**：正確な工数見積もりと現実的なスケジュール設定が可能に
- **プロジェクト停止時間の削減**：意思決定が迅速化し、開発の連続性が向上

### 間接的な改善
- **チーム間の信頼構築**：対立から協働へのマインドセット転換
- **品質と納期のバランス改善**：十分な検討に基づく実装により、手戻り削減
- **繰り返しの無駄な議論の削減**：プロセス化により、同じ議論が生じない仕組み

## 結論

納期遵守と予測可能なリリース スケジュールを実現するためには、PdMとエンジニアの衝突を「個人の問題」ではなく「プロセスの問題」として捉え、組織的に対応することが必須です。上記施策により、**3ヶ月以内に納期予測精度を20%向上させる**ことが期待できます。
