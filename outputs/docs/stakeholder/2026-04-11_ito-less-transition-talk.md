# はねりさんへの説明：LeSS移行とPO体制

> **これは何？**
> ナイキスト・フラップチーム統合およびPO体制変更にあたり、はねりさん（伊藤さん）へ説明する際のストーリーラインと補足資料。

**3行サマリー**

* LeSS／フィーチャーチーム体制に移行したい。3+3人の今がまとめやすいタイミング

* 今の体制は一時的な準備段階。人が増えたらチームを再分割する前提

* はねりさんには将来エリアPOとして自分のチームを持ち、最終的にはプロダクト全体を見渡せるPOになってほしい

***

## ストーリーライン

```
目指す方向（LeSS） → 理想の姿 → 今はその準備段階 → あなたにはこうなってほしい
```

***

### Step 1. やりたいこと — LeSSへの移行

> 開発チームをLeSS（大規模スクラム）のフィーチャーチーム体制に移行していきたいと思っている。
> 1つのプロダクトバックログから、どのチームでもどの領域でも取れる状態を目指している。
>
> エンジニアが3人＋3人になっているタイミングで、チーム規模としてもまとめやすい。
> メンバー同士の関係も良好だし、今がちょうどいいタイミングだと思っている。フィーチャーチームを体験する。

**ここで伝えたいこと：**

* 「体制を変える」ではなく「目指す方向がある」という話から入る

* はねりさん個人の話ではなく、チーム全体の方向性であることを示す

***

### Step 2. 理想像 — こういう姿を目指している

> 目指しているのはこういう姿。

**LeSS体制のイメージ：**

```
┌─────────────────────────────────────┐
│         プロダクトバックログ（1つ）        │
├─────────────────────────────────────┤
│                                     │
│            リードPO（福田さん）          │
│      クラーケン・Web・運用を横断して統括     │
│                                     │
│    ┌──────────┐    ┌──────────┐      │
│    │ エリアPO A │    │ エリアPO B │      │
│    │ チーム 5名  │    │ チーム 5名  │      │
│    └──────────┘    └──────────┘      │
│                                     │
└─────────────────────────────────────┘
```

> 1つのプロダクトバックログがあって、リードPOが全体の優先順位を見る。
> エリアPOがそれぞれのチームを持って、自律的に運営していく形。
> チームはどちらもフィーチャーチーム。モバイルだけ、Webだけではなく、機能横断で動ける。

**ここで伝えたいこと：**

* 視覚的にゴールの絵を見せることで、この後の話が「ここに向かうための変更」として自然に入る

* 「エリアPO」というポジションが存在することを先に見せておく（Step 4への伏線）

***

### Step 3. 現在地 — 今は準備フェーズ

> ただ、今はまだ人数が少ないので、いきなりこの形にはできない。
> まず1チームにまとめて、フィーチャーチームの文化を全員で体験するところから始める。

**具体的に何が変わるか：**

> ナイキストとフラップを1つのスクラムチームに統合する。
> 福田さんにリードPOとして全体を見てもらう。実質的にすでにその動きをしていたので、正式化する形。
> 村山さんもはねりさんと同じレイヤーに配置される。はねりさんだけが何か変わるわけではない。

| 項目           | 運営方針                               |
| ------------ | ---------------------------------- |
| **バックログ**    | 統合して1つに。モバイル・Webをタグで分けて管理          |
| **全体の優先順位**  | 福田さんが責任を持つ。習熟度に応じて権限を移譲            |
| **SH窓口**     | メインは福田さん。手が届かない部分をはねりさんが担当         |
| **スプリントゴール** | 最終的に1つを目指すが、過渡期は既存コミットメントを優先し段階的に  |
| **はねりさんの担当** | モバイル領域の要件詳細化とスコープコントロール。現場判断ははねりさん |

> これは一時的な構造。人が増えて10名規模になったら、5名×2チームに分割する前提でいる。
> 急に全部変えるのではなく、段階的に進めていく。
> 役割分担は福田さんも含めてこれから話し合いながら決めるので、はねりさんの意見も聞かせてほしい。

**ここで伝えたいこと：**

* Step 2の理想像との差分として「今はここ」を示す。「降格」ではなく「準備段階」

* 「一時的な構造」「段階的に」を繰り返し使うことで、固定ではないことを印象づける

* 村山さんも同列であることを明示し、「自分だけが下がる」という印象を払拭する

***

### Step 4. はねりさんへの期待 — エリアPO、その先へ

> はねりさんには、将来的にエリアPOとして自分のチームを持ってほしいと思っている。
> その先には、プロダクト全体を見渡せるPOになってほしい。

| フェーズ            | やること                                            | 目指す状態              |
| --------------- | ----------------------------------------------- | ------------------ |
| **Phase 1**（現在） | モバイル領域を担当。スクラムの理解を深める。メンバーとの信頼関係を構築。余力があればWeb側も | モバイル領域で自律的に回せる     |
| **Phase 2**     | Web領域にもスキルを拡大。福田さんから優先順位の考え方を学ぶ                 | 複数領域を横断して判断できる     |
| **Phase 3**     | エリアPOとして自分のチームを持つ。優先順位も自分で決める                   | 「何も言わなくても回る」状態     |
| **その先**         | プロダクト全体を見渡す。複数エリアの優先順位を横断的に判断                   | プロダクト戦略レベルの意思決定に参画 |

> 今はPhase 1。まずモバイル領域をしっかり固めることが一番大事。
> 習熟度に応じて、福田さんから徐々に権限を移譲していく。
>
> もちろん一人で全部やれという話ではない。
> 福田さんからフィードバックをもらえるし、疑問や納得いかないことがあれば岩田さんにぶつけてほしい。
> 自分も定期的に状況を確認するし、困ったことがあればいつでも相談してほしい。

**ここで伝えたいこと：**

* 「期待している」というポジティブなメッセージで締めに向かう

* 具体的なフェーズを見せることで、「先がある」ことを実感させる

* サポート体制もここで伝え、孤立感を与えない

***

> この体制変更は、そのためのステップ。
>

## 説明時の注意点

### やるべきこと

* 一時的な構造であることを **繰り返し強調** する

* 将来的なエリアPO復帰の **期待感を明確に** 伝える

* 村山さんも同列に配置されることを伝え、**はねりさんだけが下がるわけではない** ことを示す

* 疑問や不明点は **岩田さんにぶつけてほしい** と伝える

* 「率直にどう感じた？」と感情に寄り添う問いかけをする。沈黙があっても焦らず待つ

### 避けるべきこと

* 「降格」「格下げ」と受け取られる表現

* 福田さんとの能力差を直接的に言及すること

* はねりさんの現在のスクラム理解不足を批判的に指摘すること

* 急かすトーン（段階的な成長であることを尊重する）
