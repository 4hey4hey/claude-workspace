# PO（プロダクトオーナー）機能の確保による開発スピードの最大化について

## 成果への影響

現在の体制にはPO（プロダクトオーナー：開発の優先順位と仕様を決める役割）が不在です。このまま開発を開始した場合、エンジニアが本来の開発業務に集中できず、期待する開発スピードの実現が困難です。

## 現状

これから開発を本格的に立ち上げるフェーズですが、スプリントバックログの優先順位決定、開発バックログの作成、受入条件の定義、やらないことの判断を担うPOが不在です。PdMは本業があり、PO業務を片手間で担うことは現実的ではありません。現状では、これらの業務の担い手が決まらないまま開発開始を迎える見通しです。

## 構造的な原因

PO不在でも、バックログ作成や優先順位判断といったPO業務そのものは消えません。担い手がいなければ、エンジニアが非公式にこれらを引き受けることになります。その結果、エンジニアの開発時間が削られます。さらに、権限・責任が曖昧なまま判断を行うため、意思決定が遅れ、手戻りも発生します。つまり、POを置かないことはエンジニアリソースの温存にはならず、開発もPO業務も中途半端になる状態を生みます。

## このまま進めた場合の見通し

- **放置コスト**: エンジニアがPO業務を非公式に吸収し、開発工数の一部がバックログ整理や仕様確認に消えます。PO機能がないまま進めても、エンジニアの生産性はすでに下がります。
- **機会損失**: 開発初期にPOを明確に置けば、最初からバックログ整備と「やらないこと」の判断が機能し、立ち上がりのスピードが大きく変わります。後から置くほど、この初速の差は取り戻しにくくなります。
- **不可逆性**: 役割が曖昧なまま開発が進むと、優先順位の根拠がない仕様や不要な機能が積み上がります。これらの負債は後から巻き戻すコストが大きく、早期に解消するほど効果的です。
