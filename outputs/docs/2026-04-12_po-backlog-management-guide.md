## 1. バックログとは何か

スクラムガイド2020の定義：

> プロダクトバックログは、プロダクトの改善に必要なものの、創発的かつ順番に並べられた一覧である。

平たく言うと、**「このプロダクトでやるべきこと」を、価値が高い順に並べた1つのリスト**。POが責任を持つ唯一の公式な「やることリスト」であり、チームの作業の源泉。

バックログに入っていないものは、チームが着手する理由がない。逆に、バックログに入っているものは全て「POが判断した結果」。だからこそバックログはPOの意思決定の鏡になる。

バックログの構成要素は「**プロダクトゴール + プロダクトバックログアイテム（PBI）**」。プロダクトゴールがバックログ全体の北極星であり、PBIはそのゴールを達成するための個々の作業項目。

### バックログの3つの役割

| 役割 | 内容 |
|---|---|
| **計画のツール** | 「何を・どの順番で作るか」を表現する。リリース計画の根拠になる |
| **透明性のツール** | チーム・SH・経営が「今何に取り組んでいて、次に何が来るか」を見られる |
| **判断の記録** | POが「何をやると決めたか」「何をやらないと決めたか」の証跡になる |

## 2. バックログの階層構造

バックログの全てのアイテムを同じ詳細度で管理しようとすると破綻する。**上位は詳細に、下位は粗く**が原則。

```
┌──────────────────────────────────────────────┐
│  上位 1-5件    【Ready】                       │
│  すぐ着手できる状態。受入条件が明確。          │
│  チームが見積もり済み。1スプリントで完了可能。  │
├──────────────────────────────────────────────┤
│  中位 6-15件   【仕込み中】                    │
│  Whyは明確。Whatの粒度はまだ粗い。             │
│  次のリファインメントで詳細化する候補。         │
├──────────────────────────────────────────────┤
│  下位 16件〜   【アイデア・要望】               │
│  タイトルとWhyだけ。詳細化はまだ不要。          │
│  3スプリント以上動かなければ棚卸し対象。        │
└──────────────────────────────────────────────┘
```

### 各階層でPOがやること

| 階層 | PBIの状態 | POの主な作業 | 頻度 |
|---|---|---|---|
| **上位（Ready）** | 受入条件確定、見積もり済み、依存関係解消済み | 受入確認の準備。スプリント中の仕様質問に即応 | 毎日 |
| **中位（仕込み中）** | Whyは明確、Whatは粗い、受入条件は素案 | PBIの詳細化。SH/PdMへの確認。リファインメントへの持ち込み | 週2-3回 |
| **下位（アイデア）** | タイトルとWhyのみ | 棚卸し。不要なら優先順位を下げるか削除 | 週1回（バックログ整理時） |

**よくある失敗：**
- **全部を細かくしすぎる：** 全体が見通せなくなり、並び替えが困難になる。変化が起きると大量のアイテムが無駄になる。100件を超えるとメンテナンスが破綻する
- **全部を粗いままにする：** スプリントで完成しない。一発勝負になりがちで成果量が不安定になる
- 「上位3件が常にReadyであること」を最低ラインにする

### Ready定義（準備完了の定義）

PBIをスプリントに投入する前に満たすべき基準をチームで合意しておく。**準備不足のPBIを投入すると、スプリント中に仕様確認で止まり、着手後に規模が想定と違うことが判明し、完成確率が下がる。** 「生煮えのPBIを食べると腹を壊す」。

Ready定義の例：

- [ ] PBIの内容が用意され、価値が明確である
- [ ] 大きな疑問・未決事項がない
- [ ] 開発チーム全員が「何を作るか」を理解している
- [ ] 受入条件が明確で検証可能である
- [ ] 他PBIとの依存関係が明確である
- [ ] 見積もられている
- [ ] 非機能要件（パフォーマンス等）が明確である
- [ ] 1スプリントで完了できるサイズである

## 3. PBIの書き方

PBIは「チームがこれを見て何を作ればいいか分かる」状態にする。テンプレートに固執する必要はないが、最低限以下を含める。

### 基本テンプレート

```
■ タイトル: [ユーザーの行動] を [どうする]
  例: 「ガス料金の明細をアプリで確認できるようにする」

■ なぜ作るのか（Why）:
  - どのユーザーの、どんな課題を解決するか
  - これを作ることで何が変わるか（Outcome Criteria）
  例: 「毎月コールセンターに問い合わせている月3,000件のうち、
       明細確認の問い合わせ（推定40%）をアプリに吸収したい」

■ 何を作るのか（What）:
  - 機能の概要を箇条書き3-5個で書く
  - スコープの境界（含むもの・含まないもの）を明示する
  例: 「含む: 直近12ヶ月の月別明細表示、PDF出力
       含まない: 支払い方法の変更（別PBI）」

■ 受入条件（Acceptance Criteria）:
  - Doneかどうかを判断できる、検証可能な条件を箇条書きで
  例: 「□ 過去12ヶ月の明細が月別に表示される
       □ 月を選択すると詳細が展開される
       □ PDF出力ボタンでダウンロードできる」

■ アウトカム基準（Outcome Criteria）:
  - この機能で期待するユーザーの行動変化・ビジネス成果
  例: 「明細確認のコールセンター問い合わせが月1,200件→600件に減少」
  ※ 最初はなくてもよい。慣れてきたら書く

■ サイズの目安:
  - 1スプリントで完了できるサイズか？ Noなら分割を検討する
  - 見積もりはチームが行う（POは口を出さない）

■ 参考情報:
  - 関連するデザインモック、仕様書、SHからの要望メモなどへのリンク
```

### 最初の一歩

新任POは最初から全項目を書く必要はない。**「タイトル」「なぜ作るのか」「受入条件」の3つだけ**でスタートする。リファインメントでチームに見せて「何が足りないか教えてほしい」と聞く。チームからの質問がPBIを育てる。

### PBIの3C — フォーマットより会話

PBIで最も重要なのはフォーマットではなく、**課題・規模感・必然性に関する会話を誘発すること**。Ron Jeffriesの3Cモデルがこれを端的に表している。

| C | 意味 | POがやること |
|---|---|---|
| **Card（カード）** | PBIをカードに書く。詳細を全て書くのではなく、会話のきっかけになる程度 | テンプレートに沿って書く。ただし「書けば終わり」ではない |
| **Conversation（会話）** | PBIの詳細はPOとチームの会話から引き出す | リファインメントで「なぜこれを作るのか」を説明し、質問に答える |
| **Confirmation（確認）** | 受入条件で実装の正しさを確認する | 「何をもってDoneか」をチームと合意する |

**決まったフォーマットで書くだけでは共通理解は生まれない。会話なしに共通理解は生まれない。**

### よいPBIの判断基準 — INVEST

よいPBIが満たすべき6つの特性。全て完璧に満たす必要はないが、意識することでPBIの質が上がる。

| 特性 | 意味 | チェック |
|---|---|---|
| **I**ndependent（独立） | 他のPBIと相互依存がない。単独で着手・完了できる | 「このPBIだけ完了してもユーザーに価値が届くか？」 |
| **N**egotiable（交渉可能） | 仕様は会話の道具であり、一度決めたら絶対ではない | 「チームとの対話で内容を調整できる余地があるか？」 |
| **V**aluable（価値がある） | ユーザー・ビジネス・チームのいずれかに価値がある | 「これを作ることで誰が嬉しいか？」 |
| **E**stimable（見積もれる） | チームが規模感を把握できる程度に具体的 | 「チームが見積もりできるか？できないなら何が不明か？」 |
| **S**ized Right（適切サイズ） | 1スプリントで完了できるサイズに分割されている | 「大きすぎないか？小さすぎて価値が見えなくなっていないか？」 |
| **T**estable（テスト可能） | 完了したかどうかを客観的に判断できる | 「受入条件で○×判定ができるか？」 |

### PBIで「書かないこと」

- **実装方法の指定**（「ReactのuseStateを使って〜」）→ Howはチームが決める
- **曖昧な受入条件**（「使いやすいUIにする」）→ 検証可能な基準にする
- **複数の価値を1つに詰め込む**（「明細表示と支払い方法変更」）→ 1PBI=1価値の原則

### PBIのアンチパターン3つ

**アンチパターン1: 何も説明していないPBI**

> NG例: 「明細ダウンロード機能がほしい。なぜなら明細ダウンロード機能がないからだ」

「ない」のは事実だが、作る理由を説明していない。何らかの課題を解決したいはず。**課題を理由として明示する。** 「コールセンターへの明細確認問い合わせが月1,200件あり、ユーザー・オペレーター双方の負荷になっている」のように書く。

**アンチパターン2: 画面単位のPBI**

> NG例: 「マイページ画面を作る」

1つの画面には多くの要素が含まれる。1スプリントで完成するには大きすぎることが多い。必須と非必須の要素が混在し、非必須の部分に時間を使ってしまう。**ユースケース完結の最小単位に切り分ける。** 「マイページで契約情報を確認できる」「マイページから請求書PDFをダウンロードできる」のように分割する。

**アンチパターン3: ソリューション指向のPBI**

> NG例: 「デプロイを自動化したい」

ソリューション（解決策）を指示してしまっている。本当の課題は何か？ デプロイ時間が長い？手順をよく間違える？特定の人にしかできない？ これらの問題はフル自動化なしでも解決できるかもしれない。**課題から書く。** 「デプロイに毎回2時間かかり、その間エンジニアが拘束される」のように。

## 4. PBIの分割テクニック

PBIが1スプリントで完了できないサイズのとき、分割が必要になる。新任POが最も苦労するスキルの一つ。

### 分割の鉄則

**「技術レイヤーで切らない。ユーザー価値で切る」**

- **NG:** 「フロントエンド」「バックエンドAPI」「DB設計」で分割 → 1つだけ完了してもユーザーに価値が届かない
- **OK:** 「月別明細の表示」「PDF出力」「過去明細の検索」で分割 → 1つ完了するたびにユーザーが使える

### 10の分割パターン

| # | パターン | 例 | いつ使うか |
|---|---|---|---|
| 1 | **ワークフローの分割** | 「申し込み→確認→完了」を3つに | 一連のフローが長いとき |
| 2 | **ビジネスルールの分割** | 「通常契約」と「セット契約」を別々に | 条件分岐が複雑なとき |
| 3 | **操作（CRUD）の分割** | 「閲覧」「作成」「編集」「削除」を別々に | CRUD全部を一度に作ろうとしているとき |
| 4 | **利用者の役割で分割** | 「ユーザー向け」と「管理者向け」を別々に | 複数のロールが関わるとき |
| 5 | **ハッピーパスとそれ以外** | まず正常系→次に異常系・エラー処理 | エラーパターンが多いとき |
| 6 | **プラットフォーム分割** | 「iOS対応」と「Android対応」を別々に | 複数プラットフォーム対応のとき |
| 7 | **最適化の度合いで分割** | 「まず動く」→「速くする・使いやすくする」 | 性能要件やUX要件が重いとき |
| 8 | **入力パラメータで分割** | 検索条件を1つずつ追加 | 検索・フィルタ機能のとき |
| 9 | **ブラウザ・バージョン** | まず最新ブラウザ→次にレガシー対応 | ブラウザ互換性要件があるとき |
| 10 | **スパイク＋実装** | 「技術検証（調査のみ）」→「本実装」 | 不確実性が高いとき |

### やってはいけない分割方法

**工程で分割してはいけない：** 「設計」「開発・テスト」のように分けると、後続スプリントの作業が自動的に決まってしまい、変化に対応できなくなる。スプリントレビューでフィードバックも得られない。

**技術レイヤーで分割してはいけない：** 「UI」「DB」「サーバーロジック」のように分けると、1つだけ完了してもユーザーに価値が届かない。

### 分割の判断フロー

```
このPBIは1スプリントで終わるか？
  → チームに聞く（POが勝手に判断しない）
  → 「終わらない」と言われた場合:

1. ユーザーに届く最小の価値は何か？を考える
2. 上の10パターンのどれが当てはまるか確認する
3. 分割後の各PBIが「単体でユーザーに価値を届けるか」を確認する
4. 届かない場合は分割の軸を変える
```

### mTGでの分割例

**分割前：** 「ガス機器通報サービスをmTGに組み込む」（大きすぎる）

**分割後：**
1. 「通報フォームの画面を表示する」（ワークフロー分割 — Step 1）
2. 「通報内容を送信し、受付番号を表示する」（ワークフロー分割 — Step 2）
3. 「過去の通報履歴を一覧で確認できる」（操作の分割 — 閲覧）
4. 「通報ステータスの更新通知をプッシュで受け取る」（操作の分割 — 通知）

それぞれが1スプリントで完了可能で、1つリリースするたびにユーザーが使える状態になる。

## 5. 優先順位の付け方

バックログの並び順はPOの最も重要な判断。「全部大事」は判断の放棄。

### 4つの問いで判断する

バックログの並び替えで「どっちを先にやるべきか」迷ったとき、以下の4つの問いを順に確認する。

**問い1: どちらがプロダクトゴールに近いか？**

ゴールに直結する方を上にする。どちらもゴールと関係ないなら、そもそもバックログの上位にある理由を疑う。

**問い2: どちらの満足度ギャップが大きいか？**

「ユーザーの今の体験」と「本来あるべき体験」の差が大きい方を上にする。差が大きいほど、解決したときのインパクトが大きい。

**問い3: どちらのリスクが高いか？**

リスク（不確実性）が高いものは先にやる。小さく試して不確実性を下げるため。リスクが低い（やれば確実にできる）ものは後回しにしても大丈夫。

**問い4: どちらの依存関係が多いか？**

他のPBIや他チームの作業をブロックしているものは先にやる。自分だけで完結するものは後でもよい。

4つの問いで決まらない場合は、**PdMに相談する**（What/Whyの判断に関わる可能性がある）。

### 優先順位を変えるとき

優先順位は固定ではなく、以下のタイミングで見直す。

| タイミング | なぜ見直すか | 何をするか |
|---|---|---|
| **レビュー後** | SHフィードバックで新しい情報が入った | フィードバックをバックログに反映し、並び順を再評価 |
| **PdMとのすり合わせ後** | ロードマップの方向性が変わった | プロダクトゴールとの整合を再確認し、並び順を調整 |
| **スプリント中に仕様が変わった** | 開発中に想定外の事実が発覚した | 影響範囲を評価し、必要なら次スプリントの並び順を変更 |
| **外部環境の変化** | 競合リリース、法規制変更、障害発生 | 緊急度を再評価し、必要なら割り込みを判断 |

**優先順位を変えたときは必ず理由を記録する。** 後から「なぜこの順番になったのか」が追えなくなると、同じ議論を繰り返すことになる。

### 「やらない」判断の実践

「やらない」はPOの最も重要な判断の一つ。しかし新任POには心理的なハードルが高い。段階的に練習する。

**レベル1（最初の30日）: 小さな「やらない」**
- バックログの下位にある3スプリント以上未着手のアイテムを見つけて、「今はやらない」と決める
- チーム内だけに「この3件は優先度を下げました」と伝える

**レベル2（60日頃）: SHへの「今ではない」**
- SHからの要望に対して「プロダクトゴールとの整合性を踏まえると、今期は対応が難しいです。理由は〜」と伝える
- 根拠をセットで伝えることで、「断った」ではなく「判断した」として受け取ってもらえる

**レベル3（90日以降）: 戦略的な「やらない」**
- バックログ全体を見て「この領域は今期は投資しない」とPdMと合意する
- 資源を集中させるための意図的な判断として「やらない」を使う

## 6. リファインメントの進め方

リファインメントはPBIを「Ready」にするためのプロセス。セレモニーとしての形式より、**PBIを継続的に育てる活動**として捉える。

### リファインメントの2つの形態

| 形態 | 内容 | 頻度 |
|---|---|---|
| **フォーマル** | チーム全員で集まり、PBIの詳細化・見積もりを行う | 週1回、1-2時間 |
| **インフォーマル** | POとエンジニア1-2名で、個別PBIの疑問点を解消する | 随時（Slackや短い打ち合わせ） |

フォーマルなリファインメントだけで全てのPBIをReadyにしようとすると時間が足りない。**インフォーマルな事前整理でPBIの粗い部分を潰しておき、フォーマルなリファインメントでは確認と見積もりに集中する**のが効率的。

### リファインメントの進め方（フォーマル）

```
所要時間: 1-2時間（スプリントの10%以下が目安）

1. PBI候補の提示（PO: 5分）
   - 今回詳細化するPBI 2-3件を提示する
   - 各PBIの「なぜ作るのか」を1-2分で説明する

2. 質疑応答（チーム全員: PBIあたり15-20分）
   - チームからの質問に答える
   - 「分からない」は正直に言う。持ち帰って次回回答する
   - チームの質問からPBIの曖昧な部分が浮き彫りになる

3. 受入条件の合意（PO + チーム: PBIあたり10分）
   - Acceptance Criteria を一緒に確認・修正する
   - 「何をもってDoneとするか」をチームと合意する

4. 見積もり（チーム: PBIあたり5-10分）
   - POは見積もりの数字に口を出さない
   - 見積もりが大きすぎる場合は分割を検討する（POの判断）

5. Ready判定（PO: 5分）
   - 各PBIがReady基準を満たしているか確認
   - 満たしていないものは次回に持ち越し
```

### 「リファインメントに時間をかけられない」は逆

「時間がなくてPBIの手入れにそんなに時間かけられない」という声はよく聞く。しかし実態は逆。**PBIの手入れをしないから時間がなくなる。** 準備不足のPBIをスプリントに投入すると、スプリント中に仕様確認・認識のズレ・手戻りが発生し、結果的にもっと多くの時間を失う。

### リファインメントでPOが意識すること

- **「Why」の説明に時間を使う。** チームが「なぜこれを作るのか」を理解していれば、Howの判断がよくなる
- **チームの質問を歓迎する。** 質問が出ないリファインメントは危険。チームが理解していないか、関心がないサイン
- **「分からない」を恐れない。** 全てを知っている必要はない。「持ち帰って確認します」と言えることが大事
- **Howに口を出さない。** 「この技術で実装して」は禁句。「こういう価値を届けたい」と伝える

### リファインメントでやること一覧

- 新規PBIを追加し、見積もる
- 不要になったPBIを削除する
- 既存PBIを見積もり直す（状況が変わった場合）
- バックログの並び順が最新であることを確認する
- 上位PBIの内容を確認する
- 直近スプリント投入予定のPBIを詳細化しReadyにする
- 上位で大きすぎるPBIを着手可能サイズに分割する

## 7. スプリントゴールの設定

スプリントゴールはそのスプリントの「北極星」。チームが判断に迷ったとき「スプリントゴールに近づくか？」で意思決定できる状態を作る。

### よいスプリントゴールの条件

| 条件 | 説明 | 例 |
|---|---|---|
| **価値で表現する** | 「タスク完了」ではなく「ユーザーに届く価値」で書く | NG: 「明細APIを実装する」 → OK: 「ユーザーがアプリでガス料金の明細を確認できるようにする」 |
| **1文で言える** | 長すぎるゴールは焦点がぼやけている | NG: 「明細表示とPDF出力と検索機能を全て完成させる」 → OK: 「明細を月別に閲覧できる状態を作る」 |
| **検証可能** | スプリント終了時に達成したかどうか判定できる | NG: 「UXを改善する」 → OK: 「料金画面の離脱率を現在の35%から25%以下にする」 |
| **プロダクトゴールとつながる** | なぜこのスプリントでこのゴールなのかが説明できる | 「今期のプロダクトゴール『問い合わせ件数の削減』に向けて、最初のステップとして明細表示を作る」 |

### スプリントゴール設定の手順

```
Step 1: プロダクトゴールとの接続を確認する
  「今期のプロダクトゴールは何か？」
  「このスプリントはそのゴールにどう貢献するか？」

Step 2: 上位PBIから価値のまとまりを見つける
  バックログ上位の3-5件を見て、「これらが完了したとき
  ユーザーにとって何が変わるか？」を1文にまとめる

Step 3: チームと共有し、合意する
  プランニングで提示し、チームが「実現可能」と合意するか確認
  → 合意できない場合はスコープを調整する（ゴールの質は下げない）

Step 4: スプリント中の判断基準にする
  「この作業はスプリントゴールに貢献するか？」で
  スコープの追加・削除を判断する
```

### スプリントに投入するPBIの構成比率

スプリントゴールに直結するPBIだけで構成できれば理想だが、現実にはバグ修正・セキュリティ対応・技術的負債の返済・各種調査など、ゴールに直結しないが必要な作業もある。

**推奨比率：スプリントゴール直結 7 : その他 3**

「その他」が5割を超えるようなら、スプリントゴールの達成が危うくなる。逆に「その他」がゼロなら、重要だが緊急でない技術的投資が後回しにされ続けるリスクがある。

### 未完成PBIの扱い — 完成は0か1

スプリント終了時にPBIが未完成の場合、**「あと少し」でも未完成は未完成。** 一部完成や条件付き完成はない。

- デモ対象外
- プロダクトバックログに戻す（自動的に次スプリントに入るわけではない）
- 再見積もりする
- POが改めて優先順位を判断する
- スプリントゴールが達成できていれば、そのPBIに二度と着手しない判断もあり得る

## 8. 技術的負債のバックログでの扱い方

技術的負債（Technical Debt）はPOが避けて通れないテーマ。エンジニアから「リファクタリングしたい」「テストを書きたい」と要望が出たとき、どう判断するか。

### 原則: 技術的負債もPBIとしてバックログに入れる

技術的負債を「見えない裏作業」にしない。バックログに入れることで透明性を確保し、他のPBIと同じ土俵で優先順位を判断する。

### 判断の考え方

| 状況 | 判断 | 理由 |
|---|---|---|
| 負債を放置するとユーザーに影響が出る | **優先度を上げる** | パフォーマンス劣化、障害リスク等はユーザー価値に直結 |
| 負債を放置すると開発速度が落ちる | **投資として判断** | チームの生産性はプロダクト価値を生む基盤。ROI観点で評価 |
| 「きれいにしたい」だけ | **急がない** | 美しいコード自体はユーザー価値ではない。他のPBIが優先 |
| 負債の影響範囲が限定的 | **影響が広がる前に対処** | 小さいうちに返す方がコストが低い |

### 実践的なルール

- **スプリントの10-20%を技術的負債に充てるルール**をチームと合意する。毎スプリント少しずつ返す方が、まとめて返すより安全
- エンジニアが負債を報告しやすい環境を作る。「報告したら怒られる」状態だと、負債が隠れて大きくなる
- 負債のPBIにも「なぜ返す必要があるか」「返さないと何が起きるか」をWhyとして書く。POが理解できない負債は優先順位を判断できない

## 9. リリース計画の基本

リリース計画は「いつ頃、何が届くか」の見通し。POが作り、SHとチームに共有する。

### リリース計画の作り方

```
Step 1: スコープを決める
  - プロダクトゴールを達成するために必要なPBIをリストアップする
  - 「最低限これがあればゴール達成」（MVP）と「あったら嬉しい」を分ける

Step 2: ベロシティを確認する
  - 直近3-5スプリントの平均ベロシティを確認する
  - チームに聞く。POが勝手にベロシティを想定しない

Step 3: 見通しを立てる
  - MVPのPBI合計ポイント ÷ 平均ベロシティ = 必要スプリント数
  - これはあくまで「見通し」であり「約束」ではない

Step 4: 不確実性を加味する
  - 見通しに幅を持たせる（「3〜5スプリント」等）
  - 不確実性が高いPBIがある場合は、先にスパイクを入れて不確実性を下げる

Step 5: SHに共有する
  - 「○月頃にMVPをリリースできる見通しです。不確実性が高い部分は
    スプリントごとに精度を上げていきます」と伝える
  - SHには「予定」ではなく「現時点の見通し」であることを明示する
```

### リリース計画の更新頻度

- **毎スプリント終了時に見直す。** ベロシティの実績・PBIの追加/削除を反映
- **大きな変更があった場合はSHに即共有。** 「聞いていない」を防ぐ

### よくある失敗

- **日付を約束してしまう** → 「3月15日にリリースします」ではなく「3月中旬のリリースを目指していますが、スプリントごとに精度を上げます」
- **ベロシティを盛る** → チームの実績ベロシティを使う。「がんばればもっとできる」はリスク
- **バッファを入れない** → 予想外の障害・仕様変更は必ず起きる。見通しに幅を持たせる

## 10. バックログの健康管理

バックログは生き物。放置すると腐る。定期的な手入れで健康を保つ。

### 週次バックログ整理（30分）

| Step | 所要時間 | やること |
|---|---|---|
| **上位10件の確認** | 10分 | 並び順がプロダクトゴールと整合しているか確認。「なぜこの順番か」を自分の言葉で説明できるか自問する |
| **中位のスキャン** | 10分 | 11-30位をざっと見る。3スプリント以上動いていないアイテムは「本当に必要か？」を問う |
| **新規追加の整理** | 10分 | 前週に追加されたアイテムに最低限の「Why」と「受入条件の素案」を書き、仮の優先順位をつける |

### 月次バックログ棚卸し（1時間）

月に1回、バックログ全体を俯瞰して大掃除する。

```
チェック1: 総数は適正か？
  - 50件以下が目安。超えていたら下位を整理する
  - 「いつかやる」アイテムは別リスト（アイスボックス）に移動

チェック2: プロダクトゴールとの整合性
  - バックログの上位20件がプロダクトゴールに紐づいているか
  - 紐づかないアイテムが上位にあれば、順番を見直す

チェック3: 「化石」の発掘
  - 6スプリント以上未着手のアイテムをリストアップ
  - 「まだ必要か？」をPdMに確認し、不要なら削除

チェック4: 粒度のバラつき
  - 上位に大きすぎるPBI（複数スプリント必要）が残っていないか
  - 残っていれば分割する

チェック5: 依存関係の確認
  - 他チームへの依存がブロッカーになっているPBIがないか
  - ブロックされているものは依存先と調整し、解消の見通しを立てる
```

### 新しいPBIはどこから見つけるか

バックログのアイテムは最初に全部作るものではない。プロダクトの成長に伴い、以下の情報源から継続的に発見する。

- **スプリントレビュー** — SHからのフィードバック
- **サポートへのリクエスト** — ユーザーの実際の困りごと
- **ユーザーインタビュー・行動観察** — 顕在化していないニーズ
- **メトリクス・ログ解析** — 利用データから見える課題
- **開発チームのペイン** — 技術的負債、開発体験の改善
- **PdMのロードマップ更新** — 戦略レベルの方向性変更
- **競合動向・市場変化** — 外部環境の変化

### バックログの健康指標

自分のバックログが健全かどうかを判断する簡易チェック。

| 指標 | 健全 | 要注意 | 危険 |
|---|---|---|---|
| **Ready状態のPBI数** | 3件以上 | 1-2件 | 0件（次スプリントの着手が危うい） |
| **総アイテム数** | 20-40件 | 50-70件 | 70件超（管理が破綻する） |
| **上位5件の最終更新日** | 1週間以内 | 2週間以上前 | 1ヶ月以上前（放置されている） |
| **プロダクトゴールと無関係なPBI比率** | 10%以下 | 20-30% | 30%超（焦点がぼけている） |
| **「やらない」判断の頻度** | 月2件以上 | 月1件 | 0件（判断を避けている） |

## 11. SHからの要望をバックログに取り込む流れ

SHから「これを作ってほしい」と言われたとき、即答で「やります」とも「やりません」とも言わない。以下の手順で対応する。

```
Step 1: まず聞く
  「ありがとうございます。背景を教えてください。
   どんなユーザーの、どんな課題を解決したいですか？」
  → 要望の裏にある本当のニーズを理解する

Step 2: 記録する
  バックログに追加する（ただし優先順位は未定）
  SHに「バックログに入れました。優先順位はプロダクトゴールとの
  整合性を見て判断します」と伝える

Step 3: 評価する
  - プロダクトゴールとの関連性
  - 影響を受けるユーザー数・頻度
  - 開発コスト（チームにざっくり聞く）
  - やらなかった場合のリスク

Step 4: 判断を伝える
  「優先順位は○○です。理由は〜」と根拠とセットで伝える
  → 「やらない」場合は「今ではない」と理由を添える
  → 「やる」場合はいつ頃着手できるかの見通しを添える
```

**やってはいけないこと：**
- その場で「やります」と約束する（バックログの優先順位を無視することになる）
- 「検討します」と言って放置する（信頼を失う。1週間以内に判断を伝える）
- SHの要望をそのままPBIにする（要望の裏のニーズを掘らないと、的外れな機能を作る）

## 12. バックログ管理のチェックリスト（毎スプリント確認用）

各スプリントの開始前にこのリストを確認する。

- [ ] 上位3件のPBIがReady状態か（受入条件確定・見積もり済み・依存解消）
- [ ] スプリントゴールを1文で言えるか
- [ ] スプリントゴールがプロダクトゴールとどうつながるか説明できるか
- [ ] 前スプリントのフィードバックがバックログに反映されているか
- [ ] バックログの総数が50件以内か
- [ ] 3スプリント以上未着手のアイテムを棚卸ししたか
- [ ] SHからの新規要望に1週間以内に判断を伝えたか
- [ ] 技術的負債のPBIがバックログに含まれているか（隠れていないか）
- [ ] リリース計画の見通しが最新の状態か
