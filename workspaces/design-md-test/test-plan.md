# Design.md テストプラン

## 目的
Design.md がスキル実装に反映されているか、PPT生成品質に影響があるかを検証

## テストケース1: シンプルな3スライド提案

**ゴール**: 基盤投資の実施判断を意思決定者から得る

**スライド構成**:
```
P1: 表紙 — 「基幹システム移行」
P2: 現行システムは2027年EOL。移行が必須
P3: 提案する移行アーキテクチャ（フロー図）
P4: スケジュール・体制・見積り → 実施判断へ
```

## テストケース2: Design.md ルール検証

### 色・背景チェック
- [ ] 全スライド背景は白（FFFFFF）か
- [ ] カラーバンドは高さ1.0インチ以下か（18%以下）
- [ ] セマンティック色（Success/Warning/Error）が正しく使われているか

### フォント・タイポグラフィチェック  
- [ ] Noto Sans JP（またはフォールバック）が適用されているか
- [ ] 本文サイズは 12pt（16CSS px）以上か
- [ ] テーブル・ボディテキストの行間が150%以上か

### 構成・情報設計チェック
- [ ] タイトルが「〜について」ではなく主張文か
- [ ] 1スライド1メッセージの原則が守られているか
- [ ] 図解/フロー/マトリクスが含まれているか

### AI生成資料の典型パターン排除チェック
- [ ] グラデーション背景がないか
- [ ] 大きすぎる角丸（4px超）がないか
- [ ] 絵文字が使われていないか
- [ ] 英語カテゴリラベル見出し（"FEATURES"等）がないか

## 実行手順

### Step A: 設計テンプレ作成
```
杉崎 → pptx-creator スキル呼び出し
「次のテーマで3スライドの提案を作って：『基幹システム移行に向けた実施判断』」
```

### Step B: 設計の承認
pptx-creator が主張文・構成・「だから何？」を提示
→ ユーザーが承認

### Step C: 生成
corporate テーマ（またはDADS）で生成

### Step D: 検証
生成されたPPTXを手動チェックまたは PDF変換して確認

## 検証観点

### 観点1: Design.md ルール遵守
PPT生成物が `.claude/refs/design.md` の「Do's and Don'ts」を守っているか

### 観点2: スキルメッセージ

### 観点3: 改善ポイント発見
テスト中に「Design.md に追記すべき内容」「スキル側で参照すべきルール」を発見
