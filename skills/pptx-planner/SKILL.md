# PPT設計テンプレ作成スキル

## コンセプト：「作らせるな、埋めさせろ」

AIにスライドを「作って」と丸投げすると、きれいだけど使えないものが出る。
原因は「見た目テンプレ」しか渡していないから。

**テンプレには2種類ある:**

| 見た目テンプレ               | 設計テンプレ                   |
| ---------------------------- | ------------------------------ |
| 色・フォント・配置・ロゴ位置 | 各ページで何を主張するか       |
| 見栄えを揃えるもの           | どんな順番で読ませるか         |
| AIに渡されることが多い       | 読んだ人に何を判断してもらうか |

**このスキルの役割は「設計テンプレ」を人間とAIが一緒に作ること。**
設計テンプレが完成したら、`pptx-creator` に渡して「埋める」フェーズへ移行する。

> AIの真価は作業の時短だけじゃなく、自分のノウハウを言葉にすることを強制してくれること。
> 「何を言いたいか」は人間が決める。AIはそれを具体化・整理する。

---

## インプット受け取り

スキル起動時に以下を確認する（すでに提供済みの情報はスキップ）:

```
■ ゴール（誰に、何を判断してもらうか）
■ 聴衆・状況（誰が見るか、どんな場面か）
■ 利用可能なデータ・資料（任意）
■ スライド枚数の目安（任意）
```

情報が不足している場合は AskUserQuestion でヒアリングする。

---

## Step 1: ゴールを1行で確定する

**「この資料で、誰に、何を判断してもらいたいか」を1行で書く。**

| 要素       | 内容                           |
| ---------- | ------------------------------ |
| 誰に       | 意思決定者の属性・役職・関係性 |
| 何を       | 期待する判断・行動・合意       |
| いつまでに | 判断タイミング（任意）         |

**例:**

- 「ある企業の意思決定者に、Phase1の実施を合意してもらう」
- 「部門長に、認証基盤刷新への投資承認を得る」
- 「スクラムチームに、スプリント目標への合意を形成する」

**ゴールが明確でない場合は AskUserQuestion で必ず確認する。** 曖昧なゴールで設計を進めない。

**ドキュメントからゴールを自動読み取りした場合も、必ず AskUserQuestion で「この解釈で合っていますか？」と確認を取ること。** ゴール解釈のズレが最も後工程に影響する。

ユーザーの確認を取ってから Step 1.5 へ。

---

## Step 1.5: ストーリーラインを先出しする

**主張・根拠・つなぎを3〜5行で先に示し、全体の「話の流れ」に合意してから詳細設計に入る。**

### ストーリーラインの形式

```
1. [問題提起] 〇〇という状況が生じている
2. [診断] その原因/背景は△△にある
3. [解決策] だから□□を提案する
4. [根拠] それによって〜という効果が得られる
5. [CTA] まず〜をXX年度中に実施してほしい
```

> 行数は3〜5行が目安。「これを読めば話の骨格がわかる」レベルで十分。細かい主張はStep 2で詰める。

### ストーリーラインのチェック軸

- **一貫性**: ゴール（誰に・何を判断してもらうか）に向かっているか
- **論理の飛び**: 「なぜ？」と突っ込まれる箇所がないか
- **聴衆目線**: 聴衆が「Yes/No」を判断するのに必要な情報が揃っているか

### よくある構成パターン（例示）

| 構成タイプ | ストーリーラインの骨格 |
| ---------- | ---------------------- |
| 課題解決型 | 問題 → 原因 → 解決策 → 効果 → CTA |
| 投資承認型 | 背景/機会 → 現状の限界 → 提案 → ROI/リスク → 判断お願い |
| 経営報告型 | 結論先出し → 根拠 → 詳細 → 今後の方針 |
| プロジェクト提案型 | 背景 → 課題 → ゴール → 手段 → ロードマップ → 体制 |

**ストーリーラインが通らなければStep 1に戻ってゴールを修正する。**

### 提示の手順（必須）

1. **採用した構成パターンと選択理由を1行で明示する**（例:「投資承認型を採用。GMに判断を求める資料のため」）
2. ストーリーラインを提示する
3. **AskUserQuestion で確認を取る。「OK」を受け取るまで Step 2 に進まない**
   - 「OK、このまま進める」
   - 「順番を変えたい」→ どのパートか深掘り質問を追加する
   - 「論点を足したい / 削りたい」→ どのパートか深掘り質問を追加する

ユーザーの確認を取ってから Step 2 へ。

---

## Step 2: 各ページの主張だけ先に並べる

**合意したストーリーラインをベースに、各スライドの主張文（=スライドタイトル）を展開する。中身はまだ書かない。**

### 主張文の原則（最重要）

全コンテンツスライドのタイトルは、そのスライドで言いたいことを1行で言い切る主張文にする。

| BAD（カテゴリラベル） | GOOD（主張文）                                                |
| --------------------- | ------------------------------------------------------------- |
| 現状分析              | 現行システムは2027年のEOLまでに移行が必須                     |
| 提案内容              | 認証基盤の統合により年間1.2億円のコスト削減が可能             |
| 課題                  | 申し込みのリードタイムが業界平均の3倍になっており離脱率が高い |
| 背景                  | AI Overviewにより公益事業のWeb流入が構造的に消失している      |

### ストーリーチェック

並べた主張文を上から順に読んで、話が通じるか確認する。

```
P1：〇〇である（課題の認識）
P2：原因は△△にある（Why）
P3：解決策として□□を提案する（What）
P4：効果・根拠（So what）
P5：ネクストアクション（Call to action）
```

**ストーリーが通らなければ、順番や中身をユーザーと一緒に入れ替える。**

### 典型的な構成パターン

| パターン                                        | 適用場面               |
| ----------------------------------------------- | ---------------------- |
| 表紙→課題→現状分析→提案→効果→ネクストアクション | 外部向け提案書         |
| 表紙→サマリー→詳細→付録                         | 経営報告・意思決定資料 |
| 表紙→アジェンダ→各議題→まとめ・決定事項         | 会議体資料             |
| 表紙→背景→課題→解決策→ロードマップ→体制         | 大型プロジェクト提案   |

**Step 2 出力後は必ず一度止まる。AskUserQuestion を使い「このスライド構成で Step 3（ボディ設計）に進んでよいですか？」と確認してから Step 3 へ進む。確認なしに Step 3 に自動進行しない。**

ユーザーの確認を取ってから Step 3 へ。

### 「だから何？」の追加

データを載せるだけじゃなく、「だからどうすべきか」の結論を各スライドに1行足す。

| NG（データだけ）     | OK（だから何？付き）                                              |
| -------------------- | ----------------------------------------------------------------- |
| 市場規模は3000億円   | 市場規模3000億円。今参入しないと2年後には競合に取られる           |
| 現行システムは老朽化 | 現行システムは2027年EOL。移行しなければ業務停止リスク             |
| 顧客満足度が低下     | 満足度が前年比15%低下。解約率上昇の先行指標である                 |
| 申し込み件数が増加   | 申し込みが月次120%成長。現行体制では3ヶ月後にキャパオーバーになる |

**「だから何？」は人間の仕事。** AIから提案するが、最終判断はユーザーに委ねる。

### ボディ要素の設計

各スライドに入れるべき要素を具体化し、**図解方式を明記する**。

| 要素タイプ            | 適用場面                   | 図解方式 |
| --------------------- | -------------------------- | -------- |
| 数値・データ          | 課題の定量化・効果の根拠   | — |
| 比較表（2列）         | 現状 vs 提案・選択肢の整理 | parts:addComparisonColumns |
| フロー図（単純・3-5段）| 直線的なプロセス・手順     | parts:addFlowHorizontal/Vertical |
| フロー図（複雑）      | 分岐・合流・スイムレーン   | diagram:フロー |
| マトリクス (2x2)      | 優先度・ポジショニング     | parts:addMatrix2x2 |
| タイムライン（4-7）   | 計画・ロードマップ         | parts:addTimeline |
| ピラミッド（3-5段）   | 階層概念・優先順位         | parts:addPyramid |
| ステップ（3-5）       | シェブロン型の工程          | parts:addStepProcess |
| Before/After          | 改善効果の可視化           | parts:addComparisonColumns |
| 体制図・組織図        | 階層構造のある人員配置     | diagram:組織図 |
| アーキテクチャ図      | システム構成・ネットワーク | diagram:アーキ |
| 箇条書き（4項目以内） | 要点の列挙                 | — |

### 図解方式の判定ルール（上から順にチェック）

1. **図解不要**（テーブル・箇条書き・数値のみ）→ `—`
2. **以下に該当 → `diagram:{種別}`**
   - ツリー構造（組織図・体制図）
   - 分岐・合流のあるフロー / 双方向・循環フロー（PDCA等）
   - スイムレーン / ネットワーク図 / ER図
   - 3カラム以上の比較 / 3x3以上のマトリクス
   - parts.js の制約（ステップ数・カラム数）を超える場合
3. **上記に該当せず parts.js の制約内 → `parts:{関数名}`**

**迷ったら `diagram` に倒す。** parts.js の制約を超えると生成時にレイアウトが崩れるため。

**10枚以上のデッキでは、図解スライドを最低3枚計画する。**

ユーザーの確認を取ってから出力へ。

---

## 出力: 設計テンプレ（Markdown）

完成した設計テンプレをMarkdownで出力し、ファイルに保存する。

**保存先**: `outputs/docs/YYYY-MM-DD_{タイトル}_設計テンプレ.md`

### 出力フォーマット

```markdown
# 設計テンプレ: {タイトル}

## ゴール

{誰に・何を判断してもらうか 1行}

## 聴衆・状況

{聴衆の属性・場面}

## スライド構成

| No   | タイトル（主張文） | だから何？       | ボディ要素                         | 図解方式 |
| ---- | ------------------ | ---------------- | ---------------------------------- | -------- |
| 表紙 | {タイトル}         | —                | タイトル・サブタイトル・日付・所属 | — |
| 1    | {主張文}           | {アクション結論} | {データ・図解・表}                 | {parts:関数名 / diagram:種別 / —} |
| 2    | {主張文}           | {アクション結論} | {データ・図解・表}                 | {parts:関数名 / diagram:種別 / —} |

...
| N | ネクストアクション: {内容} | — | アクション一覧・担当・期限 |

## 参照データ

- {利用可能なデータ・資料のリスト}

## pptx-creator への引き渡しメモ

- テーマ候補: {corporate / modern / dads などの推奨}
- スライド枚数: {N}枚
- 図解スライド: {フロー/マトリクス/比較 など計画した図解}
- diagram スライド数: {N}枚（pptx-diagram で事前生成が必要）
- 特記事項: {制約・注意点}
```

## ネクストアクション

設計テンプレ完成後、以下を案内する:

1. **このまま pptx-creator に渡す** → 「pptx-creatorスキルを使って、この設計テンプレでPPTXを生成して」
2. **さらに設計を磨く** → Step 2/3 に戻ってストーリーを調整
3. **Notionに保存する** → notion-exportスキルで出力

> 設計テンプレは一度作れば組織の武器になる。
> 同じ目的の資料なら、次回からこの設計テンプレを起点にして「埋める」だけになる。
