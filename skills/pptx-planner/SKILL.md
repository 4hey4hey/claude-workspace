# PPT設計テンプレ作成スキル

## コンセプト：「作らせるな、埋めさせろ」

AIにスライドを「作って」と丸投げすると、きれいだけど使えないものが出る。
原因は「見た目テンプレ」しか渡していないから。

**テンプレには2種類ある:**

| 見た目テンプレ               | 設計テンプレ                   |
| ---------------------------- | ------------------------------ |
| 色・フォント・配置・ロゴ位置 | 各ページで何を主張するか       |
| 見栄えを揃えるもの           | どんな順番で読ませるか         |
| AIに渡されることが多い       | 読んだ人に何を判断してもらうか |

**このスキルの役割は「設計テンプレ」を人間とAIが一緒に作ること。**
設計テンプレが完成したら、`pptx-creator` に渡して「埋める」フェーズへ移行する。

> AIの真価は作業の時短だけじゃなく、自分のノウハウを言葉にすることを強制してくれること。
> 「何を言いたいか」は人間が決める。AIはそれを具体化・整理する。

---

## インプット受け取り

スキル起動時に以下を確認する（すでに提供済みの情報はスキップ）:

```
■ ゴール（誰に、何を判断してもらうか）
■ 聴衆・状況（誰が見るか、どんな場面か）
■ 利用可能なデータ・資料（任意）
■ スライド枚数の目安（任意）
```

情報が不足している場合は AskUserQuestion でヒアリングする。

---

## Step 1: ゴールを1行で確定する

**「この資料で、誰に、何を判断してもらいたいか」を1行で書く。**

| 要素       | 内容                           |
| ---------- | ------------------------------ |
| 誰に       | 意思決定者の属性・役職・関係性 |
| 何を       | 期待する判断・行動・合意       |
| いつまでに | 判断タイミング（任意）         |

**例:**

- 「ある企業の意思決定者に、Phase1の実施を合意してもらう」
- 「部門長に、認証基盤刷新への投資承認を得る」
- 「スクラムチームに、スプリント目標への合意を形成する」

**ゴールが明確でない場合は AskUserQuestion で必ず確認する。** 曖昧なゴールで設計を進めない。

**ドキュメントからゴールを自動読み取りした場合も、必ず AskUserQuestion で「この解釈で合っていますか？」と確認を取ること。** ゴール解釈のズレが最も後工程に影響する。

ユーザーの確認を取ってから Step 1.5 へ。

---

## Step 1.5: ストーリーラインを先出しする

**主張・根拠・つなぎを原典の情報量に応じて必要なだけ示す（最小3行・上限なし）。全体の「話の流れ」に合意してから詳細設計に入る。**

> 元文書が厚い場合は全論点を列挙する。AI側で先回りして「3〜5行に圧縮」しない。後工程で削るのはユーザーに委ねる。

### ストーリーラインの形式（WHAT/WHY/SO WHAT/WHAT NOW フレーム）

全体を以下の4層構造で言語化する：

```
■ WHAT（事実・現状）
  〇〇という状況が生じている

■ WHY（理由・原因）
  その原因/背景は△△にある

■ SO WHAT（含意・解決策）
  だから□□を提案する

■ WHAT NOW（アクション）
  まず〜をXX年度中に実施してほしい
```

> 「これを読めば話の骨格がわかる」レベルで十分。原典が厚いときは骨格も厚くする。
> Accenture のフレームに基づく。「読み手が何を判断すべきか」が明確になる。

### 情報密度の方針（デフォルト: 多め）

**このスキルのデフォルトは「元コンテキスト保存・スライド多め」。**

- 原典（参照データ・議事録・既存資料）に書かれた論点は**原則すべて拾う**
- AI側で「要点3つに絞る」など先回り圧縮は行わない
- スライド数が増えることを許容する（上限は設けない）
- 冗長な箇所は**ユーザーが出力後に削除**する前提で設計する

**例外**: ユーザーが明示的に「短くまとめて」「X枚以内」と指示した場合のみ圧縮する。

### ストーリーラインのチェック軸

- **一貫性**: ゴール（誰に・何を判断してもらうか）に向かっているか
- **論理の飛び**: 「なぜ？」と突っ込まれる箇所がないか
- **聴衆目線**: 聴衆が「Yes/No」を判断するのに必要な情報が揃っているか

### よくある構成パターン（例示）

| 構成タイプ | ストーリーラインの骨格 |
| ---------- | ---------------------- |
| 課題解決型 | 問題 → 原因 → 解決策 → 効果 → CTA |
| 投資承認型 | 背景/機会 → 現状の限界 → 提案 → ROI/リスク → 判断お願い |
| 経営報告型 | 結論先出し → 根拠 → 詳細 → 今後の方針 |
| プロジェクト提案型 | 背景 → 課題 → ゴール → 手段 → ロードマップ → 体制 |

**ストーリーラインが通らなければStep 1に戻ってゴールを修正する。**

### 提示の手順（必須）

1. **採用した構成パターンと選択理由を1行で明示する**（例:「投資承認型を採用。GMに判断を求める資料のため」）
2. ストーリーラインを提示する
3. **AskUserQuestion で確認を取る。「OK」を受け取るまで Step 2 に進まない**
   - 「OK、このまま進める」
   - 「順番を変えたい」→ どのパートか深掘り質問を追加する
   - 「論点を足したい / 削りたい」→ どのパートか深掘り質問を追加する

ユーザーの確認を取ってから Step 2 へ。

---

## Step 2: 各ページの主張とサポート論点を並べる

**合意したストーリーラインをベースに、各スライドの主張文（=スライドタイトル）と【メッセージライン】を展開する。中身はまだ書かない。**

原典の論点は原則すべてスライドに割り当てる。1スライドに収まらなければ分割（1a/1b）する。AI側で「要点3つに絞る」など先回り圧縮は行わない。

### メッセージラインとは何か（最重要）

**メッセージラインは、そのスライドを見た人が「覚えるべきポイント」。** 主張タイトルよりも、より具体的で記憶に残る文。

**主メッセージは1つに保つが、補足メッセージを2〜3点追記してよい**（情報密度保存のため）。原典の論点を無理に1文に畳み込まない。

| タイトル（主張文）                    | メッセージライン（最重要ポイント）                                     |
| ------------------------------------- | ------------------------------------------------------ |
| 現行システムは2027年のEOLまでに移行が必須 | 移行しなければ2027年以降、業務が完全に停止するリスク |
| 認証基盤の統合により年間1.2億円のコスト削減が可能 | 統合により、運用人員を現在の15名から8名に削減可能 |
| AI Overviewにより公益事業のWeb流入が構造的に消失している | 結果、見込み客の初期接触経路が消失し、営業効率が50%低下 |

> Accenture の設計では「スライド構成要素で【メッセージライン】が最重要」と指摘。タイトルは「何について」だが、メッセージラインは「だからどう判断すべきか」を明示する。

| BAD（カテゴリラベル） | GOOD（主張文）                                                |
| --------------------- | ------------------------------------------------------------- |
| 現状分析              | 現行システムは2027年のEOLまでに移行が必須                     |
| 提案内容              | 認証基盤の統合により年間1.2億円のコスト削減が可能             |
| 課題                  | 申し込みのリードタイムが業界平均の3倍になっており離脱率が高い |
| 背景                  | AI Overviewにより公益事業のWeb流入が構造的に消失している      |

### ストーリーチェック

並べた主張文を上から順に読んで、話が通じるか確認する。

```
P1：〇〇である（課題の認識）
P2：原因は△△にある（Why）
P3：解決策として□□を提案する（What）
P4：効果・根拠（So what）
P5：ネクストアクション（Call to action）
```

**ストーリーが通らなければ、順番や中身をユーザーと一緒に入れ替える。**

### 典型的な構成パターン

| パターン                                        | 適用場面               |
| ----------------------------------------------- | ---------------------- |
| 表紙→課題→現状分析→提案→効果→ネクストアクション | 外部向け提案書         |
| 表紙→サマリー→詳細→付録                         | 経営報告・意思決定資料 |
| 表紙→アジェンダ→各議題→まとめ・決定事項         | 会議体資料             |
| 表紙→背景→課題→解決策→ロードマップ→体制         | 大型プロジェクト提案   |

**Step 2 出力後は必ず一度止まる。AskUserQuestion を使い「このスライド構成で Step 3（ボディ設計）に進んでよいですか？」と確認してから Step 3 へ進む。確認なしに Step 3 に自動進行しない。**

ユーザーの確認を取ってから Step 3 へ。

### 「だから何？」の追加

データを載せるだけじゃなく、「だからどうすべきか」の結論を各スライドに1行足す。

| NG（データだけ）     | OK（だから何？付き）                                              |
| -------------------- | ----------------------------------------------------------------- |
| 市場規模は3000億円   | 市場規模3000億円。今参入しないと2年後には競合に取られる           |
| 現行システムは老朽化 | 現行システムは2027年EOL。移行しなければ業務停止リスク             |
| 顧客満足度が低下     | 満足度が前年比15%低下。解約率上昇の先行指標である                 |
| 申し込み件数が増加   | 申し込みが月次120%成長。現行体制では3ヶ月後にキャパオーバーになる |

**「だから何？」は人間の仕事。** AIから提案するが、最終判断はユーザーに委ねる。

### ボディ要素の設計

各スライドに入れるべき要素を具体化し、**図解方式を明記する**。
**同時に、そのボディがメッセージラインをサポートするか確認する。**

| 要素タイプ            | 適用場面                   | 図解方式 | メッセージラインとの関係 |
| --------------------- | -------------------------- | -------- | ---- |
| 数値・データ          | 課題の定量化・効果の根拠   | — | 「2027年までに移行が必須」を支える数値（EOL年月等）|
| 比較表（2列）         | 現状 vs 提案・選択肢の整理 | parts:addComparisonColumns | 現状と提案の差分がメッセージラインを実証 |
| フロー図（単純・3-5段）| 直線的なプロセス・手順     | parts:addFlowHorizontal/Vertical | フロー内の各ステップがメッセージラインの根拠 |
| フロー図（複雑）      | 分岐・合流・スイムレーン   | diagram:フロー | 複雑性がメッセージラインの必要性を示す |
| マトリクス (2x2)      | 優先度・ポジショニング     | parts:addMatrix2x2 | 位置付けがメッセージラインを支える |
| タイムライン（4-7）   | 計画・ロードマップ         | parts:addTimeline | スケジュール視点でメッセージラインの実現性を示す |
| ピラミッド（3-5段）   | 階層概念・優先順位         | parts:addPyramid | 階層構造がメッセージラインの背景を説明 |
| ステップ（3-5）       | シェブロン型の工程          | parts:addStepProcess | 工程プロセスがメッセージラインの実装方法を示す |
| Before/After          | 改善効果の可視化           | parts:addComparisonColumns | ビジュアルにメッセージラインの効果を実証 |
| 体制図・組織図        | 階層構造のある人員配置     | diagram:組織図 | 体制がメッセージラインの実現主体を示す |
| アーキテクチャ図      | システム構成・ネットワーク | diagram:アーキ | 技術構成がメッセージラインの根拠 |
| 箇条書き（4項目以内） | 要点の列挙                 | — | 各項目がメッセージラインを補強する根拠 |

### 図解方式の判定ルール（上から順にチェック）

1. **図解不要**（テーブル・箇条書き・数値のみ）→ `—`
2. **以下に該当 → `diagram:{種別}`**
   - ツリー構造（組織図・体制図）
   - 分岐・合流のあるフロー / 双方向・循環フロー（PDCA等）
   - スイムレーン / ネットワーク図 / ER図
   - 3カラム以上の比較 / 3x3以上のマトリクス
   - parts.js の制約（ステップ数・カラム数）を超える場合
3. **上記に該当せず parts.js の制約内 → `parts:{関数名}`**

**迷ったら `diagram` に倒す。** parts.js の制約を超えると生成時にレイアウトが崩れるため。

**図解は原典に図解化可能な構造（フロー・階層・比較・時系列）がある場合のみ採用する。** 原典がテキスト中心の場合、無理に図解へ畳み込まず、テキスト保持を優先する。枚数ノルマ（最低3枚）は設けない。

ユーザーの確認を取ってから出力へ。

## 出力: 設計テンプレ（Markdown）

完成した設計テンプレをMarkdownで出力し、ファイルに保存する。

**保存先**: `outputs/docs/YYYY-MM-DD_{タイトル}_設計テンプレ.md`

### 出力フォーマット

```markdown
# 設計テンプレ: {タイトル}

## ゴール

{誰に・何を判断してもらうか 1行}

## 聴衆・状況

{聴衆の属性・場面}

## ストーリーライン（WHAT/WHY/SO WHAT/WHAT NOW）

■ WHAT（事実）: 

■ WHY（理由）:

■ SO WHAT（解決策）:

■ WHAT NOW（アクション）:

## スライド構成

| No   | タイトル（主張文） | メッセージライン（主＋補足） | ボディ要素 | 図解方式 | 必須/任意 |
| ---- | ---- | ---- | ---- | ---- | ---- |
| 表紙 | {タイトル} | — | タイトル・サブタイトル・日付・所属 | — | 必須 |
| 1    | {主張文} | {主メッセージ+補足2〜3点} | {データ・図解・表} | {parts:関数名 / diagram:種別 / —} | 必須/任意 |
| 2    | {主張文} | {主メッセージ+補足2〜3点} | {データ・図解・表} | {parts:関数名 / diagram:種別 / —} | 必須/任意 |

...

| N | ネクストアクション: {内容} | — | アクション一覧・担当・期限 | — | 必須 |

**必須/任意の判定基準:**

- **必須**: 原典に直接記述がある / ゴール達成に不可欠 / 意思決定の根拠になる
- **任意**: AIが補完した論点 / 深掘りや背景説明 / 削除してもストーリーが通る

ユーザーが出力後に [任意] スライドを削除して最終版を仕上げることを想定している。

## 参照データ

- {利用可能なデータ・資料のリスト}

## スライド構成要素チェックリスト（Accenture 設計基準）

各スライドが以下を満たしているか確認：

- [ ] **タイトル**: 主張文で、「〇〇について」ではなく「〇〇である」と言い切っているか
- [ ] **メッセージライン**: 主メッセージ1つ＋補足2〜3点の形で、聞き手が覚えるべきポイントが記載されているか
- [ ] **ボディ**: 表・グラフ・概念図・テキストのいずれかの形式で具体化されているか
- [ ] **補足文**: メッセージラインの理由や根拠が書かれているか
- [ ] **出所**: データを使う場合、出典が明記されているか
- [ ] **ページ番号**: 各ページに番号が付いているか

## pptx-creator への引き渡しメモ

- テーマ候補: {corporate / modern / dads などの推奨}
- スライド枚数: {N}枚
- 図解スライド: {フロー/マトリクス/比較 など計画した図解}
- diagram スライド数: {N}枚（pptx-diagram で事前生成が必要）
- 特記事項: {制約・注意点}
```

## ネクストアクション

設計テンプレ完成後、以下を案内する:

1. **このまま pptx-creator に渡す** → 「pptx-creatorスキルを使って、この設計テンプレでPPTXを生成して」
2. **さらに設計を磨く** → Step 2/3 に戻ってストーリーを調整
3. **Notionに保存する** → notion-exportスキルで出力

> 設計テンプレは一度作れば組織の武器になる。
> 同じ目的の資料なら、次回からこの設計テンプレを起点にして「埋める」だけになる。
