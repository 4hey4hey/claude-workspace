# PPT 統合スキル（設計 → 承認 → 生成）

## フロー概要

```
Phase 1: 設計テンプレを作る（人間の仕事）
    ↓
[承認ゲート] この構成でよいか確認 → 修正 → 承認
    ↓
Phase 2: PPTXを生成する（AIの仕事）
```

**「作らせるな、埋めさせろ。」**
設計は人間が決める。AIは決まった構成に「埋める」だけ。

---

## Phase 1: 設計テンプレを作る

### Step 1: ゴールを1行で確定する

ユーザーに「この資料で、誰に、何を判断してもらいたいか」を確認する。
ゴールが明確でない場合は AskUserQuestion で必ず聞く。

> 例: 「部門長に、認証基盤刷新への投資承認を得る」

ゴールが確定したら Step 2 へ。

---

### Step 2: 各ページの主張だけ先に並べる

中身はまだ書かない。**主張（=スライドタイトル）だけ**を並べる。

**主張文の原則:**

| BAD（カテゴリラベル） | GOOD（主張文）                                                |
| --------------------- | ------------------------------------------------------------- |
| 現状分析              | 現行システムは2027年のEOLまでに移行が必須                     |
| 提案内容              | 認証基盤の統合により年間1.2億円のコスト削減が可能             |
| 課題                  | 申し込みのリードタイムが業界平均の3倍になっており離脱率が高い |

**ストーリーチェック:** タイトルだけを上から順に読んで、話が通じるか確認する。
通らなければ、ユーザーと一緒に順番や中身を入れ替える。

ユーザーの確認を取ってから Step 3 へ。

---

### Step 3: 「だから何？」とボディ要素を書き足す

各スライドに「だからどうすべきか」の結論1行と、入れるべき要素を追加する。

| NG（データだけ）     | OK（だから何？付き）                                    |
| -------------------- | ------------------------------------------------------- |
| 市場規模は3000億円   | 市場規模3000億円。今参入しないと2年後には競合に取られる |
| 現行システムは老朽化 | 現行システムは2027年EOL。移行しなければ業務停止リスク   |

**「だから何？」は人間の仕事。** AIから提案するが、最終判断はユーザーに委ねる。

**ボディ要素の候補:** 数値・データ、比較表、フロー図、マトリクス(2x2)、タイムライン、Before/After、箇条書き（4項目以内）

**図解方式の判定（各スライドに明記）:**

| 図解方式 | 条件 | 後工程 |
|----------|------|--------|
| `parts:{関数名}` | フラット構造・要素5個以内（横フロー・縦フロー・2x2・比較・タイムライン・ピラミッド・ステップ） | pptx-creator が直接生成 |
| `diagram:{種別}` | 階層・分岐・自由配置・ノード10超（体制図・組織図・スイムレーン・アーキ図） | pptx-diagram で事前に PNG を生成 |
| `—` | 図解なし（テーブル・箇条書き・数値） | 通常パーツで対応 |

**迷ったら `diagram` に倒す。** parts.js の制約を超えると生成時にレイアウトが崩れるため。

10枚以上のデッキでは、図解スライドを最低3枚計画する。

ユーザーの確認を取ってから承認ゲートへ。

---

## 承認ゲート（必須）

**設計テンプレを以下の形式で提示し、ユーザーの承認を待つ。**

```
# 設計テンプレ: {タイトル}

## ゴール
{1行}

## スライド構成
| No | タイトル（主張文） | だから何？ | ボディ要素 | 図解方式 |
|----|----------------|----------|-----------|---------|
| 表紙 | {タイトル} | — | タイトル・日付・所属 | — |
| 1 | {主張文} | {結論} | {要素} | {parts:関数名 / diagram:種別 / —} |
...

## テーマ候補
{corporate / modern / dads など}
```

**提示後、必ず確認：**

> 「この構成でPPTXを生成してよいですか？修正があれば指定してください。」

- **修正あり** → Step 2 または Step 3 に戻って修正を反映してから再提示
- **OK** → Phase 2 へ進む
- **設計テンプレを保存してほしい** → `outputs/docs/YYYY-MM-DD_{タイトル}_設計テンプレ.md` に保存してから Phase 2 へ

**承認なしに Phase 2 に進まない。**

---

## Phase 2: PPTXを生成する

### Phase 2a: diagram スライドの事前生成（該当スライドがある場合のみ）

設計テンプレの「図解方式」列に `diagram:*` があるスライドについて:

1. `skills/pptx-diagram/SKILL.md` の手順に従い draw.io XML を生成
2. VS Code でプレビュー → ユーザー確認（「OK」を待つ）
3. PNG 変換 → `assets/diagrams/{name}.png`
4. 全 diagram スライドの PNG が揃ったら Phase 2b へ

`diagram:*` がない場合は Phase 2a をスキップして Phase 2b へ直接進む。

### Phase 2b: PPTX 全体生成

承認を受けたら `skills/pptx-creator/SKILL.md` を読み込み、以下のルールで実行する:

- **設計テンプレは Phase 1 で完成済み** → Step 1（設計テンプレ作成）をスキップ
- **Step 2（テーマ・フォント選択）から開始**
- 主張文・「だから何？」・ボディ要素は承認済み内容をそのまま使う
- **ユーザーの承認なしに構成・タイトル・メッセージ方向性を変更しない**
- **`slide.background` は全スライド必ず `{ color: 'FFFFFF' }` のみ**（表紙・まとめスライドも例外なし）
- Step 3（生成）→ Step 4（QA）まで実施して完了

---

## 出力

- **Phase 1 出力**: 設計テンプレ（画面表示 + 任意でMD保存）
- **Phase 2 出力**:
  - スクリプト: `scripts/create_<name>.js`
  - PPTXファイル: `outputs/pptx/<name>.pptx`
- 完成後、スライド数・構成の概要と設計テンプレを最終確認として再掲する
