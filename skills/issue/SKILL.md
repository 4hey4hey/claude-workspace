# Issue — 壁打ち＋多視点構造化オーケストレーター

## 概要

このスキルは**2つのモード**を持ち、どちらも3フェーズで処理する：

### 課題構造化モード（デフォルト）
1. **Extract** — ユーザーのモヤモヤを引き出す（壁打ち）
2. **Multi-Perspective Review** — 4視点のサブエージェントが並列でレビュー
3. **Cross-Cut** — 視点間のトレードオフと合意点を統合

### ドキュメントレビューモード
1. **Doc Read** — ドキュメントの主張・前提・論理構造を読み解く
2. **Multi-Perspective Review** — 同じ4視点でドキュメントをレビュー
3. **Cross-Cut** — 視点間のトレードオフと合意点を統合

Phase 2・Phase 3 は両モードで共通。Phase 1 だけがモードによって分岐する。

## モード判定

ユーザーの入力から自動判定する：

| 入力パターン | モード |
|---|---|
| モヤモヤ・課題感・壁打ちしたい | 課題構造化 |
| ドキュメント / ファイルパスを提示 + レビュー系の依頼 | ドキュメントレビュー |
| 「書いたものを見て」「提案書をチェック」「論理を確認」 | ドキュメントレビュー |
| 判断がつかない場合 | ユーザーに確認する |

## なぜ多視点か

課題もドキュメントも、見る角度によって見え方が変わる。
自分の立場（SM/マネージャー）からだけ見ると、経営の視点や顧客への影響を見落とす。
4つの視点を強制的に通すことで、死角を潰し、論点の優先順位が明確になる。

ドキュメントレビューでも同様に：
- 経営視点で「投資判断の根拠は十分か」
- 開発視点で「技術的な実現性の裏付けはあるか」
- 顧客視点で「ユーザー価値の説明は説得力があるか」
- ガバナンス視点で「リスクの見落としはないか」
を同時にチェックすることで、ドキュメントの説得力と網羅性を高める。

## フェーズ構成

### Phase 1: Extract（課題構造化モード）
→ agents/01-extractor.md を読み込んで実行
→ ユーザーのラフ入力からモヤモヤを引き出す
→ 課題の輪郭が見えるまでループ
→ 完了条件を満たしたら Phase 2 へ

### Phase 1R: Doc Read（ドキュメントレビューモード）
→ agents/01-doc-reader.md を読み込んで実行
→ ドキュメントの主張・前提・論理構造を抽出する
→ ユーザーに読み取り内容を確認（最大2ターン）
→ 完了条件を満たしたら Phase 2 へ

### Phase 2: Multi-Perspective Review（4視点レビュー）
→ Phase 1 の抽出結果を入力として、4つのサブエージェントを**並列で**起動する
→ 各エージェントは agents/02-perspective-{name}.md を読み込んで実行

| エージェント | ファイル | 視点の問い |
|---|---|---|
| 🏢 経営 | 02-perspective-business.md | 事業戦略・ROI・リソース配分と整合しているか？ |
| 💻 開発 | 02-perspective-dev.md | 技術的に何が起きていて、何が制約か？ |
| 👤 顧客 | 02-perspective-customer.md | ユーザー価値・体験にどう影響するか？ |
| 🛡️ ガバナンス | 02-perspective-governance.md | リスク・コンプライアンス上の懸念は何か？ |

各エージェントは統一フォーマット（評価・Good・懸念・提言）で出力する。

#### ドキュメントレビューモード時の読み替え

レビューモードでは、各視点エージェントの評価レンズとラベルを以下のように読み替える。
エージェントファイル自体は共通だが、起動時にこの読み替え指示をコンテキストとして渡すこと。

**評価レンズの切り替え：**
- 課題構造化モード →「この課題は各視点からどう見えるか」
- ドキュメントレビューモード →「このドキュメントの主張・根拠は各視点から見て説得力があるか」

**出力ラベルの切り替え：**

| 項目 | 課題構造化モード | ドキュメントレビューモード |
|---|---|---|
| Good | 評価できる点 | 説得力のある点 |
| 懸念 | リスク・課題 | 弱い点・不足 |
| 提言 | 改善の方向性 | 補強の方向性 |

**各視点の問いの切り替え：**

| 視点 | 課題構造化 | ドキュメントレビュー |
|---|---|---|
| 🏢 経営 | この課題は事業戦略・ROIと整合しているか？ | 投資判断の根拠は十分か？ROIの説明は説得力があるか？ |
| 💻 開発 | 技術的に何が起きていて、何が制約か？ | 技術的実現性の裏付けはあるか？プロセス設計は妥当か？ |
| 👤 顧客 | ユーザー価値・体験にどう影響するか？ | ユーザー価値の説明は具体的で説得力があるか？ |
| 🛡️ ガバナンス | リスク・コンプライアンス上の懸念は何か？ | リスクの見落としはないか？統制設計は十分か？ |

### Phase 3: Cross-Cut（クロスカット統合）
→ agents/03-crosscut.md を読み込んで実行
→ 4視点の結果を横断分析し、以下を出力：
  - 視点間のトレードオフ
  - 全視点で合意できるポイント
  - 全視点で懸念があるポイント
  - 総合判定と論点整理

## フェーズ遷移ルール

### Phase 1 → Phase 2 への遷移条件（課題構造化モード）
以下の2つが**両方**揃ったとき Phase 2 に遷移する：
1. **何が起きているか**（課題の事象が1つ以上具体的に出ている）
2. **なぜ気になっているか**（影響・懸念・違和感の理由が言語化されている）

揃っていない場合は問い返す。問い返しは1ターン最大3問まで。

### Phase 1R → Phase 2 への遷移条件（ドキュメントレビューモード）
以下の3つが揃ったとき Phase 2 に遷移する：
1. **ドキュメントの主張**が明確に抽出できている
2. **前提・仮定**が特定できている（暗黙のものを含む）
3. ユーザーが読み取り内容を**確認済み**

### Phase 2 → Phase 3 への遷移
4つのサブエージェントがすべて出力を返したら自動で Phase 3 に入る。

### Phase 3 完了後
ユーザーに統合結果を提示し、フィードバックを受ける：
- 「OK」「スッキリした」等 → 最終出力として確定
- 修正指示 → 該当箇所を修正して再提示
- 「これを上に伝えたい」等 → ai-cos への接続を案内（ai-cos が利用可能な場合。なければ issue-to-stakeholder）

## フェーズ遷移時のユーザーへの通知

**課題構造化モード：**
- Phase 1 → 2: 「輪郭が見えてきた。4つの視点でレビューするね。」
- Phase 2 → 3: 「4視点のレビューが出た。クロスカットで統合する。」

**ドキュメントレビューモード：**
- Phase 1R → 2: 「ドキュメントの構造を把握した。4つの視点でレビューするね。」
- Phase 2 → 3: 「4視点のレビューが出た。クロスカットで統合する。」

冗長な説明は不要。フェーズが変わったことだけ伝える。

## 視点の情報量に応じた重み付け

Phase 1 の抽出結果の情報量によって、各視点の確信度が変わる。
情報が十分にない視点では、評価の冒頭に「⚠️ 情報限定的」と付記し、推測ベースであることを明示する。

## 出力形式

- **課題構造化モード** → `templates/output-structured.md` のテンプレートに従う
- **ドキュメントレビューモード** → `templates/output-review.md` のテンプレートに従う
