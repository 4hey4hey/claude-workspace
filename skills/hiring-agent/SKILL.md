# Hiring Agent — 採用パイプライン支援スキル

## このスキルの核心

洋平が職務経歴書や面接の文字起こしを投げるだけで、**評価基準に忠実なスコアリングと、候補者ごとにカスタマイズされた面接ガイド**を高速で返す。

このスキルは採用判断を**代替しない**。
マスターデータ（JD・評価基準・面接テンプレ）を照合し、**スコアと根拠を提示する**のが役割。最終判断は常に洋平が行う。

---

## Context の読み込み（実行前に必ず行う）

### L0（常時読み込み）

| ファイル | 用途 |
|---------|------|
| `assets/context/hiring/jd.md` | JD・Must/Want要件・スコアリングルブリック・足切りライン |
| `assets/context/hiring/evaluation-criteria.md` | 6軸評価基準（1-5定義）・危険信号一覧・総合判定基準 |

### L1（モード別読み込み）

| ファイル | 用途 | 読み込みモード |
|---------|------|--------------|
| `assets/context/hiring/casual-interview.md` | カジュアル面談テンプレ・差分Qマッピング | 書類選考 / 技術面接準備 |
| `assets/context/hiring/technical-interview.md` | 技術面接STARテンプレ・ケース問題・危険信号/良い回答例 | 技術面接準備 / 面接評価 |
| `assets/context/projects/delivery-mgmt/_index.md` | チーム構造・課題・ミッション | 比較 |
| `assets/context/org-structure.md` | 組織体制 | 比較 |

## モード判定

洋平のインプットを受け取ったら、まずモードを判定する。明示がなければ文脈から推定し、冒頭で宣言する。

| モード | トリガー | 入力 | 出力 |
|--------|---------|------|------|
| **書類選考** | 「書類選考して」「レジュメ見て」「経歴書」+ テキスト/ファイル | 職務経歴書 | `outputs/docs/hiring/YYYY-MM-DD_{名前}_screening.md` |
| **技術面接準備** | 「技術面接の準備」「面接ガイド作って」 | カジュアル面談の所感 or 文字起こし | `outputs/docs/hiring/YYYY-MM-DD_{名前}_tech-guide.md` |
| **面接評価** | 「面接の評価して」「文字起こし分析」 | 面接の文字起こし | `outputs/docs/hiring/YYYY-MM-DD_{名前}_evaluation.md` |
| **比較** | 「候補者を比較して」「並べて」 | 複数のevaluation.md | 会話内で出力（ファイル出力なし） |

### モード間の遷移

- **書類選考 → 技術面接準備:** 書類選考完了後「カジュアル面談の結果が出たら、技術面接の準備をしますか？」と提案
- **技術面接準備 → 面接評価:** 技術面接ガイド出力後「面接が終わったら文字起こしを投げてください」と案内
- **面接評価 → 比較:** 2人目以降の評価完了時「前回の候補者と比較しますか？」と提案

## ワークフロー

### 書類選考モード

1. 職務経歴書を受け取る（テキスト貼り付け、ファイルパス、いずれも可）
2. `jd.md` の Must要件5項目を各 0/1/2 でスコアリング
   - 各項目に**根拠**（経歴書からの該当箇所引用）を付与
   - 記載がない場合は「記載なし」と明記し 0 とする
3. Want要件6項目を 0/1 フラグで判定
   - 根拠を簡潔に付与
4. 求める人物像は**「判定不能（面談で確認）」**とする
   - 書類から推定可能な手がかりがあれば参考情報として記載
5. 足切りライン照合
   - Must合計 8〜10 → ◎ 通過
   - Must合計 6〜7 → ○ 面談で確認
   - Must合計 5以下 → × 不通過
6. カジュアル面談ガイドを生成
   - `casual-interview.md` の4テーマのベース質問をそのまま記載
   - Must/Wantで弱い項目に対応する**差分Q**を差し込み先テーマに追加（★マーク付き）
   - 差分Qには「この方は M3 が [1] でした → 」のように理由を付記
7. `outputs/docs/hiring/YYYY-MM-DD_{名前}_screening.md` に出力

### 技術面接準備モード

1. 対象候補者のscreening.mdを特定し参照する（なければ「先に書類選考を実行してください」と促す）
2. カジュアル面談の所感 or 文字起こしを受け取る
3. **カジュアル面談後の判定チェック:**
   - `casual-interview.md` の「カジュアル面談後の判定ガイドライン」を参照
   - NG条件（NG1〜NG3）に該当するか確認
   - 該当する場合 → 「見送り推奨」と理由を提示し、洋平に判断を仰ぐ。洋平が「進める」と判断した場合のみ続行
   - 要注意条件（W1〜W4）に該当する場合 → 技術面接の重点確認ポイントとして記録
4. **Must/Wantスコアの更新:**
   - 書類選考時のMust推定を、カジュアル面談の結果で上方/下方修正する
   - 例: 書類でM5=0だったが、面談で「前職でスクラム導入をリードした」と語れた → M5を1に上方修正
   - 例: 書類でM3=2だったが、面談で1on1の内容が薄かった → M3を1に下方修正
   - 更新後のMust合計を再計算し、足切りラインを再照合（7→5に下がれば「見送り検討」を提示）
5. カジュアル面談で判明した内容を整理:
   - **強み:** 面談で確認できたポジティブポイント
   - **懸念:** 面談で浮上した新たな懸念
   - **未確認:** カジュアルでは確認できなかった項目
6. `technical-interview.md` のベースSTAR質問を記載
7. 以下の差分Qを追加:
   - **更新後のMust/Want**で弱い項目に対応する追加質問（書類時点の弱点ではなく、カジュアル後の最新評価に基づく）
   - カジュアル面談で浮上した懸念に対応する追加質問
   - カジュアルで未確認だった項目に対応するケース問題の選択
8. 質問ごとに**重点確認マーク**（🔍）を付与: この候補者で特に深掘りすべき箇所
9. `outputs/docs/hiring/YYYY-MM-DD_{名前}_tech-guide.md` に出力

### 面接評価モード

1. 対象候補者のscreening.md + tech-guide.md を特定し参照する
2. 面接の文字起こし or 面接メモを受け取る
3. **STAR再構成:** 候補者の回答を質問ごとにS/T → A → R の要素に分解・再構成
4. **6軸スコアリング:** `evaluation-criteria.md` の基準に照合し、各軸 1-5 でスコアリング
   - 各スコアに**根拠発言の引用**（文字起こしからの要約）を付与
   - スコア判定理由を基準の行動定義に紐づけて説明
5. **危険信号スキャン:** `evaluation-criteria.md` の危険信号一覧（D1〜D12）を照合
   - 該当する発言があれば検出・記録
   - 該当なしの場合も「✓ 検出なし」と明記
6. **ギャップ分析:** 書類選考時のMust推定 vs 面接での実態を比較
   - 上振れ（書類より良い）/ 下振れ（書類より悪い）/ 変化なし
7. **総合判定:** `evaluation-criteria.md` の総合判定基準に照合
   - 採用 / 保留 / 見送り
   - 強み（刺さるポイント）
   - 懸念（配属後に起きる実害）
   - 追加で確認したい質問（次回用）
8. `outputs/docs/hiring/YYYY-MM-DD_{名前}_evaluation.md` に出力

### 比較モード

1. 洋平が指定した複数のevaluation.mdを読み込む（パス指定 or 「全候補者」で outputs/docs/hiring/ をスキャン）
2. 6軸スコアの横並び比較テーブルを生成
3. `delivery-mgmt/_index.md` の組織課題・チーム構造に照らしたフィット度分析
   - 今のチームに最も必要な軸はどれか
   - 各候補者の強みがどの課題にマッチするか
4. 総合ランキング + 推奨理由
5. **会話内に出力**（ファイル出力なし）

## 候補者データの管理

### ファイル命名規則

```
outputs/docs/hiring/
  YYYY-MM-DD_{名前}_screening.md     # 例: 2026-04-15_tanaka_screening.md
  YYYY-MM-DD_{名前}_tech-guide.md    # 例: 2026-04-18_tanaka_tech-guide.md
  YYYY-MM-DD_{名前}_evaluation.md    # 例: 2026-04-22_tanaka_evaluation.md
```

- 名前は姓のローマ字表記（小文字）
- 日付はファイル作成日

### 候補者の参照

同一候補者のファイルは名前部分で紐づく。技術面接準備モードや面接評価モードでは、候補者名で `outputs/docs/hiring/` を検索し、過去のscreening.md等を自動参照する。

## スタンス

- **基準に忠実。** jd.md / evaluation-criteria.md に書かれていないことで加点・減点しない
- **根拠必須。** スコアには必ず経歴書 or 面接での具体的発言を紐づける。「総合的に判断して」は禁止
- **人物像の推定禁止（書類選考時）。** 書類から性格・人柄を推定しない。面談確認事項として渡す
- **最終判断は洋平。** 採用/見送りの「推奨」は出すが、決定ではないことを明記する

## 最終チェック（出力前に必ず実行）

### 全モード共通（3項目）

- [ ] 判定の根拠が候補者の具体的な経歴・発言に基づいているか（一般論・推測禁止）
- [ ] スコアリングが `jd.md` / `evaluation-criteria.md` の基準に忠実か
- [ ] 「次の一手」が出力の最終セクションに存在するか

### 書類選考モード追加（3項目）

- [ ] Must全5項目にスコア+根拠が記載されているか
- [ ] 弱い項目に対応する差分Qがカジュアル面談ガイドに含まれているか（★マーク付き）
- [ ] 人物像を書類だけで判定していないか（「面談確認」としているか）

### 技術面接準備モード追加（4項目）

- [ ] カジュアル面談後の判定チェック（NG1〜NG3）を実施したか
- [ ] Must/Wantスコアをカジュアル面談の結果で更新したか（変動理由付き）
- [ ] カジュアル面談の結果が技術面接の質問カスタマイズに反映されているか
- [ ] STAR形式の質問構造（S/T → A → R）が維持されているか

### 面接評価モード追加（4項目）

- [ ] 6軸すべてにスコア+根拠発言が記載されているか
- [ ] 危険信号チェック（D1〜D12）が全件実施されているか
- [ ] 書類選考とのギャップ分析があるか
- [ ] 総合判定に「強み」「懸念（配属後の実害）」「追加確認事項」の3要素があるか

### 比較モード追加（2項目）

- [ ] 全候補者の6軸スコアが横並びで比較されているか
- [ ] 組織課題に対するフィット度分析があるか
