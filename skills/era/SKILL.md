# Agent Era Review — AIエージェント時代の施策レビュースキル

## このスキルの目的

施策・Epic・企画書・議事録を受け取り、**「AIエージェントが普及した世界でこの施策は価値を持ち続けるか」**を構造的にレビューする。

良い・悪いの二元論ではなく、「どの部分が耐久性があり、どの部分が逓減リスクを持つか」を切り分けて指摘する。

## 前提知識（レビュー時に内部的に参照する判断軸）

### お客さまの行動変化：3段階モデル

AIエージェントの普及により、お客さまの行動は段階的に変化する。施策がどの段階で影響を受けるかを見極める。

| 段階 | 行動変化 | 時間軸 | mTGへの影響 |
|---|---|---|---|
| 第1段階：AIに聞く | 料金・手続き方法をAIに質問。サイトを開かない | 既に発生中 | 情報閲覧目的のアクセスが減少 |
| 第2段階：AIに操作を任せる | 開栓予約・プラン変更をAIが代理実行 | 2〜3年以内 | 手続き系の画面遷移が減少 |
| 第3段階：AIに判断を委ねる | エネルギーコスト最適化をAIに一任 | 5年以内（若年層） | 東京ガスが「選ばれる」側に |

### 5つの評価軸

すべての施策を以下の5軸で評価する。

#### 軸1：固有データ依存度
- **問い：** この施策は「東京ガスの固有データがなければ成立しないか？」
- **固有データの例：** 家庭のエネルギー消費データ、給湯器・床暖房の稼働データ、物理的な訪問点検で得た機器情報、契約情報、ガス管内の地域データ
- **判定基準：**
  - ◎ 固有データが中核（これなしでは機能しない）
  - ○ 固有データで差別化可能（なくても最低限は機能するが、質が大きく変わる）
  - △ 固有データは補助的（他社でもほぼ同じことができる）
  - × 固有データとの接点がない

#### 軸2：チャネル・KPI非依存性
- **問い：** この施策は「特定チャネル（アプリ画面）やチャネル依存のKPI（MAU/DAU）に閉じず、API/エージェント経由でも同等の価値を届けられるか？」
- **この軸が問う3つの観点：**
  - **配信チャネル** — 画面を開くことに依存していないか？ファネル（到達→閲覧→クリック→申込）がすべて画面前提の場合、第1段階の進行だけでファネル入口が細くなる
  - **成功指標** — KPIがMAU/DAUに閉じていないか？ガス・電気は毎日確認する必然性がない。「毎日使いたくなる機能」でMAUを増やす戦略はエージェント時代に成立しにくい
  - **API転用可能性** — 今作るロジック・データ・基盤は、将来エージェント向けに転用できるか？
- **判定基準：**
  - ◎ チャネル非依存、KPIもMAUに閉じない、ロジックがAPI化可能（例：使用量比較→増加検知→SW提案のロジック。APIとしても公開可能）
  - ○ 画面が主だが、ロジックはAPI化可能。KPIもMAU以外を併用している
  - △ 画面体験に強く依存し、KPIもMAU中心。ただし一部ロジックは転用可能
  - × 「アプリを開かせること」が目的で、KPI=MAU/DAU。API転用先がない（例：ゲーミフィケーションUI。画面体験そのものが価値）

#### 軸3：代替困難性
- **問い：** この施策は、AI検索・汎用AIアシスタント・専業アプリで代替されにくいか？
- **代替の類型（これらに該当すると代替困難性が低い）：**
  - AI検索に吸われる（情報提供型コンテンツ → AI OverviewでCTR 61%減）
  - 専業アプリの方が強い（家計簿→マネーフォワード、天気→Yahoo!天気、家事→TimeTree等）
  - プラットフォーマーが統合する（室温管理→Apple Home/Google Home、ゲーム→SEGA等）
  - 汎用AIアシスタントがやる（パーソナライズされていないTips、FAQ回答）
- **判定基準：**
  - 高（東京ガスの固有データ・物理接点がなければ実現不可能。代替されにくい）
  - 中（固有データで差別化可能だが、類似サービスは存在する）
  - 低（他社・AI検索・専業アプリで同等以上の体験が提供される。代替されやすい）

#### 軸4：耐久性の時間軸
- **問い：** この施策はいつまで有効か？
- **判定基準：**
  - 短期のみ（〜1年）：エージェント普及前なら効果あるが、逓減が見える
  - 中期まで（〜3年）：第1段階の影響は受けるが、第2段階までは機能する
  - 長期（5年〜）：エージェント時代でも価値の源泉が変わらない

#### 軸5：競争のモート（堀）
- **問い：** この施策は、東京ガスが「AIに選ばれる」ための競争優位を築くか？
- **背景：** Gartner予測では、API-first・ヘッドレスアーキテクチャで設計された製品が構造的な競争優位を確立する。検証可能なデータの提供能力がエージェント間取引の参加条件になる
- **判定基準：**
  - ◎ データの堀を構築する（他社が真似できないデータ資産・API基盤を作る）
  - ○ 間接的に堀に寄与する（データ収集・同意取得・認証基盤の拡充に貢献）
  - △ 堀とは無関係（顧客体験の改善だが、エージェント時代の競争力には直結しない）
  - × 堀を弱める可能性がある（汎用コンテンツに投資を分散し、固有データ投資が遅れる）

## 実行プロトコル

### Step 1：入力の受け取り

ユーザーから以下のいずれかを受け取る。
- Notion ページ（URLまたはコピーテキスト）
- Markdown文書
- Epic / PBI の説明
- 議事録・会議メモ
- 施策の口頭説明

入力が不十分な場合は、以下を聞く：
- 「この施策の主な目的は何ですか？（顧客価値 / 収益 / MAU向上 / その他）」
- 「主な成功指標は何ですか？」
- 「対象のお客さまは？」

### Step 2：施策の要約と前提の抽出

入力から以下を抽出し、ユーザーに確認する。

```
## 施策の理解

- **施策名：**（入力から推定）
- **目的：**（何を達成しようとしているか）
- **主な前提：**（この施策が成立するために必要な条件）
- **想定チャネル：**（アプリ画面 / Web / プッシュ通知 / API / その他）
- **成功指標：**（KPIとして何を見ているか）
- **東京ガス固有データの利用：**（使っている / 使えるが使っていない / 接点なし）

この理解で合っていますか？
```

### Step 3：5軸評価

5つの評価軸それぞれについて評価を行い、以下の形式で出力する。

```
## AIエージェント時代レビュー

### サマリー

| 評価軸 | 評価 | 一言 |
|---|---|---|
| 固有データ依存度 | ◎/○/△/× | ... |
| チャネル・KPI非依存性 | ◎/○/△/× | ... |
| 代替困難性 | 高/中/低 | ... |
| 耐久性の時間軸 | 短期/中期/長期 | ... |
| 競争のモート | ◎/○/△/× | ... |

### 総合判定：[A / B / C / D]

- **A：エージェント時代に強い** — 固有データが中核で、チャネル・KPIともに非依存。やるべき
- **B：コアは良いが設計に注意** — 固有データを使っているが、画面・MAU依存の部分がある。API化・KPI設計を工夫すれば長期的に価値を持つ
- **C：短期は有効だが逓減する** — 現時点では効果があるが、2〜3年で投資対効果が悪化する。やるなら撤退基準を明確に
- **D：エージェント時代に不整合** — 「開かせるための機能」か、固有データとの接点がない。再考を推奨
```

### Step 4：具体的な指摘

各軸の評価理由を詳細に説明する。特に以下を重視する。

1. **何がダメか** — 具体的にどの前提がエージェント時代に崩れるか
2. **何は良いか** — 施策の中で耐久性がある部分を見落とさない
3. **どう直せばよいか** — 判定がB/Cの場合、具体的な改善案を提示する
   - 例：「画面に表示するだけでなく、比較ロジックをAPI化して、エージェントからも呼べる設計にする」
   - 例：「MAUだけでなく、チャネル問わずの転換率（NRR）を並行して測る」
   - 例：「撤退基準に『エージェント経由のアクセス比率がX%を超えた時点で画面施策を縮小』を追加する」

### Step 5：問い返し（必要な場合）

施策の前提に曖昧な点がある場合、評価の後に問い返す。

- 「この施策で使用量データを使っていますが、このロジックはAPI経由でも提供できる設計になっていますか？」
- 「成功指標がMAUのみですが、エージェント経由の転換が増えた場合、この施策の効果をどう測りますか？」
- 「この機能は〇〇（専業アプリ）でも提供されていますが、東京ガスが提供する必然性はどこにありますか？」

## 出力形式

### デフォルト出力

Step 3のサマリーテーブル + Step 4の具体的指摘をMarkdownで出力。

### 出力先

`outputs/docs/reviews/agent-era/YYYY-MM-DD_施策名.md` に保存。

---

## やらないこと

- 施策の全否定。「エージェント時代にはすべて無意味」という極論は取らない
- 短期効果の無視。「今は効果がある」ことと「3年後に逓減する」ことは両立する
- 技術的な実装可否の判断。API化が「できるか」ではなく「設計として考慮されているか」を見る
- 組織政治の判断。「これを指摘すると角が立つ」は考えない。事実ベースで指摘する

## 注意事項

- レビュー対象の施策を頭ごなしに否定しない。検討者の意図と文脈を尊重した上で、見落とされているリスクを指摘する
- 「エージェント時代」は不確実性を含む。時間軸の幅（短期/中期/長期）を常に明示し、「いつから」「どの確率で」影響が出るかを添える
- 判定がAの施策に対しても、「さらに強化するなら」の視点で提案する
