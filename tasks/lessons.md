## [2026-04-04] npm install は必ず事前に許可を得ること

**何が起きたか**: `pptx-diagram` スキル実装中に `@mermaid-js/mermaid-cli` を確認なしにインストールした。

**なぜ問題か**: パッケージのインストールはシステム状態を変える非可逆操作。サプライチェーンリスクもあり、ユーザーが意図しない変更になりえる。

**ルール**:

- `npm install` / `pip install` / `brew install` 等のパッケージインストールは **必ず実行前に明示的な許可を求める**
- 「このスキルに必要なので〇〇をインストールする必要があります。よろしいですか？」と確認してから実行する
- 許可なくインストールした場合は即座に報告し謝罪する

## [2026-04-04] 表紙スライドの紺色背景問題（再発防止）

**何が起きたか**: `addCoverTitle` が `bandH=2.85`（スライドの51%）の紺色バンドを描画し、ユーザーが何度指摘しても「背景が紺色」になり続けた。

**なぜ再発したか（根本原因3層）**:
1. **parts.js の共有関数が大面積バンドをデフォルトで描画** — スクリプト側で `slide.background = "FFFFFF"` を設定しても、addShape が上書きする
2. **ppt-rules.md の「部分的なバンドは可」という表現が曖昧** — AI が bandH=2.85 を「部分的」と解釈してしまう
3. **テーマ設計が暗いバンド前提** — `theme.TP = "FFFFFF"` がタイトル色に使われ、紺バンド上の白文字レイアウトを誘導

**修正内容**:
- `addCoverTitle` のバンドを 0.8 インチに縮小、タイトルを白エリアにダーク色で配置
- ppt-rules.md に「バンド高さ1.0インチ以下」「表紙で白文字禁止」を明記

**ルール**:
- 表紙の `addShape` カラーバンドは **高さ1.0インチ以下**（スライド高さの18%以下）
- 表紙のテキストは **ダーク系**（`theme.P`, `theme.DT`）。`theme.TP`（白）は表紙タイトルに使わない
- 「部分的なバンド」= **あくまで装飾ライン程度**。スライドの20%以上を色で覆うのは「全面カバー」と同じ

## [2026-04-05] AI出力の認知負荷問題 — 「次の一手」必須化

**何が起きたか**: AIの抽象的な出力（計画・方針・構造化結果）は情報量が多くフラットで、読んだ後に「で、何から始める？」が分からない状態が発生する。

**なぜ問題か**: 出力を読んだだけで仕事した気になるが行動に移れない。脳の処理能力とAIの出力速度のギャップ。

**対策（実施済み）**:
1. **CLAUDE.md** に「出力の認知負荷ルール」セクションを追加（全スキル横断）
2. **ai-cos SKILL.md** に「行動変換フェーズ」を追加（構造化・翻訳・壁打ち全モード共通）
3. ai-cos の最終チェックリストに3項目追加

**ルール**:
- 抽象出力には必ず「次の一手」（最大3項目、優先順、誰が/何を/いつまでに）をつける
- 壁打ち中は「次に決めるべきこと」を1問で提示する
- 「検討する」「整理する」のような抽象動詞は禁止。具体的な行動に落とす

## [2026-04-05] outputs/docs/ ファイル蓄積問題 — 棚卸しモード追加

**何が起きたか**: AI との壁打ち・構造化の結果ファイルが outputs/docs/ に蓄積し（9日間で20ファイル）、同一トピックの重複ファイルが増えて「どれが最新か」「全体像が掴めない」状態になった。

**なぜ問題か**: ファイルが増えるとコンテキスト負荷が上がり、過去の議論が活かされず同じ議論を繰り返すリスクがある。

**対策（実施済み）**:
1. ai-cos に「棚卸し（tidy）」モードを追加（4番目のモード）
2. directory-structure.md に `status: "superseded"` の frontmatter 規約を追加
3. CLAUDE.md のスキル選択ルールに棚卸しルーティングを追加

**設計判断**:
- 新スキルではなく ai-cos のモード追加にした（context 読み込み・L1→L2→L3 を再利用するため）
- ファイルは削除せず superseded マークのみ（Notion outbox の source 参照保護）
- 統合結果を assets/context/ に昇格する道筋を用意（outputs は一時的、context が正規）

## [2026-04-05] SKILL.mdに確認指示を書くだけではAIは連続出力する — 強制停止設計が必要

**何が起きたか**: pptx-planner に Step 1.5（ストーリーライン確認）を追加したが、実際のテストで Step 2 と Step 3 を確認なしに連続出力してしまった。SKILL.md に「ユーザーの確認を取ってから Step 3 へ」と書いてあったにもかかわらず。また Step 1（ゴール確定）でもドキュメントから自動読み取りして AskUserQuestion なしで進んだ。

**なぜ問題か**: 段階確認はスキルの設計価値そのもの（「先に合意してから詳細に入る」）。一気出力するとユーザーが修正を入れる機会がなく、スキルの目的（段階合意）が達成されない。

**根本原因**: 「確認を取ってから進む」という文言だけでは不十分。AIは「コンテキストが揃っているなら出力できる」と判断して連続出力する傾向がある。

**対策（実施済み）**:
1. Step 1 末尾: ドキュメント読み取り時も AskUserQuestion 必須を明記
2. Step 1.5: 「提示の手順（必須）」セクションを追加（パターン明示→提示→AskUserQuestion→OK待ち）
3. Step 2 末尾: 「確認なしに Step 3 に自動進行しない」を明記

**ルール**:
- SKILL.md に段階確認を設計するときは「AskUserQuestion を使う」「自動進行禁止」を明示的に書く
- 「確認を取ってから進む」だけでは AIは自動進行する。**ツール（AskUserQuestion）と停止条件を具体的に指定**する
- 前レッスン([2026-04-05] Skill Optimizer 初回監査)の「書いてあるのに守られない」と同じ構造問題。**物理的な手順と停止トリガーを指定**しないと守られない



**何が起きたか**: Skill Optimizer（メタエージェント）で ai-cos の直近出力5件を eval 基準でスコアリングした結果、「次の一手」が**全5件で欠落**（軸1スコア: 1.0）していた。SKILL.md に「出力の最後に必ず実行」と明記されているにもかかわらず。

**なぜ問題か**: ルールを書くだけでは守られない。「書いてあるのに守られない」は、ルールの書き方自体に構造問題がある。

**根本原因（3層）**:
1. **構造的競合**: 長い出力の末尾にモード固有のクロージング（アクションテーブル等）を書いた後、「次の一手」を追加し忘れる
2. **テンプレートとの不整合**: 「次の一手」をどの位置に置くかが不明確
3. **チェックリスト肥大**: 20項目のフラットなリストでは、関係ない項目の読み飛ばし中に関係ある項目も見落とす

**対策（実施済み）**:
1. 「次の一手」を出力の**物理的な最終セクション**として強制する構造ルールを追加
2. チェックリストをモード別に分離（全モード共通3項目 + モード別追加）
3. 翻訳プロトコルにパターンB（計画承認型）を追加
4. 構造化モードにタスクタイプ分岐（分析型 vs 設計・計画型）を追加

**ルール**:
- SKILL.mdにルールを書くときは「物理的な出力位置」まで指定する（「必ず付ける」だけでは不十分）
- チェックリストは10項目以上をフラットに並べない。モード別に分離する
- 改善の効果は eval 基準で定量検証する（感覚ではなくスコアで）

**横展開**:
- team-pulse に「判断ガイド」セクションを追加（ai-cos の「次の一手」を「判断非代替」原則に合わせた翻訳形式）
- team-pulse eval.md に軸4（行動導線性）を追加

## [2026-04-06] Claude セッション起動時に CronCreate 登録する仕組みが必要

**何が起きたか**: GitHub への定期 auto-commit を CronCreate で設定したが、毎回セッション起動時に **手動で登録コマンドを実行する必要がある**（セッション横断で継続しない）。

**なぜ問題か**: セッション起動のたびに 1 ステップ増える。手順書に頼ると、開発者が忘れる可能性が高い。

**対策（実施済み）**:
1. `.claude/commands/periodic-commit.md` ファイルを作成（開始手順を文書化）
2. memory に本レッスンを追記（セッション横断での思い出し）
3. 今後の改善案を検討予定

**改善検討中**:
- Option A: `.claude/hooks/` に `OnSessionStart` フック追加（自動実行）
- Option B: `CLAUDE.md` の「セッション開始チェックリスト」に記載
- Option C: Copilot の方で `notion-sync` 定期実行フック化（24時間×365日必須な場合）

**ルール（暫定）**:
- セッション開始時に `.claude/commands/periodic-commit.md` を開き、「CronCreate コマンド」セクションを実行する
- 毎日 19:00 に自動 commit/push が稼働していることを GitHub Web UI で週 1 回確認
- GitHub PAT トークンの有効期限（90日）を 1 ヶ月単位でチェック
  - トークン更新時は `.claude/settings.json` の `git config` も更新（手動）

**横展開**:
- 定期自動スクリプトが複数出てきたら（Notion Sync、スキル改善监査等）、共通の「セッション起動スクリプト」セクションを `.claude/refs/` に切り出す検討
