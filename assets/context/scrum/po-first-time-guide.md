# 初めてPOになった人のための実践ガイド

> **これは何：** スクラムガイド2020の原則をベースに、mTG（myTOKYOGAS）のスクラム運営の知見を織り込んだ、新任PO向けの実践ガイド。「どこから始めるか」「何をやるのか」「何に気をつけるか」の3つの問いに答える。

**3行サマリー**

* **Part 1（どこから始めるか）：** 最初の1週間チェックリスト、30/60/90日ロードマップ、施策Readyの確認方法

* **Part 2（何をやるのか）：** POのアカウンタビリティ、4象限の業務構造、PdMとの分界線、セレモニー別の動き方、日々の過ごし方、バックログ管理の要点

* **Part 3（何に気をつけるか）：** 10の典型的な失敗パターン、能力マップによる成長の道しるべ、困ったときの相談先

***

# Part 1: どこから始めるか

## 1. 最初の1週間チェックリスト

PO就任直後に「まずこれをやる」具体的なアクション。上から順に進めていく。

### 現状把握（Day 1-2）

* [ ] プロダクトバックログの現状を確認する（どこにあるか、何が入っているか、誰が管理していたか）

* [ ] プロダクトゴールを確認する（PdMに聞く：今期のゴールは何か）

* [ ] 直近2-3スプリントのレビュー資料を読んで、チームが何を作っているか把握する

* [ ] チームのベロシティ・スプリント期間・Doneの定義を確認する

### 関係構築（Day 2-3）

* [ ] PdMと1on1を設定し、PdM/POの分界を合意する（[§5 PdMとPOの境界](#5-pdmとpoの境界)を元に会話する）

* [ ] SMと1on1を設定し、チームの状態・セレモニーの進め方・現在の課題を把握する

* [ ] 開発チーム全員と顔合わせし、「POとして何を大事にしたいか」を伝える

* [ ] ステークホルダーのリストを作る（誰の要望を聞く必要があるか）

### 実務準備（Day 4-5）

* [ ] バックログの上位5件について「なぜ作るのか」を自分の言葉で説明できる状態にする

* [ ] 次のリファインメントの事前準備をする（PBIの背景・受入条件の素案）

* [ ] 福田さん（Naikist PO）の動き方を1スプリント観察する時間を確保する

* [ ] 自分の判断で決めてよい範囲と、エスカレーションすべき範囲をSM・マネージャーと合意する

### 環境整備

* [ ] バックログツールへのアクセス権を確認する

* [ ] 関連するSlackチャンネル・定例会議への参加設定を完了する

* [ ] 過去のスプリントゴールと実績の記録を見つける

## 2. 30/60/90日ロードマップ

### 30日目標：サポートを受けながら動ける

セレモニーでPOとしての基本動作ができている状態。

| やること                           | 到達イメージ                           |
| ------------------------------ | -------------------------------- |
| リファインメント・プランニング・レビューにPOとして参加する | SMや先輩POのサポートを受けながら、自分でPBIの説明ができる |
| バックログの上位10件の優先順位の根拠を説明できる      | 「なぜこの順番か」を聞かれて自分の言葉で答えられる        |
| 受入条件を自分で3件以上書く                 | チームが「Doneかどうか」を判断できる精度で書ける       |
| PdMとの定期すり合わせリズムを確立する           | 週1回程度の頻度で、ロードマップとバックログの整合を確認している |

**能力マップとの対応：** 全6領域のLv.1に着手している段階。

### 60日目標：並走のもと主導的に動ける

セレモニーを主導し、優先順位の判断根拠を自分で組み立てられる状態。

| やること                       | 到達イメージ                                                                                |
| -------------------------- | ------------------------------------------------------------------------------------- |
| スプリントゴールを自分で設定する           | PdMのロードマップを踏まえて、スプリント単位の価値を自分の言葉で定義できる                                                |
| 「やらない」判断を1件以上行い、根拠をSHに説明する | 「ROIの観点でこの判断をした」と説明でき、相手が納得する                                                         |
| ベロシティを理解し、リリース計画の初版を作る     | 「この機能はあと何スプリントで届けられるか」の見通しを立てられる                                                      |
| チームとの信頼関係を意識的に築く           | 「何を・なぜ作るか」を伝え、「どう作るか」はチームに委ねている。情報をオープンにしている                                          |
| 優先順位判断に「満足度ギャップ」の視点を使ってみる  | 「ユーザーが今体験していることと、本当はこう体験したいと思っていることの差」が大きいPBIほど、解決したときのインパクトが大きい。この視点で優先順位の根拠を組み立てられる |

**能力マップとの対応：** 価値駆動の意思決定力・PBI翻訳力・スクラム運営力のLv.1を安定させている段階。

### 90日目標：自律的に判断できる範囲が明確

日常のPO判断を自律的に行い、エスカレーション基準が自分の中で確立している状態。

| やること                      | 到達イメージ                        |
| ------------------------- | ----------------------------- |
| バックログ全体の優先順位を自分の判断で管理する   | 上位の並び替えを自分で行い、根拠を説明できる        |
| SHからの要望に対して、ROI観点で整理・判断する | 受け入れ/見送り/保留の判断を自分で下せる         |
| 最初の90日の振り返りをSM・PdMと実施する   | 何ができるようになったか、次の課題は何かを言語化できている |
| エスカレーション基準が自分の中にある        | 「ここまでは自分で決める、ここからは相談する」が明確    |

**能力マップとの対応：** Lv.1が安定し、一部Lv.2に着手。Lv.2以降は半年〜1年かけて段階的に伸ばしていく。

## 3. 施策Readyの確認

POが打席に立つ前に「弾が込められているか」を確認する習慣をつける。

### 施策Readyとは

POがPBIに分解して開発パイプラインに投入できる状態。逆に言えば、以下の前提が揃っていない施策にPOが打席を使うと、開発途中で「そもそもの前提が違った」が発覚し、打席が無駄になる。

### 6つの前提条件チェックリスト

| # | 前提          | 確認ポイント                 | 揃っていないとどうなるか          |
| - | ----------- | ---------------------- | --------------------- |
| 1 | **as-is整理** | 現行の業務・システムの実態が把握されているか | 機能重複・仕様矛盾が開発中に発覚→作り直し |
| 2 | **to-be定義** | 目指すべき姿が関係者間で合意されているか   | ゴールが曖昧→期待のズレ→手戻り      |
| 3 | **ビジネス試算**  | ビジネスの投資対効果の根拠があるか      | 途中でROI疑問→凍結リスク        |
| 4 | **業務フロー調整** | リリース後のオペレーションが設計されているか | リリース直前にオペ不整合→延期 |
| 5 | **既存契約・業者整理** | 移行や統廃合の制約が把握されているか | 二重運用コスト・計画頓挫 |
| 6 | **SH間合意** | 関係する部署間で方向性が合意されているか | 「聞いていない」→やり直し・信頼毀損 |

### 揃っていない場合どうするか

**POが自分で全部揃えようとしない。** 上流に戻す。

施策の前提整理は、PdM・事業部門・企画担当の仕事。POの守備範囲は「前提が揃った施策をPBIに分解し、開発パイプラインに載せること」。前提が揃っていない施策を無理に開発に入れると、POが上流の仕事を抱え込み、バックログ管理と開発チームへの対応が手薄になる。

# Part 2: 何をやるのか

## 4. POとは何をする人か — 責任と日常業務の全体像

### なぜPOが必要なのか

スクラムガイドの定義に入る前に、そもそもなぜPOという役割が存在するのかを理解しておく。

**mTGが置かれている前提：**
- BtoCという変化の速い市場では、どんなに緻密に計画しても仮説は外れる可能性がある
- エンジニアのリソースは潤沢ではなく少数精鋭。仮説が外れたときに立て直す余力が小さい
- レガシーシステムという負債を抱えたまま事業を前に進めており、リプレースにも保守にもコストがかかる

この前提に立つと、**「大きなプロジェクトで一発で当てにいく」開発は、外れたときの影響が大きく、戦略ではなくギャンブルになる。** 成果を出すには「小さく試して、当たる確率を上げる」ことが必要。

#### 成果 = 打席数 × 打率

事業の成果は、**「どれだけ多くの施策を試せたか（打席数）」×「そのうちどれだけ成果につながったか（打率）」** で決まる。

```
成果 ＝ 打席数（試行回数） × 打率（成功確度）

打席数を増やす ← PBIを小さく分割し、リードタイムを短くし、試行回数を増やす
打率を上げる   ← 前提が揃った施策だけに打席を使い、無駄打ちを減らす
```

**POは両方に関わる：**

| 成果の要素 | POの判断がどう効くか | 具体的な行動 |
|---|---|---|
| **打席数を増やす** | PBIを小さく分割し、1つずつ完了させて試行回数を増やす | スコープ分割、リードタイム短縮、「まず最小限で出す」判断 |
| **打率を上げる** | 前提が揃っていない施策に打席を使わない。確度の高い施策に集中する | 「やらない」判断、施策Readyの確認、ROI評価 |

だからPOの判断は「バックログを管理する事務作業」ではない。**事業の成功確率を左右する戦略的判断**。

---

### スクラムガイドが定めるPOの責任

> **プロダクトバックログから得られるプロダクトの価値を最大化することの結果に責任を持つ。**

これがスクラムガイド2020のPO定義。「打席数×打率」のフレームで読み直すと：

**「限られたチームのリソース（打席）を、最も成果につながる施策に集中させ、事業の成功確率を最大化する人」**

この定義を「マーケット」「価値の最大化（ROI）」「責任」の3つに分解して説明する。

***

### 「マーケット」= PO以外の全て

PMとPOの最大の違いは「何を見て判断するか」にある。

* **PM（プロジェクトマネージャー）が見るもの = スケジュール。** 「期限内に終わらせる」のが仕事

* **PO（プロダクトオーナー）が見るもの = マーケット。** 「マーケットのニーズを満たして価値を生む」のが仕事

「マーケット」は社外の顧客だけを指さない。**PO以外の全てがマーケット**。

| 区分          | 対象                      | ニーズの例                                 |
| ----------- | ----------------------- | ------------------------------------- |
| **外向き（社外）** | 顕在顧客、エンドユーザー、まだ見えていない顧客 | 「ガス料金をアプリで見たい」「手続きをオンラインで済ませたい」       |
| **内向き（社内）** | PdM、部長、他部署、開発チーム、マネージャー | 「KPIをあげたい」「開発効率をあげたい」「プロダクトの方向性を知りたい」 |
| **潜在マーケット** | まだ認識していない顧客や市場          | 認識していないので、自ら見つけにいく必要がある               |

なぜ社内もマーケットなのか？ 開発チームが「何を作ればいいかわからない」状態では、どんなに優れた戦略も実装に落ちない。POが社内のニーズに応えることは、間接的にプロダクトの価値を高めている。

***

### 「価値の最大化（ROI）」= 開発チームのリソースから最大の価値を引き出す

POにとっての「投資」とは、**開発チームのリソース（時間・スキル・集中力）を使うこと**。この投資は有限で、取り戻せない。1スプリントは2週間しかなく、チームの人数も決まっている。だからこそ、**限られたリソースをどのPBIにどの順番で使えば最も大きな価値が生まれるかを判断すること**がPOの最も重要な責務になる。

#### なぜこの判断が難しいか

投資（チームのリソースを使うこと）はPOがコントロールできる。しかしリターン（実際に価値が生まれるかどうか）は、やってみるまでわからない。

|  | Investment（投資）          | Return（リターン）            |
| ------ | ----------------------- | ----------------------- |
| **性質** | コントロールできる               | 予測不可能                   |
| **例**  | 「エンジニア3名 × 2スプリントを投入する」 | 「問い合わせが月500件減る……かもしれない」 |

リターンが確定しないからこそ、バックログの優先順位は「仮説」になる。「この順番で作れば最も価値が出るはず」という仮説を立て、スプリントごとに検証し、学びに応じて判断を変えていく。プロダクトゴールも同様に仮説。正しいゴールを最初から設定できると思わないこと。

#### 具体例で考える：mTGで2スプリントを何に使うか

チームに次の2スプリントで取り組めるキャパシティがあるとする。候補が3つある。

| 候補               | 投資（チームの工数） | 期待リターン                       | 不確実性             |
| ---------------- | ---------- | ---------------------------- | ---------------- |
| A. 料金明細のアプリ表示    | 2スプリント     | コールセンター問い合わせ月1,200件→600件（推定） | 低い（類似機能の実績あり）    |
| B. ガス機器の通報サービス連携 | 2スプリント     | 新サービスの立ち上げ。利用者数は未知           | 高い（業務フロー未整理）     |
| C. マイページのUI刷新    | 2スプリント     | ユーザー満足度向上（定量効果は不明）           | 中（デザインはあるが効果未検証） |

**PM的な発想：** 「ロードマップに載っているBを予定通りやる」

**PO的な発想：** 「今この2スプリントをどこに使えば、チームのリソースから最大の価値を引き出せるか？」

POならこう考える：

* **候補A:** リターンが具体的（月600件削減）で不確実性が低い。投資に対して確度の高い成果が見込める

* **候補B:** リターンのポテンシャルは大きいが、Ready条件を満たしていない（業務フロー未整理）。このまま着手すると、チームのリソースが「前提整理」に消える。Readyが揃うまで待つべき

* **候補C:** 「満足度向上」は魅力的だが、何をもって成功とするか定義がない。まず小さく試す（1スプリント分のスパイク）方が安全

結論：**Aに2スプリント投資し、Bは上流の整理を先に進めてもらい、Cは次のスプリントで小さく検証する。** これがPOのROI判断。

#### 「小さく作って試す」ことで無駄な投資を防ぐ

ROI判断で最も効果が大きいのは、**「作らずに済ませる」判断**。小さく作って試せば、効果の薄い施策を早い段階で止められる。

**例：** Bizから「ユーザーの行動履歴・属性・時間帯・天気・位置情報を全て分析するAIで、最適なプッシュ通知を配信したい」という要望が来た。フルスペックで作ると1年かかる。

POの判断：「まず時間帯セグメント3パターン（朝・昼・夜）だけでA/Bテストしましょう。効果が確認できたら、次に属性軸を追加しましょう」 → 2ヶ月で検証できる。

もし時間帯だけで十分な効果が出れば、**10ヶ月分の開発を回避できる**。仮に効果が出なければ、2ヶ月で「この方向は違う」と学べる。どちらに転んでも、1年かけてフルスペックを作るより賢い。これが「打率を上げる」判断。

---

### 「責任を持つ」= 最終判断を下し、その結果を引き受ける

スクラムガイドは「プロダクトオーナーは1人の人間であり、委員会ではない」と明記している。POが持つ「責任」は2つの側面がある。

1. **判断を下す責任：** バックログの並び順、「やる/やらない」、スコープの調整——これらの判断はPOが最終的に下す。委員会で多数決ではなく、1人の人間が判断する
2. **結果を引き受ける責任：** 判断の結果うまくいかなかったとき、「チームの実装が悪い」とは言わない。自分の判断（優先順位、スコープ、受入条件）が適切だったかを振り返る

ただし「1人で全部やる」という意味ではない。PBIの作成をチームに手伝ってもらうのは問題ない。ただし「なぜこの順番なのか」「なぜこれはやらないのか」を聞かれたとき、答えられるのはPOでなければならない。

**POは組織から権限を委譲されている。** SGEPは「プロダクトオーナーが成功するためには、すべてのステークホルダーがその意思決定を尊重すべきである」と明記している。POの判断は個人の意見ではなく、組織から委譲された権限に基づく判断。新任POは「自分にそんな権限があるのか」と不安になるかもしれないが、バックログの並び順を決める最終権限はPOにある。SHがバックログを変更したいときは、POを説得する必要がある。

***

### POの日常業務 — 4つの象限

ここまでの「マーケット」「ROI」「責任」を日常業務に落とすと、4つの象限に分かれる。

|      | 管理系（判断する） | 協力系（人と動く）    |
| ---------- | --------- | ------------ |
| **プロダクト軸** | バックログ管理   | チームとの協力      |
| **事業軸**    | 経済性の管理    | ステークホルダーとの協力 |

**バックログ管理（プロダクト × 管理）：** PBIの内容を明確に表現し、優先順位を決め、受入条件を定義し、チームからの質問に答える。最初から全PBIの詳細化は不要。**上位が明確であれば開発は回る。**

**経済性の管理（事業 × 管理）：** 複数の要求のトレードオフを評価し、最もROIが高い順に並べる。「やらない」判断をする。経済的な観点を開発チームにも共有する（透明性がチームの自律を育む）。

**チームとの協力（プロダクト × 協力）：** 「何を・なぜ作るか」を提示し、「どう作るか」はチームに委ねる。開発チームが自律的に動けるよう、マーケットの課題・優先順位の判断基準・スコープ範囲を示す。

**ステークホルダーとの協力（事業 × 協力）：** スプリントレビュー等の場でフィードバックを引き出す。ステークホルダーの要求は尊重するが、鵜呑みにしない。全ての要求に対して「これはROI向上につながるか？」を問う。POはSH・チーム・PdMとの意見の相違が頻繁に起こる役割であり、それは正常な状態。衝突を恐れて全員の顔色をうかがうと、パターン4（SH要望全受入）に陥る。

要求をそのまま受け取らず、**問い返しで本質を掘る**習慣をつける。

| SH/チームの発言 | POの問い返し | なぜ効くか |
|---|---|---|
| 「この機能が欲しい」 | 「解決したい問題は何ですか？」 | ソリューションではなく問題にフォーカスを戻す |
| 「これは不可能だ」 | 「可能にするにはどうしたらいいですか？」 | 制約を特定し、代替案の探索に切り替える |
| 「緊急です、すぐやって」 | 「何もしなかったらどうなりますか？」 | 本当の緊急度を見極める |
| 「説明不要、私が欲しいのだから」 | 「チームに背景を伝えるために、教えていただけますか？」 | 権限ではなく透明性の文脈で理由を引き出す |

SHは一様ではない。**影響力と関心度で4分類**し、対応の濃淡をつける。

|     | 影響力が大きい           | 影響力が小さい             |
| --------- | ----------------- | ------------------- |
| **関心が高い** | 緊密に管理する（人数は最小限に）  | 定期的に情報提供し、ニーズを先読みする |
| **関心が低い** | 満足させておく（関連情報のみ伝達） | 最小限の距離感で見ておく        |

### バックログ秘書にならない — POの多面的な役割

POは「バックログ秘書ではなく戦略的リーダーである」（SGEP）。バックログ管理に1日の大半を使っているなら、それは警告サイン。PBI作成はチームと協力して進め、POは「なぜこの順番か」「何をやらないか」の判断と、マーケット・SHとの対話に時間を使う。

SGEPはPOが状況に応じて使い分ける「顔」を列挙している。

| 顔 | いつ使うか |
|---|---|
| **ビジョナリー** | プロダクトの方向性をチームやSHに語るとき |
| **顧客代表** | ユーザーの声をチームに届けるとき |
| **協力者** | チームやSHと一緒に解決策を考えるとき |
| **実験者** | 仮説を立てて小さく試すとき |
| **意思決定者** | 優先順位や「やらない」を判断するとき |

新任POは全部を同時にやろうとしなくてよい。最初の30日は「意思決定者」と「協力者」に集中し、60日以降に「実験者」や「顧客代表」の動きを広げていく。

### プロダクトのナラティブを持つ

SGEPは「一貫性のあるプロダクトのナラティブを構築し、効果的に伝えること」もPOの責任としている。ナラティブとは「このプロダクトは誰の何を解決し、どこへ向かっているか」を一貫したストーリーとして語れること。

新任POが最初にやるべきことは、PdMに聞いて以下の3つを自分の言葉で説明できる状態にすること。

1. **誰のためのプロダクトか**（ターゲットユーザー）
2. **何を解決しているか**（中核的な課題）
3. **これからどこへ向かうか**（プロダクトゴール）

これが言えれば、SHへの説明、チームへの「なぜ作るのか」の共有、レビューでのフィードバック依頼が全てつながる。

### POが日々自問すべき3つの問い

1. **「このPBIに投資して、誰のどんな課題が解決されるか？」** — リターンが説明できないPBIは、バックログの上位にある理由がない
2. **「同じリソースを別のPBIに使った方がリターンが大きいのでは？」** — 今バックログの1位にあるPBIが本当に1位か、常に問い直す
3. **「そもそもこのPBIは必要か？ やらない方がROIが高いのでは？」** — 「やらない」ことで、より価値の高いPBIにリソースを集中できる。やらない判断もROI向上

### POの行動原則

日々の判断で拠り所にする5つの原則。「何をやるか」は前のセクションで述べた。ここでは「どう振る舞うか」を示す。

**1. 素早く判断し、結果から学ぶ**

完全な情報を待たない。判断は十分に時間をかけても間違える可能性がある。やり直し可能な判断は素早く下し、スプリントレビューで結果を検査して次の判断精度を上げる。「打席数を増やす」とは、判断の速度を上げて試行回数を稼ぐこと。優柔不断な癖をつけるくらいなら、間違った意思決定をするほうがマシ。

**2. 成功はチームの功績、失敗は自分の責任**

うまくいったらチームを褒める。うまくいかなかったら「チームの実装が悪い」とは言わない。自分の判断（優先順位、スコープ、受入条件）が適切だったかを振り返る。他責にしない姿勢がチームの信頼を生み、チームの信頼がPOの判断を支える。

**3. 情報をオープンにする**

SHから得た情報、マーケットの変化、ビジネス状況をチームに加工せずに共有し、一緒に考える。透明性がチームの自律を育てる。情報を絞るとチームは「言われたことだけやる」存在になり、創意工夫が失われる。

**4. 健康を管理する**

POは判断の質が命。持続可能なペースを自分で守る。疲弊した状態での意思決定は、チーム全体のリスクになる。POにはやることが多い。だからこそ、自分の判断力を維持するための健康管理は業務の一部。

**5. やめる判断をする**

「やらない」だけでなく「もうやめる」も重要な判断。途中まで作ったものを捨てるのは心理的に辛いが、サンクコスト思考に陥ると、効果が出ていない施策にリソースを注ぎ続けることになる。うまくいっていない施策を止めることは、チームのリソースを守る行為。

## 5. PdMとPOの境界

mTGで最も混乱しやすいのがこの分界線。ここが曖昧だと、守備範囲を間違えて疲弊する。

### 分界の原則 — 3つの問いと3つの担い手

プロダクト開発には3つの問いがある。それぞれに担い手が決まっている。

| 問い             | 担い手     | 具体的にやること                          |
| -------------- | ------- | --------------------------------- |
| **何を・なぜ作るか**   | **PdM** | 仮説立案、顧客検証、ロードマップ策定。プロダクト戦略の最終判断   |
| **どこまで・いつ作るか** | **PO**  | バックログ管理、受入条件定義、優先順位決定。スプリント内の最終判断 |
| **どう作るか**      | **チーム** | 技術選択、設計、実装方法。開発チームの自律的な判断         |

この3つの境界を守ることが、それぞれが力を発揮するための前提になる。

| **POの役割**      | レビューでは2つの観点を確認する。**(1) Output検査:** 作ったものが受入条件を満たしているか。**(2) Outcome検査:** 作ったものがユーザーの行動変化やビジネス成果につながっているか（または検証する計画があるか）。その内容を踏まえてステークホルダーからフィードバックを引き出し、得た情報をバックログに反映する |

**パターン1: PdMがスプリント内の仕様判断に介入してくる**

* 「どこまで・いつ作るか」はPOの守備範囲であることを伝える。「スプリント内のスコープ判断はPOが担当しています。『何を・なぜ作るか』のレベルについてはぜひ相談させてください」

* PdMの関心は「価値が届いているか」。レビューで成果を見せることで、スプリント内の判断への介入は自然に減る

* 優先順位やスプリント内のスコープ判断は基準・ルールを整備してPdM含めステークホルダーとすり合わせる

**パターン2: POがロードマップレベルの判断を勝手にする**

* 「何を・なぜ作るか」の大きな方針変更はPdMの判断。POが独断でロードマップを変えない

* 開発中に「この方向性は違うのでは」と感じたら、POからPdMにエスカレーション

**パターン3: POが施策の前提整理まで抱え込む**

* as-is調査・業務フロー設計・SH間調整はPOの守備範囲外。[§3 施策Ready](#3-施策readyの確認)を確認し、揃っていなければ上流に戻す

### 迷ったときの判断フロー

```
この判断は「何を・なぜ作るか」に関わるか？
  → Yes → PdMに相談
  → No → 「どこまで・いつ作るか」に関わるか？
           → Yes → POが判断する
           → No → 「どう作るか」ならチームの判断
```

## 6. セレモニー別ガイド

各スクラムイベントで「何を準備し、何をし、何を持ち帰るか」の具体ガイド。

### リファインメント（必須）

| 項目             | 内容                                                                                               |
| -------------- | ------------------------------------------------------------------------------------------------ |
| **事前準備**       | 1〜3スプリント先の候補のPBIについて「なぜ作るのか」「誰のどんな課題を解決するのか」を説明できる状態にしておく                                        |
| **POの役割**      | PBIの背景・価値を説明し、チームが見積もれる状態にする。チームからの質問に答える。受入条件（Acceptance Criteria）とアウトカム基準（何の指標をもって成功とするか）を合意する |
| **持ち帰り**       | チームの質問から見えた曖昧な点のリスト。次回リファインメントまでに解消する                                                            |
| **やってはいけないこと** | 実装方法を指定する。見積もりの数字に口を出す                                                                           |

<br />

> **受入条件の2つの層：** 受入条件には「何をもってDoneとするか」（Acceptance Criteria = Output）と「なぜこれを作るのか・どんな行動変化を期待するか」（Outcome Criteria = Outcome）の2層がある。「要件通りに作ったのに成果が出ない」を防ぐために、Outcome Criteriaも合わせて定義する習慣をつける。最初はAcceptance Criteriaだけで構わない。慣れてきたら「この機能で何が変わるはず？」を1行加えてみる。

### プランニング（必須）

スプリントプランニングは3つのトピックで構成される。

**トピック1: なぜこのスプリントは価値があるのか？（スプリントゴール）**

POがまず「今回のスプリントでプロダクトの価値をどう高めたいか」を提案する。これは命令ではなく、議論の始まり。チーム全体で「このスプリントのゴール」を合意する。

よいスプリントゴールの例：
- 「購入に必要なステップ数を減らし、コンバージョンレートを改善する」
- 「サポート問い合わせのうち『使い方がわからない』系を30%減らす」
- 「デザイン変更の主要導線部分を新デザインに変更し、評価可能にする」

スプリントゴールは**スプリント中は変更しない**。もしゴールを犠牲にしてでも対応すべきことが発生したら、それはスプリントのキャンセルを検討する事態。

**トピック2: このスプリントで何ができるか？（PBI選択）**

チームが過去のパフォーマンスと今回使える時間を踏まえて、バックログからPBIを選ぶ。**選ぶのはチームであり、POが「これとこれをやって」と指定するのではない。** POは優先度と価値を伝え、チームが「この量なら完了できる」と判断する。

**トピック3: どう成し遂げるか？（作業計画）**

チームが選んだPBIをどう進めるか計画する。**これはチームだけの裁量。** POは口を出さない。

| 項目 | 内容 |
|---|---|
| **PO事前準備** | スプリントゴールの素案を用意する。バックログの上位PBIがReady状態か確認する。マーケットやビジネスの最新状況を共有できるよう準備する |
| **POの役割** | スプリントゴールを提案する（命令ではない）。PBIの優先度と価値をチームに伝える。チームの質問に答える。チームが選んだ量を尊重する |
| **持ち帰り** | チームが選んだPBIとスプリントゴールの合意内容。スプリント中にPOが判断すべきことのリスト。誰をスプリントレビューに呼ぶかの決定 |
| **やってはいけないこと** | 「全部やって」と詰め込む。チームの計画づくりに口を出す。Ready状態でないPBIを投入する（「生煮えPBIを食べると腹を壊す」） |

### デイリースクラム（任意）

| 項目             | 内容                                          |
| -------------- | ------------------------------------------- |
| **参加する場合**     | 「聞く」に徹する。進捗を把握し、仕様の疑問が出ていたらその場か直後に回答する      |
| **やってはいけないこと** | 進捗報告を求める。「あれどうなった？」と追及する。デイリーをPOへの報告会にしてしまう |

### スプリントレビュー（必須）

| 項目             | 内容                                                                                                                                                            |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **事前準備**       | ステークホルダーに参加を依頼する。レビューの目的（「タスク完了報告」ではなく「価値が届いたかの検査」）を共有しておく                                                                                                    |
| **POの役割**      | レビューの前に2つの検査を行う。**(1) Output検査:** 作ったものが受入条件を満たしているか。**(2) Outcome検査:** 作ったものがユーザーの行動変化やビジネス成果につながっているか（または検証する計画があるか）。ステークホルダーからフィードバックを引き出し、得た情報をバックログに反映する |
| **持ち帰り**       | フィードバック一覧。バックログへの反映方針。次スプリントの方向性の修正点                                                                                                                          |
| **やってはいけないこと** | 「やったこと」の羅列で終わる。ステークホルダーのフィードバックをすべて次スプリントに入れると約束する                                                                                                            |

> **Output DoneとOutcome Done　**従来のDoneの定義を「Output Done（技術的に完了したか）」と「Outcome Done（価値が届いたか）」の2層に分けている。最初のうちはOutput検査だけで精一杯でよい。慣れてきたら「この機能をリリースして、ユーザーの行動は変わったか？」を問う習慣をつける。機能をリリースすること自体を祝うのではなく、ユーザーの行動変化とビジネス成果を測定・所有することが、現代のPOに求められている。

### レトロスペクティブ（任意）

| 項目             | 内容                                    |
| -------------- | ------------------------------------- |
| **参加する場合**     | チームだけでは解決困難な課題（組織的障害・SH間の問題等）の解決を支援する |
| **やってはいけないこと** | チームのプロセス改善に干渉する。「もっと速く」と圧をかける         |

## 7. POの1スプリントの過ごし方

**POは片手間ではできない。** セレモニーだけで1スプリント（2週間）あたり最低16時間を使う。それ以外にSH対応・ユーザー接触・PBI仕込み・チームの質問対応がある。

| 活動                                | 2週間スプリントでの目安 |
| --------------------------------- | ------------ |
| セレモニー合計（プランニング・リファインメント・レビュー・レトロ） | 約16時間        |
| SHとの対話・ユーザー接触                     | 約8時間         |
| PBI仕込み・バックログ整理                    | 約8時間         |
| チームからの質問対応・受入確認                   | 約4時間         |

ジェフ・サザーランド（スクラム共同考案者）は「POは半分の時間を顧客やSHと過ごし、残りの半分をチームと過ごす」と述べている。セレモニーは全体の2-3割。残りの7-8割をどう使うかがPOの実力差になる。以下は2週間スプリントを想定した典型的な過ごし方。

### スプリント前半（Day 1-5）: 今のスプリントを支えつつ、次を仕込む

| 日           | 主な活動              | 具体的にやること                                                         |
| ----------- | ----------------- | ---------------------------------------------------------------- |
| **Day 1**   | プランニング + 着手支援     | プランニングでスプリントゴールを合意。開発着手後は「仕様の疑問にすぐ答える」状態を保つ。Slackで質問が来たら30分以内に返す |
| **Day 2-3** | 次スプリントのPBI仕込み     | 次スプリント候補のPBIを3-5件下書きする。不明点があればPdMやSHに確認を入れる                      |
| **Day 4-5** | リファインメント準備 + SH対応 | リファインメントに持ち込むPBIを最終化。SHからの要望・問い合わせを整理し、バックログへの反映要否を判断する          |

### スプリント後半（Day 6-10）: 検証と学び、次の準備

| 日           | 主な活動                  | 具体的にやること                                               |
| ----------- | --------------------- | ------------------------------------------------------ |
| **Day 6-7** | リファインメント実施 + 受入確認     | リファインメントで次スプリントのPBIをチームと詳細化。並行して今スプリントの成果物の受入確認を始める    |
| **Day 8**   | バックログ整理               | バックログ全体を見直す。上位10件の順番が正しいか再確認。「もうやらない」ものがあれば下位に移動し理由を記録 |
| **Day 9**   | レビュー準備                | SHへの参加依頼。デモシナリオの確認。「このスプリントで何が変わったか」を1-2文で言語化          |
| **Day 10**  | レビュー + レトロ + 次スプリント着手 | レビューでフィードバック収集。レトロに参加する場合はPO起因の課題を持参。翌スプリントのプランニング準備   |

### セレモニー以外の時間でPOが毎日やること

**朝（15分）：**

* バックログツールを開き、今スプリントで進行中のPBIの状態を確認する

* チームのSlack/チャットで仕様に関する質問が出ていないか確認する

* 質問があれば午前中に回答する（放置するとチームが止まる）

**日中（1-2時間）：**

* 次スプリント以降のPBIの仕込み（下書き・SH確認・情報収集）

* SHからの要望を整理し、バックログへの追加・優先順位変更を判断する

* PdMとの非同期コミュニケーション（Slackやドキュメント共有で方向性のすり合わせ）

**夕方（15分）：**

* 今日回答できなかった仕様の質問がないか確認する

* 明日やるべきことを1-3行でメモする

## 8. バックログ管理の要点

バックログはPOの「作品」であり、POの意思決定が最も可視化される成果物。ここでは要点だけを示す。PBIの書き方、分割テクニック、リファインメントの進め方、リリース計画などの詳細は\*\*[別冊: バックログ管理ガイド](po-backlog-management-guide.md)\*\*を参照。

### バックログの3つの役割

| 役割          | 内容                                  |
| ----------- | ----------------------------------- |
| **計画のツール**  | 「何を・どの順番で作るか」を表現する。リリース計画の根拠になる     |
| **透明性のツール** | チーム・SH・経営が「今何に取り組んでいて、次に何が来るか」を見られる |
| **判断の記録**   | POが「何をやると決めたか」「何をやらないと決めたか」の証跡になる   |

### 階層構造: 上位は詳細に、下位は粗く

```
上位 1-5件  【Ready】   すぐ着手できる。受入条件明確。見積もり済み
中位 6-15件 【仕込み中】 「なぜ作るか」は明確。「何を作るか」はまだ粗い
下位 16件〜  【アイデア】  タイトルと「なぜ作るか」だけ
```

「上位3件が常にReadyであること」を最低ラインにする。全PBIを詳細化しようとしない。

### PBIの最低限の書き方

最初は以下の3つだけでよい。

1. **タイトル**: 「〇〇が△△できるようにする」
2. **なぜ作るのか（Why）**: どのユーザーの、どんな課題を解決するか
3. **受入条件**: Doneかどうかを判断できる、検証可能な条件

チームにリファインメントで見せて「何が足りないか教えてほしい」と聞く。チームからの質問がPBIを育てる。

### 優先順位で迷ったときの4つの問い

1. **どちらがプロダクトゴールに近いか？** → ゴール直結を上に
2. **どちらの満足度ギャップが大きいか？** → ユーザーの「今」と「理想」の差が大きい方を上に
3. **どちらのリスクが高いか？** → 不確実性が高いものを先に（小さく試す）
4. **どちらの依存関係が多いか？** → 他チームをブロックしているものを先に

# Part 3: 何に気をつけるか

## 9. よくある失敗パターン

新任POが陥りやすい10のアンチパターン。いずれも構造的な問題として記述している。

***

### パターン1: PMになってしまう（スケジュール駆動）

**症状：** ガントチャートを引く。進捗管理が中心になる。「いつ終わるか」ばかり聞く。

**原因：** PM経験者がPOになったとき、慣れた思考パターンに戻る。あるいはPMとPOの違いが整理されていない。

**対処：**

* PMが見るのは「スケジュール」。POが見るのは「マーケット」

* 自分が毎日いちばん考えていることが「いつ終わるか」ならPM化のサイン

* 「このスプリントでユーザーにどんな価値が届くか」を自問する

***

### パターン2: マイクロマネジメント（「どう作るか」に口を出す）

**症状：** 実装方法を指定する。ベロシティに口を出す。タスクの進め方を指示する。

**原因：** チームを信頼できない。または「自分がコントロールしないと不安」。エンジニア出身のPOに多い。

**対処：**

* POの仕事は「何を・なぜ作るか」の提示。「どう作るか」はチームが決める

* 「こう作ってほしい」ではなく「こういう価値を届けたい」と伝える

- チームの自己組織化に価値を置く。やり方・ベロシティに口を出さない

***

### パターン3: チームに「発注」してしまう

**症状：** POがPBIの詳細仕様を全て書いて、チームに渡す。実現する価値よりも機能詳細・仕様の議論に時間を使っている。プランニングやレビューがPOの一方的な説明で終わる。

**原因：** パターン2（マイクロマネジメント）と似ているが角度が違う。パターン2は「どう作るか」に口を出す問題。このパターンは**POが全てを決めてチームに渡す「発注者」になってしまう**問題。チームはPOの指示を実装するだけの存在になり、創意工夫が失われる。

**なぜダメか：** スクラムチームは「ステークホルダーとのコラボレーション、検証、保守、運用、実験、研究開発など、プロダクトに関して必要となり得る全ての活動に責任を持つ」（スクラムガイド2020）。POが全てを決めると、実質的にウォーターフォールと同じ構造になる。

**対処：**

* POは「何を・なぜ作るか」と「どんな価値を届けたいか」を伝え、「何を・どう作るか」の具体化はチームとの対話で決める

* PBIは完成品ではなく「会話のきっかけ」。詳細はリファインメントでチームと一緒に育てる

* プランニングやレビューで、チームが発言する時間がPOより少ないなら危険信号

***

### パターン4: SHの要望を全て受け入れる（ROIフィルターなし）

**症状：** バックログが膨張する。優先順位がころころ変わる。「声の大きい人」の要望が上位に来る。

**東京ガスでの実例：** Krakenプロジェクトでは、TG固有のカスタマイズが多発し開発遅延の主因になった。Krakenの標準機能で始めるのではなく、カスタマイズする方向に判断が倒れ続けた構造。「標準機能で妥協する」交渉ができる人がいなかった。

**対処：**

* 全ての要望に対して「これを作ることでどのユーザーのどの課題が解決され、ROIはどうか」を問う

* ステークホルダーの要求を尊重しつつも、ROI向上につながらない要求には「今ではない」と説明する

***

### パターン5: 「やらない」判断ができない

**症状：** バックログが増える一方。古いアイテムが溜まる。全部やろうとして全部中途半端。

**対処：**

* 「やらない」はネガティブな判断ではない。最も価値のあることに集中するためのポジティブな判断

* 「やらない」と判断したら、バックログから削除するのではなく、優先順位を下げて理由を記録する

* 最初は小さな「やらない」判断から始める。慣れてきたらSH説明にチャレンジする

***

### パターン6: 上流の前提整理を自分で抱え込む

**症状：** POがas-is調査・業務フロー設計・SH間調整まで全部やっている。開発チームへの対応が手薄になる。

**原因：** 施策Readyの概念がなく、渡された施策の前提が揃っていないことに気づかないまま受け取ってしまう。

**対処：**

* [§3 施策Ready](#3-施策readyの確認)のチェックリストを使い、揃っていないものは上流に戻す

* 「POの守備範囲は『どこまで・いつ作るか』。上流の前提整理はPdM・事業部門の仕事」を自分とチームの共通認識にする

***

### パターン7: SM兼務してしまう

**症状：** POとSMを1人が兼ねる。どちらの役割も中途半端になる。

**なぜダメか：** SMとPOは利益相反の関係にある。

| 場面          | PO判断       | SM判断           | 兼務すると             |
| ----------- | ---------- | -------------- | ----------------- |
| バックログが溢れている | もっと詰め込みたい  | キャパ超過、削るべき     | 自分で自分にブレーキをかけられない |
| 割り込み要望が来た   | 事業的に受けたい   | スプリントを守りたい     | 受ける側の立場が勝ちがち      |
| 見積もりが甘い     | スコープを維持したい | リスクを可視化して調整したい | 都合の悪い情報を握りつぶせてしまう |

**対処：** POとSMは別の人が担う。人手不足で兼務せざるを得ない場合は、「今この場面ではどちらの帽子をかぶっているか」を常に意識し、矛盾が生じたらマネージャーにエスカレーションする。

***

### パターン8: PO不在の隠れコストを放置する

**症状：** PO業務をエンジニアが非公式に吸収。1人あたり日1-2時間が仕様確認・優先順位の相談に消える。権限が曖昧なまま判断し、手戻りが発生する。

**対処：**

* POが着任したら「仕様判断はPOに集約する」をチームに明示する

* 仕様の質問がエンジニア間で回っている状態を見つけたら、POが引き取る

* 「POがいることで、エンジニア全員が開発に集中できる」状態を作る

* **PO自身の不在にも備える：** 代理判断者を事前に設定し、「POにしかできない判断」の範囲を絞って他は委譲する。POが忙しすぎてスクラムイベントに出られないなら、それは「このプロダクトより重要なことがある」という意思表示。常態化しているなら異常

***

### パターン9: 機能リリースが目的になっている（Feature Factory）

**症状：** 毎スプリント機能をリリースしているが、ユーザー行動やビジネス指標が動いていない。ベロシティは安定しているのに、プロダクトの価値が上がっている実感がない。

**原因：** 「作ること」がゴールになり、「価値を届けること」が見えなくなっている。Pendo社の2019年調査では、**プロダクトの機能の80%はほとんど、またはまったく使われていない**。機能を増やすこと自体には価値がない。使われない機能を作ることは、チームのリソースを捨てているのと同じ。

**対処：**

* PBIを作成するとき「この機能で何が変わるはず？」を毎回問う

* レビューで「作ったもの」だけでなく「ユーザーの行動が変わったか」を検査する

* 変わらなかったら、なぜかを学ぶ。それ自体が価値ある情報

* ベロシティやストーリーポイントは活動指標であって、価値指標ではない

* 「成果なし × 生産性高い = ゴミの高速生産」。全員が成果（Outcome）にフォーカスする

***

### パターン10: 顧客やユーザーを見ていない

**症状：** 社内ミーティングでスケジュールが埋まっている。機能の必要性がユーザーフィードバックではなく社内の仮説に基づいている。ユーザーがプロダクトを実際に使う様子をほとんど見たことがない。

**原因：** POの時間がSH対応・社内調整に吸い取られ、最も大事な「マーケットを見る」活動が後回しになっている。あるいは「社内の偉い人が言ったこと＝ユーザーが求めていること」と混同している。

**なぜ危険か：** プロダクトの価値はユーザーの課題を解決することで生まれる。ユーザーを見ずに作った機能は、社内では「よさそう」に見えても、実際には使われない。スティーブ・ブランクの言葉：**「建物の外に出よ」**。

**対処：**

* 月に1回はユーザーがプロダクトを使う場面を観察する（問い合わせログの確認でもよい）

* 「この機能を求めているのは社内の誰か？ それともユーザーか？」を常に区別する

* スプリントレビューにユーザーやユーザーに近い立場の人を招待する

* 社内ミーティングの時間を週に1時間減らし、その分をユーザー接触に充てる

***

## 10. PO能力マップ — 成長の道しるべ

PO能力は6つの領域に分かれる。最初の90日ではLv.1の安定化を目標にする。

### レベル定義

| Lv       | 定義                        |
| -------- | ------------------------- |
| **Lv.1** | 概念を理解し、サポートを受けながら実践できる    |
| **Lv.2** | 並走のもと主導的に実践でき、判断の根拠を説明できる |
| **Lv.3** | 自律的に実践し、状況に応じて応用できる       |

### 6領域と最初に目指すLv.1の到達像

| # | 領域                 | Lv.1で到達する姿                                                   |
| - | ------------------ | ------------------------------------------------------------ |
| 1 | **価値駆動の意思決定力**     | ビジネス価値・ユーザー価値・リスク・コストを総合的に評価し、「今やるべきこと」と「やらないこと」を根拠をもって判断できる |
| 2 | **PBI翻訳力**         | ロードマップの「なぜ作るか」を理解し、「何を作るか」のレベルに分解できる。受入条件を検証可能な形で書ける         |
| 3 | **ユーザー理解・リサーチ活用力** | KPI・ユーザー行動ログを読み、課題の大きさや優先度を数値で把握できる                          |
| 4 | **スクラム運営力**        | プランニングでスプリントゴールを提示し、リファインメントでPBIの詳細化をファシリテートできる              |
| 5 | **ステークホルダー調整力**    | 自分で判断すべきこと・相談すべきこと・エスカレーションすべきことを使い分けられる                     |
| 6 | **チームとの協働力**       | バックログの透明性を保ち、「なぜこれを作るのか」をチームに伝え、オーナーシップを引き出せる                |

最初の90日でLv.1を安定させることが目標。Lv.2以降は半年〜1年かけて段階的に伸ばしていく。焦る必要はない。

> 各領域の詳細（Lv.2-3の到達像を含む）は、[PO能力マップ](../../assets/context/scrum/po-capability-map.md)を参照。

# 付録

## 付録A: スクラムガイド2020 PO該当箇所

> スクラムガイド2020「プロダクトオーナー」より抜粋

**プロダクトオーナーは、スクラムチームが生み出すプロダクトの価値を最大化することの結果に責任を持つ。** 組織、スクラムチーム、個人によって、その方法はさまざまである。

プロダクトオーナーは、効果的なプロダクトバックログ管理にも責任を持つ。たとえば、

* プロダクトゴールを策定し、明示的に伝える。

* プロダクトバックログアイテムを作成し、明確に伝える。

* プロダクトバックログアイテムを並び替える。

* プロダクトバックログに透明性があり、見える化され、理解されるようにする。

**プロダクトオーナーは上記の作業を自ら行うこともできるし、他の人にやってもらうこともできる。いずれにしても最終的な責任はプロダクトオーナーが持つ。**

プロダクトオーナーはひとりの人間であり、委員会ではない。プロダクトオーナーは、ステークホルダーの多くのニーズをプロダクトバックログで表しているかもしれない。ステークホルダーがプロダクトバックログを変更したいときは、プロダクトオーナーを説得する。

***

## 付録B: 用語集

| 用語                   | 説明                                                                     |
| -------------------- | ---------------------------------------------------------------------- |
| **PBI**              | プロダクトバックログアイテム。バックログに含まれる個々の項目                                         |
| **PBL**              | プロダクトバックログ。プロダクトに必要なものを優先順位順に並べた一覧                                     |
| **PdM**              | プロダクトマネージャー。「何を・なぜ作るか」の最終判断者                                           |
| **PO**               | プロダクトオーナー。「どこまで・いつ作るか」の最終判断者                                           |
| **SM**               | スクラムマスター。スクラムプロセスの健全性に責任を持つ                                            |
| **SH**               | ステークホルダー。スクラムチーム外のプロダクト関係者                                             |
| **スプリントゴール**         | そのスプリントで達成する価値を表す1文。「タスク完了」ではなく「価値の達成」を軸にする                            |
| **受入条件**             | PBIが「Done」であることを判断するための具体的・検証可能な条件                                     |
| **ベロシティ**            | チームが1スプリントで完了できるPBIの量（ストーリーポイント等）                                      |
| **ROI**              | Return on Investment。投資に対するリターン。POはこの最大化に責任を持つ                         |
| **施策Ready**          | 開発着手前に6つの前提条件（as-is、to-be、ビジネス試算、業務フロー、契約整理、SH合意）が揃った状態                |
| **LeSS**             | Large-Scale Scrum。複数チームで1つのプロダクトを開発するフレームワーク。mTGが採用                    |
| **リファインメント**         | PBIの詳細化・見積もりを行うイベント。POとチームが協力して実施                                      |
| **Outcome Criteria** | PBIの「なぜこれを作るのか」「どんな行動変化を期待するか」を示す基準。受入条件（Acceptance Criteria）とセットで定義する |
| **Outcome Done**     | 技術的な完了（Output Done）に加え、ユーザーの行動変化やビジネス成果が検証された状態。SGEPで導入された概念           |
| **満足度ギャップ**          | ユーザーが今体験していることと、本当はこう体験したいと思っていることの差。PBI優先順位判断のフレーム                    |
| **Feature Factory**  | チームが機能のリリース数（Output）に集中し、成果（Outcome）を見失っている状態。SGEPが警告するアンチパターン         |
| **SGEP**             | Scrum Guide Expansion Pack。スクラムガイド2020の公式コンパニオン文書群                     |
