## GM（直属上長：及川 敬仁）

### 1. 主な関心事・責任範囲

- **mTG事業全体のスピーディーな成果創出**（大橋部長からの最大の要請）
- **中長期を見据えた体制づくり**（内製化体制のゴール：施策の打席数を増やして、成果を継続的に出し続けられる状態を作ること）
- 会員増加・会員施策（KRで最重視）
- Kraken連携の着実な推進と顧客価値提供の並行
- EA/SoE（認証基盤刷新・CRM等）の全体統括リスクの把握
- 組織拡大・内製化領域の拡大（経営からのDX推進圧力）
- ステークホルダーマネジメントの強化（多部署調整が最大のボトルネックと認識）
- 外部契約・コスト管理

### 2. 報告を聞くときの思考回路

**「よし、わかった」と言う条件：**

- **結論＋代替案がセットで提示**されている報告。問題提起だけでは動かない。「じゃあどうするの？」が先回りされていると即断する
- **ファクトベース**（本人発言の直接引用、複数名の一致した見解、数字）で裏付けられた報告
- 成果とスピード感が見える報告（「スピーディーに施策が動いている」が安心材料）

**「で、どうしたいの？」と返す条件：**

- **問題提起で止まっている報告**。課題の説明だけで提案がない場合、必ずこの反応になる
- 感情的・主観的な訴えが多い報告（「現場が困っている」だけでは動かない）

**「それ大丈夫？」と心配する条件：**

- **上層部（大橋部長・西村GM）に波及するリスク**が見えた場合
- スケジュール遅延やステークホルダー調整の停滞が匂う報告
- メンバーの離職・信頼低下に関わる話題

### 3. 安心材料になるもの

- **具体的な日程・マイルストーン**（「いつまでに何が見える」が最重要）
- **数字**（会員数推移、影響件数、ROI、リソース配分の根拠）
- **複数名の一致した見解**（構造的分析であることの証明）
- **上層部にそのまま使える説明フレーム**（及川さん自身が大橋部長に翻訳して伝える必要があるため）

### 4. 地雷ワード・NGパターン

- **「できない理由」の長い説明** → 部長レイヤーの「ごちゃごちゃ言わず早くやって」を及川さんも代弁する立場にある
- **感情的・個人的な不満に見える報告** → 「それは君の個人的な感情じゃない？」と切り返される
- **Howの詳細を延々と語る** → 及川さんは短時間で結論を形成するタイプ。冒頭で印象が固まりやすい（**最初の30秒が勝負**）
- **論点が整理されていない報告** → 「網羅的に整理し論点シャープにしないと及川さんも意思決定できない」

### 5. 好む報告の粒度・形式

- **結論先出し＋短い要点整理**型。長い前置きは逆効果
- **口頭ベースで素早く**。Notionで書面化されていればさらに良い（「基本Notionで書面で済むならそれでOK」）
- 及川さんの課題・関心に沿ったものであること
- 冒頭でフレームを設定する（「排除ではなく適材適所の提案」など）
- 代替案（A案・B案）をセットで提示。問題提起だけでは動かない

| **結論から入る** | 「結論として〇〇を提案します。理由は3つ…」のフォーマット。前置きは最小限 | スピード重視。長い前置きは「時間の無駄」と感じる。結論ファーストで「できるやつ」認定される |

| **プロとしての敬意を示す** | 「高松さんの〇〇の知見を踏まえると…」「この領域は高松さんが一番詳しいと思うので…」と専門性を認める前置き | プライドが高く、自分のスキル・経験への自負が強い。「あなたの専門性を尊重している」というメッセージが信頼構築の起点 |
| **事前の頭出し** | 大きな方針変更は、正式な場の前にチャットや短い口頭で「こういう方向で考えている」と事前共有 | 「聞いていない」状態を嫌う。事前に知らされていれば、会議では建設的な議論に入れる。知らされないと「軽んじられた」と感じる |

### 4. シーン別コミュニケーション設計

#### 📌 報告・相談のとき

> **テンプレート**: 「結論は〇〇です。根拠は3点：①（数値）②（構造）③（比較）。判断いただきたいのは△△の部分です」
> 
> - 数値を必ず入れる（なければ「現時点の仮説として〜」と前置き）
> - 判断を仰ぐポイントを明確にする（全体を丸投げしない）
> - 背景説明は聞かれたら答える程度でOK

#### 📌 フィードバックを伝えるとき

> **テンプレート**: 「〇〇の成果は素晴らしかった（具体的な成果を数値で）。一方で、△△の部分はこう変えるとさらに効果が上がると思う（改善の提案形式で）」
> 
> - **絶対にやること**: まず成果を具体的に認める（プライドを満たす）
> - **改善点は「能力の問題」ではなく「環境適応」のフレームで伝える**（例：「東京ガスの合意形成文化に合わせた伝え方の工夫」）
> - **弱みの直接指摘はNG**。「さらに強みを活かすために」のフレームで
> - 1on1など**プライベートな場で伝える**（公の場でのフィードバックは避ける）

#### 📌 依頼するとき

> **テンプレート**: 「〇〇の件で高松さんの力を借りたい。ゴールは△△で、期日は□□。高松さんには××の部分をお願いしたい。進め方はお任せします」
> 
> - **Why（なぜあなたか）を明示する**（専門性・経験への敬意）
> - **ゴールと制約条件を明確にし、Howはお任せ**（自律性を尊重）
> - 曖昧な「なんとなくお願い」は絶対NG
> - 進捗確認は「いつまでに何が見える？」の形で

#### 📌 対立・意見が割れたとき

> **原則**: 「感情 vs 論理」の構図にしない。高松さんの土俵（論理・数値）で議論する
> 
> - 「高松さんの視点はこうだと理解している。一方で、データを見ると別の解釈もあり得る」と**事実ベースで対等に議論**
> - 公の場で真っ向対立は避ける → **1on1で事前にすり合わせ**
> - 高松さんが折れない場合、**第三者（上位者）のデータや判断基準を持ち込む**と着地しやすい
> - **「高松さんが間違っている」ではなく「別の観点もある」**のフレーム

### 5. 注意すべきトリガー（地雷）

| **トリガー** | **こうなると危険** | **回避策** |
| --- | --- | --- |
| **自分の提案が軽視される** | 表面上は冷静だが、内心でフラストレーションが溜まり、以降の協力度が下がる | 提案に対して「なるほど、〇〇の点は筋が通っている」と一度受け止めてから別案を出す |
| **意思決定が進まない** | 「この組織は動かない」というフラストレーション。最悪の場合、組織への信頼自体が低下 | 決められない場合も「次のマイルストーンは〇日。それまでに△△を確認する」と具体的に示す |
| **「できる人に頼る」文化** | 負荷が集中しても弱みを見せないため、限界まで抱え込む。そしてある日突然「もう無理」となる | 負荷状況を定期的に確認。「高松さんの工数を守るために、これは別の人にアサインしたい」と配慮を見せる |
| **プロ意識への無理解** | 「バリューないやつは切っていい」「プロとして大人になるべき」が価値観の根幹。プロ意識の低い相手に強いストレスを感じる | 高松さんの基準に合わせた議論をする。プロフェッショナリズムへの共感を示す |

### 6. まとめ：高松さんへの最適コミュニケーション

> **🎯 3つの鉄則**
> 1. **数値と構造で語れ** — 感情論・曖昧な話はNG。ファクトとロジックが信頼の通貨
> 2. **プライドを傷つけるな** — 弱みの直接指摘・公の場での否定は致命的。成果を認めてから改善を「提案」する
> 3. **事前に巻き込め** — 事後報告は不信感の種。大きな決定は必ず事前に頭出し。「聞いていない」は地雷

> **⚡ 裏の思考を踏まえた上級テクニック**
> - 高松さんは**打算的に状況を読んでいる**。こちらも「高松さんにとってのメリット」を明示すると動きやすい
> - 「高松さんが決めた」という形を作る。選択肢を提示し、最終判断を委ねる
> - 高松さんの提案を通したいときは、**データで武装して「高松さんの提案が正しいことを裏付ける」形**で援護射撃する → 最も信頼を得る行動
> - フィードバックは**「能力」ではなく「環境・文化との適応」**のフレームで。能力否定と受け取られると防御壁が上がる

## ステークホルダーコミュニケーション原則

### 原則1: 伝わらないのは論理ではなく「言語」の問題

技術的に正しいことを言っていても伝わらないとき、問題は**ロジックの欠陥ではなく、使っている言語の違い**にある。

| エンジニアの言語 | 経営の言語（翻訳後） |
| --- | --- |
| コンテキストスイッチングのコストが高い | 集中環境を確保することで、リリース頻度がX%向上する |
| 技術的負債が溜まっている | 現状のまま放置すると、半年後にリリース速度がX%低下する |
| 作業が追いつかない | 同規模企業の開発投資比率はX%。自社はY%で、事業計画との乖離がある |
| オフショアは難しい | 検討済み。現フェーズではA・Bのリスクがあり、代替としてCを選択している |
| リファクタリングが必要 | 年間X時間の手戻りを削減し、施策のリードタイムをY日短縮する投資 |

### 原則2: 説明すればするほど「言い訳」に見える構造を避ける

- **「聞かれてから答える」パターンは後手に見える**。相手の指摘→技術的反論の順番になると、どれだけ正しくても言い訳に映る
- **分業している以上、専門領域のすべてを相手に理解してもらうのは不可能**。ロジックの完全理解を求めること自体が非効率
- **やるべきこと**: 先に検討結果を出す。聞かれる前に整理しておく

### 原則3: 事前に考えていたことが「信頼」になる

経営者が口を出してくるのは信頼が足りないとき。信頼は**「この人はちゃんと考えている」という確信**から生まれる。

**信頼を構築する行動パターン:**

| 行動 | 具体例 |
| --- | --- |
| **事業の言葉で事前整理** | ベンチマーク（同業他社の開発投資比率）を用いて、自社の投資水準の妥当性を数字で示す |
| **選択肢を検討済みで提示** | 「オフショアも検討したが、現フェーズではリスクA・Bがあるため、アプローチCを選択」と先出し |
| **ロードマップに投資計画を組み込む** | 「売上がX水準に達したら、チーム構成はこうあるべき」を事前提示。人員追加が計画の一部と伝わる |
| **定期的な投資レポート** | 四半期ごとに「エンジニアリング投資レポート」を報告。生産性指標・技術負債・採用市場を事業指標と紐づけ |

### 原則4: CTOは「翻訳者」である

技術リーダーの2つの役割:

1. **技術的な正しさを追求する**こと
2. **その活動が事業成果にどう貢献するかを翻訳する**こと

後者を軽視すると、技術的に正しい判断をしていても組織の中で孤立する。

**翻訳者としての実践:**

- 事業計画を読み込み、競合動向を追い、経営者が何を気にしているかを把握する
- 自分たちの活動がそこにどう接続するかを**相手の言葉**で表現する
- これは迎合ではなく、**プロフェッショナルとしての責任**

### アンチパターン早見表

| アンチパターン | なぜダメか | 代わりにやること |
| --- | --- | --- |
| 技術用語で詳細に説明する | 相手にとっては「主張」以上にならない | 事業KPIへの影響に翻訳する |
| 聞かれてから打ち返す | 言い訳に見える構造 | 聞かれる前に検討結果を出す |
| 「できない理由」を積み上げる | 言い訳の量が増えるだけ | 代替案とトレードオフをセットで提示 |
| ロジックの完全理解を求める | 分業の否定。非効率 | 結論と判断根拠だけ伝える |
| 「理解されない」と嘆く | 翻訳責任の放棄 | 自分が経営を理解しに行く |
| 成果を技術指標だけで報告する | 経営にとって意味不明 | 事業指標・同業他社比較と紐づける |
