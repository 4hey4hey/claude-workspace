# EA（SoE領域）構想 全体概要（2026年3月 mTG向け共有）

## 前回1月のEA5兄弟の説明からの主な更新点

- ID管理基盤と顧客管理基盤は一つの体制として4月から基本計画を実施することになった。
- ID管理基盤ではマイクロサービスを活用する方向性となる可能性が高い。
- mTGへのヒアリング/意見調整について4月中旬以降実施する方向で調整中。

## 目指す姿

- デジタル/リアル接点で得られる独自データをTG-IDで統合
- ID、認証、ポイントをSoE領域で共通化
- 他社との外部連携コネクタを整備し、プラットフォーマーへ

## TG-ID基本コンセプト

- TG-IDをキーとして、各種サービス・接点のデータを統合管理する構想

## SoE領域の4つの要素

1. CDP（Databricks）＆Braze
2. ID認証基盤
3. ID管理基盤
4. 顧客管理基盤（新CRM）

## EA取り組みのロードマップ

1. CDP（Databricks）＆Braze導入：25年下期～26年中
2. 認証基盤構築：26年上期～27年上期
3. ID管理基盤＆顧客管理基盤（新CRM）構築：26年下期～28年上期
4. （構想中）サイト統合/マーケティング高度化～

## 個別概要

### 1. CDP（Databricks）＆Braze

- **目的**
  - Webサイト上でのユーザー行動情報の収集
  - メール、SMS、LINE等のユーザーへの発射プラットフォーム
  - 行動情報、属性情報を分析し、最適な提案を行えるようにする
- **基本的思想**
  - PUSH配信ツールの統合により一貫性のあるコミュニケーションを実現する
  - ツール間の機能重複を無くし、コスト最適化を図る
  - 全商材のデータを分析し、一貫した提案ができるようにする
- **スケジュール**
  - 2026年3月：Braze運用開始
  - 2027年3月：現行MA（SalesForce）離脱完了、Braze運用一本化

### 2. ID認証基盤

- **目的**
  - 最新のセキュリティ基準に基づいた安全な認証を提供できるようにする
  - 認証サービスの品質向上
  - TG-ID導入に向けた基盤整備
- **基本的思想**
  - ID認証基盤は認証に必要最低限の機能と情報のみ担務する
  - 将来を見据えて最新のセキュリティ要件に対応する
  - リリース時点でID管理基盤で必要とする機能は基本的にmTGへ残置する
    - 現時点でTG-ID導入後のユーザージャーニー等が不明確な状態のため、大規模な改修を行うことはリスクが高いと判断し、対応を先送り
    - KrakenPJでの会員発行機能の改修と重複するためそちらの意味でも先送り
    - 当該機能は後述のID管理基盤の構築PJにてmTGからの引き剥がし、再構築を実施
- **スケジュール**
  - 26年4月～7月：要件定義
  - 26年10月～27年4月：設計・構築
  - 27年3月～9月：テスト・導入（＝mTGの脱IDCS）
  - 27年10月～12月：フォロー
- **ユーザーに見える機能変更内容**（NIST SP 800-63B-4準拠）
  - パスキーの導入
  - GoogleAuthenticatorによる二要素認証の導入
  - メール（確実）/SMS（調整中）による二要素認証の廃止
  - パスワードの要求桁数の増加（8桁→12桁？16桁？）
  - 漏洩済パスワードの拒否機能の追加

### 3. ID管理基盤

- **目的**
  - TG-IDの運用に伴い、ID認証基盤では不足する部分を担う
    - ID発行、削除API
    - 認証情報変更画面
  - 認証に必要ではないがIDとして共通して持つべき情報（属性情報）を管理する
- **基本的思想**
  - myTOKYOGASのバックエンドにある①TG-ID発番・管理、②許諾管理、③付随情報管理をmTGから引き剝がして共通サービス化する
    - mTG的にはサポート切れしたTGFW脱却も同時に実現したい
  - 各サービスのユースケースがID管理基盤に入り込んでサービス側の柔軟性を損ねることが無いよう、APIと画面を使い分け、管理する情報も限定する
- **スケジュール**
  - 2026年4月～10月：基本計画
    - **基本計画においてmTGに対してもヒアリングが行われる。内容、日程は別途調整。**
  - （予定）2026年下期：要件定義
  - （予定）2027年：開発
  - （予定）2028年：リリース
- **補足**
  - 大部分についてマイクロサービスとして提供することとなる可能性がある
  - 例えば許諾管理については規約同意管理MSにてカバーができる想定

### 4. 顧客管理基盤（新CRM）

- **目的**
  - 各サービスに分散する情報を集約し整形して提供する
  - 情報の集約はTG-IDをキーとして行う
  - 事実ベースの信頼度の高いデータを各サービスに提供する（データオリジンではない、完全な信頼性は担保しない）
- **基本的思想**
  - オリジンサービスからのユーザー由来の事実情報のみを取り扱い、推定情報は扱わない
  - システム内で新しいファクトの追加/修正は行わない
  - 投入されるデータはTG-IDに紐付いた形で取り扱う
- **スケジュール**
  - 2026年4月～10月：基本計画
  - （予定）2026年下期：要件定義
  - （予定）2027年：開発
  - （予定）2028年：リリース
