# デリバリーマネジメントチーム — Mission・OKR概念書

## これは何

チームの存在意義・行動原則・OKRの核をチームメンバー向けに伝えるドキュメント。

* Mission：作る力を、届く価値に変える。ムダなく・ムリなく・ムラなく流れを整える

* 日々の判断基準は「守る・速くする・強くする・見極める」の4原則

* SM・PO・DMの3ロールが原則を分担する。SMはプロセス面、POは投資面、DMは人面

* 5つのKRは連動しており、入口品質→フロー効率→学習ループの順に回す

***

## このチームは何のために存在するか

### Mission

> **作る力を、届く価値に変える。**

作る力が価値まで流れ切るように、流れを整える。
ムダなく・ムリなく・ムラなく。

開発組織は、作る力を持っている。しかしその力は、価値に届いて初めて意味を持つ。リリースされても使われない機能、仕様の曖昧さで生まれる手戻り、人への過負荷、チーム間のばらつき——こうしたものが間に挟まると、作る力は価値に変換されないまま終わる。
デリバリーマネジメントチームは、作る力と価値の間をつなぐ。作ったものが顧客価値・事業価値・社会価値として届き切る状態を、流れを整えることでつくり出す。

***

### 何を整えるのか — 3つの流れ

作る力は、そのままでは価値に届かない。途中で流れが滞ると、せっかくの力が価値になる前に失われる。私たちが整えるのは、この3つの流れだ。

|      | 意味                           |
| ---------- | ---------------------------- |
| **ムダなく流す** | 手戻り・不要な工程・使われない機能で流れを止めない    |
| **ムリなく流す** | 人に過負荷をかけず、持続可能なペースで流す        |
| **ムラなく流す** | チーム間でプロセス・品質のばらつきをなくし、安定して流す |

***

### どう動くのか — 4つの行動原則

日々の仕事で迷ったら、この4つのどれに当たるかで判断する。
**どれにも当てはまらないなら、デリバリーチームの仕事ではない。** 逆に言えば、この4つに当たる行為はロールを問わず全員の仕事だ。

| 原則       | 意味           | 効く流れ  | 主に動く  | 具体的な場面 |
| -------- | ------------ | ----- | ----- | ------ |
| **守る**   | フロー・人の健全性を守る | ムダ／ムリ | SM（プロセス面）・DM（人面） | **SM:** ブロッカー検知・除去、リスク可視化、フロー阻害の早期検知 / **DM:** 1on1、コンディション管理、負荷分散、メンタルケア |
| **速くする** | フロー効率の最大化    | ムダ    | SM＋PO | ボトルネック解消、PBIの小分割、仕様起因の手戻り削減、リードタイム短縮 |
| **強くする** | プロセスと人を強くする  | ムラ    | SM（プロセス面）・DM（人面） | **SM:** ふりかえりの質向上、スクラム標準化、運営の型化 / **DM:** リーダー育成、オンボーディング、採用、キャリア支援 |
| **見極める** | 投資判断・価値の選択   | ムダ    | PO    | 優先順位判断、やらないことの決定、受入条件定義、スコープ判断、レビューでの価値検証 |

※「主に動く」ロールはリード役。原則そのものは全員の共通言語。

***

### 誰が担うのか — 3つのロール

4つの原則は全員の共通言語だが、日々の動きの中で誰がリードするかは分かれている。このチームは3つのロールで構成される。

| ロール | 一言で言うと | 守備範囲 | 主にリードする原則 |
| --- | --- | --- | --- |
| **SM**（スクラムマスター） | プロセスの番人 | フローの障害除去・スクラム運営・プロセス改善。チームが止まらず流れ続ける状態を作る | 守る（プロセス面）・速くする・強くする（プロセス面） |
| **PO**（プロダクトオーナー） | 投資の番人 | バックログの優先順位・受入条件・スコープ判断。流す価値そのものを見極める | 見極める・速くする |
| **DM**（デリバリーマネージャー） | 人の番人 | 1on1・コンディション管理・リーダー育成・採用・オンボーディング。人が健全に走り続け、成長できる状態を作る | 守る（人面）・強くする（人面） |

ロールは分かれているが、原則は共通。自分のロールでなくても、原則に当たる場面では全員が動く。

**境界の原則：** SMはプロセスの番人、DMは人の番人。「守る」も「強くする」も、プロセスに効くものはSM、人に効くものはDMがリードする。チームの心理的安全性はSMが「場」を作り、DMが「人」を見る。

> **現時点の体制（2026年4月）:** PO: 福田さん（Naikist/Web）、伊藤さん（Fluppuccino PO候補・育成中）、室山さん（Guardians PO候補）、森さん（Discovery 兼務）。SM: 中道さん（Fluppuccino）、長島さん（Discovery/Guardians・新任）、堤さん（Naikist SM補佐）。DM: **採用中**（入社までは杉崎が兼務）。

***

### 誰にとって、何を意味するか

同じMissionでも、読む人によって見え方は変わる。経営にはこう見え、チームにはこう感じられる。

**経営にとって：** 同じ予算・体制で、施策をもっと速く・多く回せるようにすること。

**チームにとって：** 作る力が価値まで流れ切るよう、プロセスを守り速くし強くし、人を守り育て、投資を見極めること。

## 杉崎のOKR

### Objective

> **正しいものを、速く、持続的に届けるデリバリーラインを確立する**

「速く作る」だけではない。入口（何を作るか）から出口（届けた結果から学ぶ）まで、一気通貫で回す状態を作る。

***

### 6つのKR（FY26更新版）

| KR        | 名前    | 何を目指すか                       | 対応する行動原則 | 主な担い手 | 効く流れ |
| --------- | ----- | ---------------------------- | -------- | ----- | ---- |
| **挑戦KR1**   | 入口品質  | 開発ラインに流す仕事の質を上げ、仕様起因の手戻りをなくす（DPG KR②アプローチ②直接受け） | 見極める     | PO | ムダ   |
| **挑戦KR2**   | フロー効率 | 着手からリリースまでの時間を短縮する（DPG KR②アプローチ③直接受け）           | 速くする     | SM + PO | ムダ   |
| **挑戦KR3**   | 学習ループ | 届けた結果から学び、次の投資判断に活かすサイクルを作る（DPG KR②アプローチ④直接受け）  | 見極める     | PO + SM | ムダ   |
| **基盤KR1** | 健全性＋開発環境の安定 | チームが持続可能なペースで走れる状態を守りつつ、開発環境（AVD/EDR/オンボーディング/ツール）を安定運用する | 守る       | DM（人面）+ SM（プロセス面）+ 岩渕（環境面） | ムリ   |
| **基盤KR2** | 能力の多面的強化 | プロセス標準化＋次世代リーダー育成＋デリバリー基盤刷新（計測/inet/KARTE）を3層で進める | 強くする     | SM（プロセス面）+ DM（人面）+ 岩渕（基盤刷新）+ 堤（計測基盤） | ムラ   |
| **基盤KR3** | 事業貢献PDCA | 事業側（toC各部）との四半期相互検証で外部妥当性を担保する（DPG KR⑦直接受け） | 見極める×強くする | 3者共同責任（片山TL・檜垣TL・杉崎TL） | 外部妥当性 |

### 6KRの連動構造

挑戦3KRは因果ループで連動し、基盤3KRはそれぞれ異なる役割で挑戦KRを支える:

```
   ┌─ 挑戦KR1 入口品質 ─→ 挑戦KR2 フロー効率 ─→ 挑戦KR3 学習ループ ─┐
   │                                                                  │
   └─────────────── 学びが仮説精度を高める（KR3→KR1） ─────────────────┘

   ─────── 挑戦3KRすべての前提・支え（基盤3KR） ───────

   基盤KR1 健全性＋開発環境  → 3KRすべての前提条件＋安全装置（守る/ムリ）
   基盤KR2 能力の多面的強化  → 3KRすべての中長期持続性（強くする/ムラ）
   基盤KR3 事業貢献PDCA      → 6KR全体の外部妥当性チェック（見極める×強くする）
```

**挑戦KR1〜3の循環**:

* KR1が改善 → 手戻りが減り、フロー（KR2）が速くなる
* KR2が改善 → 速く届く → 検証頻度が上がり、学習（KR3）が回る
* KR3が回る → 学びが仮説精度を高め、入口品質（KR1）がさらによくなる

**どれか1つだけやっても効果は限定的。3つが連動して初めてMissionが達成される。**

**基盤3KRの役割分担**:

* **基盤KR1（健全性＋開発環境）** — 3KRの「前提条件」＋「安全装置」。人が走り続けられる状態と、開発環境（AVD/EDR/オンボーディング/ツール）の安定運用を担保しつつ、DORAサブKPI（変更失敗率・SPACE）で「速さの暴走」を防ぐ
* **基盤KR2（能力の多面的強化）** — 3KRの「中長期持続性」。FY26単年では成果が見えにくいが、FY27以降にDORA Highレベルへ到達するための組織能力（プロセス標準化／次世代リーダー育成／デリバリー基盤刷新）を蓄積する
* **基盤KR3（事業貢献PDCA）** — 他5KRがすべて内向き（デリバリー機能の内部最適）であるのに対し、唯一外向き（事業側から見た信頼度）の軸。デリバリーチームが内向きに自己完結しないための外部妥当性チェック

挑戦KR3つはムダ対策に集中させ、ムリは基盤KR1、ムラは基盤KR2、外部妥当性は基盤KR3でカバーする。3つの流れ全体＋外向きの信頼を取りに行く構造だ。

***

### PO・SM・DMを貫くひとつの原則

> **価値を小さく分割し、比較可能な単位で届け、仮説検証のスピードを上げる**

| 実践            | やること                         | 効くKR    |
| ------------- | ---------------------------- | ------- |
| 価値の垂直スライス     | 技術レイヤーではなく、ユーザー価値の単位で分割する    | KR2・KR3 |
| 比較可能な単位で出す    | A案/B案を同時に出して定量比較できるサイズにする    | KR3→KR1 |
| 仮説を明示してから着手する | PBIに「検証したい仮説」を1行書いてスプリントに入れる | KR3・KR1 |
| 「やらない」を先に決める  | 分割時に今回やらないことを明示的にリストアップする    | KR1・KR2 |

## Appendix: DPG OKR → チーム6KR の導出過程

ロジックツリーは「DPG OKR → 成果の方程式 → チームO → 6KR」の4層で読む。上位OKRから杉崎チームO・6KRがどう導出されたかを一気通貫で示す（PLAID方式）。

```
【上位接続】DPG OKR
   ├ KR② 内製化組織の組織能力指標 ← 主責任（4アプローチ全対応）
   ├ KR⑥ コスト構造改善           ← 直接貢献（inet発注最適化）
   └ KR⑦ 事業貢献PDCA             ← 直接受け（3者共同責任）
       ↓ そのために（成果の方程式）

【成果の方程式】
   どうやって内製開発のリソースを事業成果に転換して、投資対効果を最大化するか？
   （試行回数 × 施策の精度 ＝ 事業成果）
       ↓ デリバリー機能側ができることは

【杉崎チームO】
   顧客価値を、正しく・速く・持続的に届けるデリバリーラインを確立する
       ↓ そのために、デリバリーチームは何を実現すべきか？

【6つの分解問い → 6KR】
│
├─ 開発ラインに流す仕事の質と判断スピードをどう高め、
│   仕様起因の手戻りと無駄な開発をなくせるか？
│   （→ 挑戦KR1: 入口品質 / 見極める / ムダ）
│
├─ 顧客に届くスピードをどう上げるか？
│   （→ 挑戦KR2: フロー効率 / 速くする / ムダ）
│
├─ 届けた価値を検証し、学びを次のサイクルにどう反映し続けるか？
│   （→ 挑戦KR3: 学習ループ / 見極める / ムダ）
│
├─ チームが走り続けられる土台と開発環境をどう担保するか？
│   （→ 基盤KR1: 健全性＋開発環境の安定 / 守る / ムリ）
│
├─ デリバリー組織そのものを中長期にどう強くしていくか？
│   （→ 基盤KR2: 能力の多面的強化 / 強くする / ムラ）
│
└─ 事業側から見て、このデリバリー機能は本当に価値を生んでいるか？
    （→ 基盤KR3: 事業貢献PDCA / 見極める×強くする / 外部妥当性 ※DPG KR⑦直接受け）
```

**4層の読み解き**:

* **（1）DPG OKR** — 上位（及川GMが持つ組織OKR）。デリバリーチームはKR②の主責任、KR⑥に直接貢献、KR⑦を直接受ける
* **（2）成果の方程式** — 「事業成果＝試行回数×施策の精度」のうち、**事業成果は事業側責任、その先行指標（組織能力）はデリバリー側責任**という建付け
* **（3）杉崎チームO** — 「事業成果のためにデリバリー機能側ができること」を言い切り形で表現
* **（4）6つの分解問い** — チームOを6つの問いに分解した結果が6KR

**KR間の因果接続:** 挑戦KR1（入口）→ 挑戦KR2（フロー）→ 挑戦KR3（出口・学習）で開発バリューチェーンの全区間をカバーする。挑戦KR3の学習結果が挑戦KR1の入口品質をさらに改善する循環構造。基盤KR1〜3はそれぞれ「前提条件＋安全装置」「中長期持続性」「外部妥当性」の役割で挑戦KRを支える。
