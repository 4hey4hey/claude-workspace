## 概要

もともとPO・SM・PdM・総務（経理・契約・コーポレートエンジニアリング）が一体のチームだった。PdM領域を分離し、デリバリー領域に特化したチームとして再定義する。コーポレート機能（経理・契約）もデリバリーチームからは分離。

## Mission

### 対経営

> **開発投資が最短で顧客価値に変換される状態を作る**

### 行動原則（対チーム）

> **開発ラインの流れを 守り、速くし、強くする**

- **守る** — 障害除去・健全性維持（ブロッカー検知、1on1、メンタル、負荷分散）
- **速くする** — WIP最小化・ボトルネック解消（バックログ優先順位、受入条件精度、手戻り削減）
- **強くする** — 育成・プロセス改善（チーフ育成、オンボーディング、ふりかえり、標準化）

日々の判断基準：「今やろうとしていることは、流れを守ることか？速くすることか？強くすることか？」——どれにも当てはまらないなら、デリバリーチームの仕事ではない。

### 設計思想：リーン生産方式のアナロジー

| 製造業 | ソフトウェア開発 | デリバリーチームの役割 |
|---|---|---|
| 工場の機械・ライン | エンジニア／開発チーム | — |
| 生産計画（何をどの順で投入するか） | PO機能 | 速くする |
| ライン監視（停止検知・障害除去） | SM機能 | 守る |
| 予防保全（定期点検・状態監視） | ピープル（1on1・負荷分散） | 守る |
| 品質管理（不良品の工程内作り込み） | 手戻り削減（受入条件・仕様精度） | 速くする |
| 生産技術・カイゼン推進 | プロセス改善・ふりかえり | 強くする |
| 技能訓練 | チーフ育成・オンボーディング | 強くする |

**製造と異なる重要な点：**
- エンジニアは「同じ製品を繰り返す機械」ではない。毎回異なる問題を解く
- 稼働率100%が最適ではない。余白（slack）がないと改善・学習・創造ができない
- 配置換えコストが桁違いに高い（ドメイン知識・コードベース理解・チーム関係性）

## Missionの2つの柱

```
開発投資のリターンを最大化する
  ├─ スループット（速さ）
  │    How / When（どの順番で・いつ届けるか）← PO機能
  └─ サステナビリティ（持続性）
       Who / Sustainably（誰が・持続可能に走れるか）← ピープル・プロジェクト
```

片方だけでは「人を削って速度を出す」か「チームを守って遅くなる」に寄る。両輪をMissionに内包していることがデリバリーチームの特徴。

## 所管領域

| 領域 | 所管 | 具体業務 |
|---|---|---|
| **ピープルマネジメント** | 主管 | 1on1・育成支援・オンボーディング・関係性の早期検知・チーフ育成 |
| **プロジェクトマネジメント** | 主管 | スクラム運営・リスク可視化・進行管理・スプリント計画・障害除去 |
| **PO機能** | 主管 | バックログ管理・優先順位決定・受入条件定義・スコープ判断 |
| **プロダクトマネジメント** | PdMチーム | What/Whyの決定・仮説検証・ロードマップ |
| **テクノロジーマネジメント** | 中島さん＋各チームリーダー | 技術意思決定・アーキテクチャ・DevOps |

## PdMチームとの境界

| | PdMチーム | デリバリーマネジメントチーム |
|---|---|---|
| **問い** | 何を作るか・なぜ作るか（What / Why） | どの順番で・いつまでに届けるか（How / When） |
| **責任** | プロダクト戦略・仮説立案・顧客検証 | バックログ実行・スプリント運営・チーム稼働 |
| **接続点** | ロードマップと優先方針をインプット | POがそれを開発チームの動きに翻訳 |
| **判断権限** | 「何を作るか」の最終判断 | 「どの順番で・どこまで作るか」の最終判断 |

**原則：PdMが「作るべきもの」を決め、POが「届け方」を決める。PdMはスプリントに介入せず、POはロードマップに介入しない。**

## 構造的課題

- チーム数増加に対してマネジメント人数が追いついていない
- SM実質1名（中道さん）、PO専任不在
- ピープル・プロジェクトマネジメントがスクラムマスター等に集中
- チーフエンジニア層のマネジメント力育成が計画的にできていない
- 問題が顕在化してからの事後対応型になりやすい

## 成果指標（案）

| 指標 | 意味 | 守/速/強 |
|---|---|---|
| **リードタイム** | 着手→リリースの日数 | 速くする |

## 組織拡大方針との関係

デリバリーマネジメントチームは、BtoC領域の開発組織拡大方針（[org-expansion-policy.md](org-expansion-policy.md)）の実行基盤でもある。組織拡大の3方針：

1. **のれん分け方式** — myTOKYOGASをLeSS化し、育成機関として文化を維持しながら拡大
2. **inet人材活用** — グループ会社人材のmTG開発体制への参画で育成・輩出を加速
3. **手の内化外注** — 依存度×変化の2軸で内製/外注を判断

デリバリーチームはこの拡大に伴い増加するチームの **SM・PO機能の供給** と **チーフ育成の加速** を担う。組織が拡大するほどデリバリーチームの「強くする」機能の重要性が増す。
